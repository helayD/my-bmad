# Story 1.3: 团队工作空间创建与项目上限治理

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为组织管理员，
我希望能创建团队工作空间并管理其中的项目，当活跃项目达到 50 个上限时系统阻止继续创建并给出明确指引，
以便团队项目在可控范围内有序运行。

## 验收标准

1. **Given** 用户已认证且拥有创建团队工作空间的权限 **When** 用户提交创建团队工作空间请求 **Then** 系统创建 `type=TEAM` 的工作空间，创建者自动成为 OWNER **And** 工作空间 slug 唯一且基于团队名称生成 **And** 同时创建一条 `WorkspaceMembership`（`role=OWNER`）记录 **And** 创建成功后调用 `revalidatePath("/")` 刷新 sidebar

2. **Given** 团队工作空间已有 50 个活跃项目 **When** 管理员尝试创建新项目 **Then** 系统阻止创建，并展示明确提示：归档已完成项目、升级套餐或申请扩容（FR2） **And** 提示信息包含当前活跃项目数和上限值

3. **Given** 团队工作空间有 49 个活跃项目 **When** 管理员创建新项目 **Then** 项目创建成功，活跃项目计数更新为 50

4. **Given** 管理员归档一个项目使活跃数降至 49 **When** 管理员再次尝试创建新项目 **Then** 创建成功

5. **Given** 用户提交的团队名称为空或仅含特殊字符 **When** 表单校验执行 **Then** 系统拒绝创建并给出有效的输入校验错误

6. **Given** 两个用户同时创建相同 slug 的工作空间 **When** 发生 slug 唯一约束冲突 **Then** 系统自动追加随机后缀重试创建，不要求用户手工修改（复用 Story 1.2 已有并发安全模式）

### 关键架构约束

- **Brownfield 集成**：Story 1.1 已创建 Workspace、Project、WorkspaceMembership Prisma 模型。Story 1.2 已创建个人工作空间供应逻辑。本 Story 在应用层扩展团队工作空间能力，**不修改 Prisma schema**。

### slug 生成规则（团队工作空间）

```typescript
import { generateSlug, randomHex } from "@/lib/workspace/slug-utils";

// generateSlug 签名: (input: string, fallback?: string) => string
// 默认 fallback = "user"（向后兼容个人空间）
// 团队空间调用: generateSlug(teamName, "team")
//
// 处理步骤:
// 1. 若输入含 @，取 @ 前缀；否则使用原始输入
// 2. toLowerCase()
// 3. 替换所有非 [a-z0-9] 字符为 "-"
// 4. 合并连续 "-"、去首尾 "-"
// 5. 截断至 39 字符
// 6. 如果为空，回退到 fallback 参数值（"team" 或 "user"）
// 7. 如果 slug 冲突（P2002），追加 "-" + randomHex(4)
//
// 示例:
//   generateSlug("My Awesome Team", "team") → "my-awesome-team"
//   generateSlug("团队Alpha", "team") → "alpha"（非ASCII替换后）
//   generateSlug("团队", "team") → "team"（全非ASCII，使用 fallback）
//   generateSlug("团队", "user") → "user"（个人空间 fallback）
//   冲突时 "my-team" → "my-team-b7f2"
```

### 项目上限检查流程

```
createProjectAction(input) 调用 → 
  1. 认证：getAuthenticatedSession()
  2. Zod 校验：workspaceId (cuid2), name (trim 1-100)
  3. getWorkspaceMembership(workspaceId, userId) → 检查 role 为 OWNER 或 ADMIN
  4. getWorkspaceById(workspaceId) → 获取 workspace（需要 type 和 slug）
  5. createProject({ workspaceId, name, workspaceType: workspace.type })
     ↓ 领域函数内部：
     5a. if (workspaceType === "TEAM") → checkProjectLimit(workspaceId)
     5b. if (!allowed) → throw ProjectLimitExceededError(currentCount, limit)
     5c. prisma.project.create + P2002 冲突处理
  6. revalidatePath(`/workspace/${workspace.slug}`)
  7. return { success: true, data: project }
```

**注意**：`createProject` 领域函数需要接收 `workspaceType` 参数来决定是否检查上限，而不是在函数内部再查一次 workspace。这避免了 N+1 查询。
