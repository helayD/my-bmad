# Story 4.6: 执行边界与上下文安全控制

Status: ready-for-dev

## Story

作为系统，
我希望限制任务执行在其授权的项目与上下文边界内进行，
以便防止跨项目上下文泄露或未授权的文件访问。

## Acceptance Criteria

1. **Given** 任务属于项目 X
   **When** 执行监督器配置 tmux session 环境
   **Then** 工作目录限定为项目 X 的根目录
   **And** 上下文注入仅包含项目 X 授权范围内的文件和配置（FR24）

2. **Given** 执行中的 agent 尝试访问项目根目录外的路径
   **When** 执行监督器检测到路径穿越尝试
   **Then** 访问被拒绝，事件记录到审计日志
   **And** 任务不因此中断，但异常行为被标记

3. **Given** 项目关联了本地目录
   **When** 执行监督器配置执行环境
   **Then** 符号链接访问被拒绝
   **And** 文件访问深度、文件数和文件大小受限
   **And** 敏感路径（如 `.env`、密钥文件）被保护，不注入到 agent 上下文

## Dev Notes

### 核心实现目标

- 4.6 的核心不是“再写一层提示词让 agent 自觉别越界”，而是把**项目根目录真值、受控路径解析、allowlist 上下文注入、敏感路径保护、审计留痕**串成一条可验证的控制链。
- 4.6 必须让控制面可以诚实回答三个问题：
  - 这次执行到底绑定到了哪个项目根目录？
  - 平台到底注入了哪些上下文，跳过了哪些敏感内容？
  - 如果出现越界访问尝试，平台在哪一层拦住了它，并留下了什么证据？

### 默认实现决策

- **边界保证以“平台可控表面”为准。** 受控路径解析、上下文准备、写回目标、session 启动 cwd 必须可强约束；无法截获的原始 shell 行为不能被宣传成已完整拦截。
- **project root 真值必须 relation-first 且 canonical。** 启动前拿到 `localPath` 后，应得到 `projectRootRealPath` 并保留在 execution boundary profile 中；不要只保存用户配置时填的字面路径。
- **上下文必须 allowlist 注入。** 平台只把已经通过校验的文件交给 agent，而不是把整棵项目树暴露出去后再靠 prompt 限制。
- **敏感路径保护优先于“多给点上下文”。** 一旦命中 `.env` / 密钥 / SSH / 凭据类文件，默认跳过并留痕，不因用户体验方便而默认放行。
- **违规默认非致命、启动前置失败默认阻止启动。** 运行中受控访问违规应记录并标记；根目录缺失、scope mismatch、context preparation 失败等前置问题则应阻止 session 创建。
- **边界状态必须在真实启动时重算。** 队列等待、reroute、retry 后不能盲用旧快照。
- **不要再发明第三套 metadata-only 边界系统。** 4.4 的 `ExecutionSession`、4.5 的 queue / admission、现有 `content-provider` helper 才是本 story 的基础。

### 最新技术信息

- 截至 **2026-04-19**，Next.js 官方 `revalidatePath()` 文档仍说明：在 Server Functions 中调用可立即更新当前视图；在 Route Handlers 中则是标记目标 path，在**下次访问时**执行 revalidation。4.6 如果通过 action / 内部 route 更新边界状态，应继续精确刷新任务详情与项目页，而不是依赖客户端手工刷新。[Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
- 截至 **2026-04-19**，Node.js 官方 `child_process` 文档仍明确区分：`spawn()` 异步创建子进程；`exec()` 通过 shell 执行；`execFile()` 默认不先起 shell。4.6 / 4.4 的 tmux 控制命令仍应优先走参数数组的 `spawn()` / `execFile()`，不要把 project path、session name、命令参数拼进 shell 字符串。[Source: https://nodejs.org/api/child_process.html]
- 截至 **2026-04-19**，Node.js 官方 `path.resolve()` 文档仍强调它是 lexical absolute-path resolution；`fsPromises.lstat()` 只在目标是 symbolic link 时返回 link 自身的 stats；`fsPromises.realpath()` 会给出实际路径。4.6 做项目根目录与目标路径校验时，应把 lexical `resolve()` 与 symlink-aware `lstat()` / `realpath()` 组合使用，而不是只做字符串前缀判断。[Source: https://nodejs.org/api/path.html] [Source: https://nodejs.org/api/fs.html]
- 截至 **2026-04-19**，Prisma 官方 transactions 文档仍把 read-modify-write 场景的重点放在 **idempotent APIs** 和 **optimistic concurrency control**。4.6 的边界准备与 4.5 的 admission 类似，属于“读当前状态 → 校验 → 写入准备结果”的短事务场景，应保持可重试、可幂等，不把 tmux / 文件系统 I/O 包进长事务。[Source: https://www.prisma.io/docs/orm/prisma-client/queries/transactions]
- tmux 官方手册仍说明：`new-session -c <start-directory>` 可为 session 指定工作目录，而 `pane_pid` 表示 pane 首个进程 PID。4.6 的执行边界必须继续沿用 tmux 原生 `-c` 工作目录控制，而不是由应用层口头假定“agent 会自己待在正确目录”。[Source: https://man.openbsd.org/OpenBSD-7.2/tmux.1]
