# Story 4.1: 任务定义与执行意图提交

Status: review

## Story

作为用户，
我希望能为任务指定目标、优先级和执行意图，
以便系统明确我期望达成的结果和执行偏好。

## Acceptance Criteria

1. **Given** 用户在项目上下文中发起新任务（手动或从工件衔接） **When** 用户填写任务表单 **Then** 表单包含：任务目标（必填）、优先级（HIGH/MEDIUM/LOW）、执行意图描述（可选）、偏好 agent 类型（可选）
   **And** 表单持续显示关联的 Story/Epic 上下文（UX-DR19）

2. **Given** 用户提交任务 **When** 表单验证通过（Zod 4 校验） **Then** 系统创建 Task 记录，状态为 PLANNED，包含 `projectId`、`workspaceId`、目标、优先级、意图、来源工件引用
   **And** 用户立即看到任务已创建的反馈和当前阶段（FR18, UX-DR2）

3. **Given** 用户提交的任务目标为空 **When** 用户点击提交 **Then** 系统拒绝并展示内联错误提示

### 关键实现守栏

- 4.1 的核心是真实创建 `planned` Task，并保存用户对目标、优先级、意图和 agent 偏好的输入；**不是**执行路由、派发、启动或编码。
- 手动新任务不能伪造 `planningRequestId`，也不能假装来自 planning handoff；planning-derived task 的 `taskHandoffSummary`、dispatch queue 和 deferred artifact 语义都不属于 4.1。
- 不要把 4.1 简化成“给现有 artifact form 加两个字段”：验收标准明确包含项目上下文下的手动新任务，因此 action 和 UI 都必须支持**无来源工件**的创建路径。
- 对新创建任务，用户可见反馈必须说真话：
  - 可以说“已计划”“等待派发”“等待审批”。
  - 不能说“已开始编码”“已分配 agent”“后台会话已启动”。
- `preferredAgentType` 必须是后续 4.2 可以直接消费的稳定值，而不是 UI 专用文案；值和 label 的映射要集中定义。
- `intentDetail` 仅是对结构化 intent 的补充，不应取代现有 `intent` 枚举，否则 4.2 的 agent 路由会失去稳定输入。
- 对空 goal 的错误提示必须内联沉淀在表单中，并使用中文文案；toast 只能做辅助，不可替代表单级错误表达。

### 最新技术信息

- Next.js 官方 `revalidatePath()` 文档说明，该 API 用于按路径按需使缓存失效；4.1 的任务创建 mutation 仍应坚持“Server Action 写入 + `revalidatePath()` 精确刷新 + 客户端 `router.refresh()` 辅助”的组合，而不是只靠客户端本地 state。
- React 官方 `useTransition()` 文档强调：若需要让 `await` 之后的状态更新也保持 transition 语义，需要再次包裹在 `startTransition()` 中。4.1 如果在提交成功后要做多次异步后的 UI 更新，应注意这一点，避免新表单出现非预期卡顿。
- Zod 官方文档确认 `safeParse()` 与 `z.string().trim().min(1)` 适合做“客户端即时轻校验 + 服务端最终校验”的共享 schema。4.1 应尽量复用同一 schema，而不是客户端手写一套、服务端再写一套。
- 当前仓库实际依赖版本以 `package.json` 与 `_bmad-output/project-context.md` 为准：Next.js `16.1.6`、React `19.2.3`、Prisma `6.19.2`、Zod `4.3.6`。本 Story 不做依赖升级。

### 范围边界

**本 Story 包含：**

- ✅ 支持项目上下文下的手动新任务创建
- ✅ 支持从工件衔接创建任务并保留 Story / Epic 上下文
- ✅ 新任务以 `planned` 为真实初始状态
- ✅ 采集结构化 intent、可选 intent 说明和可选 preferred agent
- ✅ 创建成功后的“状态 + 原因 + 下一步”反馈
- ✅ 任务详情与共享读模型对新字段和 `planned` 的兼容
- ✅ 审计事件、表单校验和回归测试

**本 Story 不包含：**

- ❌ 不执行 agent 路由，不决定 `codex` / `claude code` 最终分配（Story 4.2）
- ❌ 不创建 `AgentRun`、`ExecutionSession` 或 tmux 后台会话（Story 4.2 / 4.4）

### Project Structure Notes

- 当前项目信息架构更适合“项目页内发起，任务详情页承载结果”，而不是“先去全局任务中心，再回项目页补上下文”。4.1 应继续顺着这个结构演进。
- 工件树和结构化详情已经形成“列表-详情-上下文”的稳定模式；共享任务创建表单最好复用这套节奏，而不是新开一个脱离上下文的表单页面。
- 与 4.1 直接相关的共享领域真值优先落在 `src/lib/tasks/` 与 `src/actions/task-actions.ts`；UI 只消费它们，不在组件内私自拼生命周期或路由偏好。
