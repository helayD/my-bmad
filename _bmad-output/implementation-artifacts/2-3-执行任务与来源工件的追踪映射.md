# Story 2.3: 执行任务与来源工件的追踪映射

Status: ready-for-dev

## Story

作为用户，
我希望系统能维护执行任务与其来源 BMAD 工件之间的可追踪映射关系，
以便我随时知道某个任务是从哪个 Story/Epic/PRD 发起的，以及某个工件产生了哪些执行。

## Acceptance Criteria

1. **Given** 任务已从某个 `BmadArtifact` 发起 **When** 用户查看任务详情 **Then** 详情中展示来源工件类型、名称和层级路径（如 `PRD > Epic 3 > Story 3.2`） **And** 提供跳转到来源工件的链接

2. **Given** 用户查看某个 Story 工件 **When** 用户查看该 Story 的关联执行区域 **Then** 展示所有从该 Story 发起的任务列表，包含任务状态、创建时间和最近活动 **And** 列表支持按状态筛选

3. **Given** 一个 Story 已产生多次执行（重试、不同分支等） **When** 用户查看该 Story 的执行历史 **Then** 所有执行记录按时间倒序排列 **And** 每条包含任务 ID、状态、agent 类型和执行结果摘要

## Dev Notes

### 当前基线与关键依赖

- Story 2.2 已经落地了最小来源映射链路：`Task.sourceArtifactId` 作为来源工件外键，任务创建时会把 `context.sourceArtifact.hierarchy` 写入 `Task.metadata.sourceContext.hierarchy`，这是本 Story 的直接基础，不要重做来源模型。
- `prisma/schema.prisma` 当前的 `Task` 模型已经具备 `workspaceId`、`projectId`、`sourceArtifactId`、`status`、`currentStage`、`nextStep` 与 `metadata`；对本 Story 而言，这些字段足以承载“直接来源映射 + 同一 Story 的多次执行历史”。
- `src/actions/task-actions.ts` 中的 `createTaskFromArtifactAction()` 已负责创建任务、写入来源工件关联和最小即时反馈；Story 2.3 应建立在此之上，**不要**绕开现有创建路径再造一条并行流程。
- `src/lib/db/helpers.ts` 中已存在 `getTaskById()`，并且会连带查询 `sourceArtifact -> parent -> parent` 层级；Story 2.3 的反向历史查询应沿用同类 `cache()` helper 模式，而不是直接在页面或客户端中散写 Prisma 查询。
- `src/app/(dashboard)/workspace/[slug]/project/[projectSlug]/tasks/[taskId]/page.tsx` 当前在本地定义了 `buildSourceHierarchy()`；这意味着来源链路组装已经有可复用逻辑，但仍停留在页面私有实现，适合在本 Story 中抽为共享 helper。
- `src/components/tasks/task-detail-view.tsx` 已展示最小的来源工件引用卡片；本 Story 需要把它从“最小引用”升级为“稳定可追踪映射展示”。
- `src/components/artifacts/artifact-detail-sheet.tsx` 当前只覆盖“查看工件详情 + 发起执行 + 创建后的即时反馈”，还没有“反向查看此工件产生了哪些任务”的能力，这是本 Story 的主要 UI 增量落点。
- 当前仓库中**没有** `AgentRun` 模型、`agentType` 字段或执行结果摘要的统一表结构；因此 AC #3 所需的 `agent 类型` 和 `执行结果摘要` 必须通过现有 `Task` 读模型与 `metadata` 兜底方式实现，不要把 Story 2.3 扩成完整执行子系统。

### 建议的历史读模型契约

- 工件历史列表的最小展示单元应是“面向 UI 的整形结果”，而不是裸 Prisma 记录。建议包含：`taskId`、`title`、`status`、`createdAt`、`currentStage`、`currentActivity`、`agentTypeLabel`、`resultSummary`、`sourceArtifactName`、`taskDetailHref`。
- `currentActivity` 读取优先级建议为：`metadata.currentActivity` → `currentStage` → `nextStep` → `暂无最近活动`。
- `agentTypeLabel` 与 `resultSummary` 若当前 `metadata` 中不存在，直接回退到中文兜底，不要为了“看起来完整”而虚构执行事实。
- 任务详情页和工件历史列表应复用同一套来源层级 helper / 展示整形 helper，避免任务页显示一套路径、工件侧栏显示另一套路径。

### 实现护栏

- **不要重建追踪模型**：`Task.sourceArtifactId` 已经是任务到来源工件的直接事实链路；本 Story 的目标是补齐查询与展示，不是设计一套新的 `TaskArtifactLink` 或多对多映射表。
- **不要提前引入执行子系统**：`AgentRun`、`Session`、`Heartbeat`、`Writeback`、SSE、tmux 监督等能力分别属于后续 Epic / Story，不应因为 AC #3 的展示需求被提前拉进来。
- **不要把 Prisma 查询塞进客户端组件**：`artifact-detail-sheet.tsx` 是客户端组件，但数据读取必须通过 Server Action 或服务器入口提供，避免在浏览器侧承载持久化真值与权限判断。
- **不要把“最近活动”写成假状态**：优先读取 `metadata.currentActivity` 或统一整形后的展示字段；如果当前没有数据，展示真实的中文兜底文本，而不是臆造“运行中”“已完成”等状态。
- **不要让 UI 依赖来源关系永远存在**：`sourceArtifact` 可能因软删除、状态变化或历史数据缺口出现不完整链路；展示层必须能回退到 `metadata.sourceContext.hierarchy`，而不是因为父链断裂而崩溃。
- **不要将状态筛选做成新枚举系统**：继续复用 `src/lib/tasks/types.ts` 中既有的 `TASK_STATUS_VALUES` 和 `TASK_STATUS_LABELS`。
- **不要忽略查询作用域**：所有工件历史查询都必须显式绑定 `projectId`，并在入口校验工件归属；不能只凭 `artifactId` 或 `sourceArtifactId` 查询。
- **不要把 `all` 当成真实状态写入数据库或 action schema**：它只是前端筛选用的虚拟选项。
- **不要把反向执行历史扩展成聚合执行中心**：本 Story 仅要求“某个 Story 直接发起的任务列表”；Epic 级聚合、跨 Story 汇总、产物视图和回写链路分别属于 Story 2.4 / 2.5。
- **默认不做 Prisma schema 改动**：若 `metadata` 和现有字段已能满足 AC，则不要额外新增列；只有在你能证明现有结构无法支撑稳定展示时，才考虑最小化、可逆的字段扩展，并同步补迁移、`prisma generate` 与测试。
- **不要默认做实时刷新或轮询**：本 Story 只要求稳定查看历史与状态，不要求引入 SSE、轮询器或后台自动刷新机制。

### 技术要求

- 所有新增 Server Actions 统一放在 `src/actions/`，入口先做 Zod 4 校验。
- 所有 Server Actions 返回 `ActionResult<T>`，失败分支统一使用 `sanitizeError()`，错误消息必须为中文。
- 读取执行历史的入口使用 `requireProjectAccess(workspaceId, projectId, userId, "read")`；不要误用 `execute` 权限。
- 导入路径继续使用 `@/` 别名；Prisma 类型继续从 `@/generated/prisma/client` 导入。
- 读取 helper 优先放入 `src/lib/db/helpers.ts` 并使用 `cache()` 包装，保持与现有任务详情、工件详情数据流一致。
- 若对任务创建流程补充 metadata 字段，必须继续在 `createTaskFromArtifactAction()` 中一次性落库，不要在创建后再发第二次 mutation 修补来源数据。
- 用户可见文案、空状态、筛选标签、错误消息和兜底文本统一使用中文。
- 组件交互继续沿用现有客户端模式：局部状态、`useTransition`、`sonner` 反馈、必要时 `router.refresh()`；不要引入全局客户端状态库。
- 若新增历史查询 action 的筛选参数，建议 schema 只接受 `TaskStatus` 真值或可选空值；`all` 在进入 action 前就应被转换为“不传筛选条件”。

### 架构一致性要求

- 架构文档要求 PostgreSQL 继续作为统一事实源，因此“工件 -> Task 历史”应由数据库查询得出，不要依赖前端缓存列表或 URL 参数拼装伪历史记录。
- 架构文档的“核心实体 + 状态投影 + 事件审计”原则意味着：本 Story 可以通过 `Task` 当前字段 + metadata 先满足映射与展示，但不要把原始执行日志、产物、心跳等高频信息硬塞进 `Task` 主表。
- 鉴权必须保持在服务端 action / 页面 / 领域服务层，不依赖中间件或前端隐藏按钮来保证边界。
- 前端架构要求默认 Server Components、局部客户端交互。因此任务详情页继续使用服务器数据加载；筛选和侧栏交互属于客户端增强即可。
- 服务边界要求核心领域逻辑落在 `src/lib/tasks`、`src/lib/artifacts`、`src/lib/db`；不要把映射整形逻辑埋进 `src/components/**`。
- 该 Story 的 UI 集成点应继续留在现有工件详情侧栏与任务详情页内，避免为了显示历史列表新增平行路由或新的任务监控页面。

### 库与框架要求

- 当前仓库固定技术栈：Next.js `16.1.6`、React `19.2.3`、TypeScript strict、Prisma `6.19.2`、Better Auth `1.4.18`、Tailwind CSS `4.2.2`、Vitest `4.0.18`、Zod `4.3.6`。
- 本 Story 不做依赖升级。仓库与架构文档都明确要求沿用当前主版本组合，不要因为外部“最新版本”建议而临时升级框架或 ORM。
- 由于当前 schema 还没有执行域实体，本 Story 不需要也不应该做“最新执行框架”选型；重点应放在现有任务与工件模型的稳定追踪、查询和展示。
- 对 AC #3 的实现，优先采用现有 `Task` 字段与 `metadata` 读模型完成列表展示；把真正的执行域细化留给后续 Story。

### 范围边界

本 Story 范围：

- ✅ 在任务详情中稳定展示来源工件类型、名称、路径、层级和返回入口
- ✅ 在 Story 工件详情中查看直接由该 Story 发起的任务列表
- ✅ 支持按任务状态筛选，并按时间倒序展示多次执行历史
- ✅ 为 agent 类型、执行结果摘要、最近活动提供稳定的读取与中文兜底策略
- ✅ 抽取共享层级 helper，减少来源链路展示逻辑重复
- ✅ 补齐对应 action / helper / 组件 / 测试
- ✅ 保证历史查询严格受 `projectId` 和读取权限约束

本 Story 不包含：

- ❌ 不实现 `AgentRun`、`Session`、心跳、tmux、SSE、日志流
- ❌ 不实现 Epic / PRD 级别的跨 Story 聚合执行统计
- ❌ 不实现产物列表、产物详情或执行日志查看（Story 2.4）
- ❌ 不实现执行结果回写到工件链路（Story 2.5）
- ❌ 不实现外部 API、Webhook、事件订阅或审计导出
- ❌ 不引入新的全局状态管理或新的任务来源模型
- ❌ 不为非 `STORY` 工件补做跨层级直接执行历史聚合

### Project Structure Notes

- 当前项目的主信息架构仍以“工作空间 → 项目 → 工件 / 任务详情”为主轴，因此反向执行历史最自然的落点是结构化工件树的详情侧栏，而不是新建一个平行页面。
- `artifact-tree.tsx` 负责选中工件，`artifact-detail-sheet.tsx` 负责工件详情与交互；Story 2.3 应延续这种结构，让“查看工件执行历史”仍然发生在同一交互上下文内。
- 任务详情页已经是项目级路由，因此执行历史列表中的每条记录都应该复用现有任务详情页作为深入查看入口，而不是创建嵌套详情弹层。
- Tailwind 继续遵循当前项目规则：不滥用 arbitrary values，优先使用标准类名与现有 shadcn/ui 组件。
