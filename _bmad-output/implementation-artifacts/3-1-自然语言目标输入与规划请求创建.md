# Story 3.1: 自然语言目标输入与规划请求创建

Status: review

## Story

作为用户，
我希望能通过一句自然语言描述我的目标来发起新的项目规划请求，
以便我无需手动拆解需求就能让系统启动规划流程。

## Acceptance Criteria

1. **Given** 用户在项目上下文中 **When** 用户在规划入口输入一句自然语言目标（如“为项目添加用户反馈收集功能”）并提交 **Then** 系统创建 `PlanningRequest` 记录，包含原始目标文本、项目 ID、发起人和创建时间
   **And** 用户立即看到请求已接收的反馈，包含当前阶段和系统即将执行的步骤（UX-DR2）

2. **Given** 用户提交的目标文本为空或过短 **When** 用户点击提交 **Then** 系统提示需要更明确的目标描述，不创建请求

3. **Given** 规划请求已创建 **When** 用户查看请求状态 **Then** 展示请求的当前阶段（分析中/规划中/待确认/已完成）和预估进度

## Tasks / Subtasks

> **建议实现顺序：** Task 1（建立规划请求真值模型） → Task 2（实现 server action 与读取 helper） → Task 3（接入项目页入口与即时反馈） → Task 4（补最小状态视图） → Task 5（测试与验证）

## Dev Notes

### 当前基线与关键依赖

- 当前仓库已经有 `Task`、`BmadArtifact`、`Writeback`、`AuditEvent` 等执行域对象，但**尚无** `PlanningRequest`。这意味着 3.1 应新增独立规划域，而不是把规划请求挤进执行任务模型。
- `src/app/(dashboard)/workspace/[slug]/project/[projectSlug]/page.tsx` 已经是项目级聚合页面，并加载项目、仓库和工件树；一句话目标入口最适合落在这里，而不是工件侧边栏或任务详情页。
- `src/components/artifacts/artifact-detail-sheet.tsx` 当前承载的是“从工件发起执行任务”路径，其 `goal` 字段语义是执行目标，不是项目级规划请求的原始输入。3.1 必须避免复用这条路径，以免后续 3.2~3.5 无法表达一条规划请求衍生多个工件与任务的事实。
- `src/components/workspace/project-no-repo.tsx` 当前在项目未关联仓库时只展示“导入仓库”提示。如果 3.1 的入口被放进当前项目页的 repo 条件分支里，无仓库项目将完全无法发起规划请求，这与 FR13 / UX-DR2 的产品主路径冲突。
- 当前认证与权限模式已经固化为 `getAuthenticatedSession()` + `requireProjectAccess()`；mutation 后的缓存刷新模式以 `revalidatePath()` 为主。3.1 应复用这些仓库既有规则。
- 当前仓库部分 legacy 组件仍有英文 UI 文案示例（如 Sprint/Stories 视图），但 `AGENTS.md`、`project-context.md` 与 `src/lib/errors.ts` 已把“用户可见文本统一中文”明确为当前源规则。新 Story 不应照抄旧英文文案。

### 默认实现决策

- **新增独立规划域：** 推荐增加 `src/lib/planning/` 与 `src/components/planning/` 目录，而不是继续把规划请求逻辑塞进 `src/lib/tasks/`。这是最符合架构文档“按领域拆分模块”的做法。
- **项目级入口，不新开大页面：** 3.1 只需要“创建请求 + 看见初始状态”，因此最合适的 MVP 是项目页上的输入面板 + 最近请求列表，而不是一开始就建完整规划中心路由。
- **无仓库也可发起：** 一句话规划是“从零到规划”的引导入口，不应依赖仓库或工件树先存在。实现时要让 planning composer 位于 repo 条件判断之外，或直接增强 `ProjectNoRepo`。
- **确定的初始阶段：** 新请求创建后立即进入 `analyzing`，并持久化一条默认进度快照（建议 10%）和下一步说明；不要在 3.1 中等待真实 agent 编排完成后才给用户反馈。
- **原始输入优先保真：** `PlanningRequest` 必须保留用户原话文本，后续如果需要摘要、归类或结构化拆分，再在 3.2~3.5 增量补充；不要在 3.1 过早做不可逆“清洗后覆盖原文”。
- **范围边界清晰：** 3.1 只负责“请求创建与初始状态可见”，不负责 PM Agent 选择、Skill 执行、工件生成、任务派发、tmux 启动或链路详情图。

### 架构一致性要求

- 内部 mutation 继续放在 `src/actions/`，页面只负责读取与组装；不要把 Prisma 写入直接放进 `page.tsx` 或客户端组件。
- 新增共享类型、状态标签和映射应集中定义，例如 `src/lib/planning/types.ts`；不要在 action、组件、测试里重复硬编码 `analyzing => 分析中` 这样的映射。
- 所有写操作都要显式带 `workspaceId` 与 `projectId` 约束，保持与现有多租户边界一致；不要仅凭 `projectId` 或 `planningRequestId` 直接读写。
- 新增错误码时必须写入 `src/lib/errors.ts`，错误文本统一中文；**不要**返回原始数据库错误、英文异常或 `error.message`。
- 如果 3.1 引入审计事件，命名遵循架构文档的过去时/领域语义模式，例如 `planningRequest.created`；不要发明另一套难以复用的事件格式。
- 规划请求列表与创建流程必须仅依赖数据库与权限层，不应耦合 GitHub API、本地 provider 或工件扫描链路；否则项目页的一次简单提交会被错误地绑到仓库可用性上。

### UX 与交互护栏

- 用户提交一句目标后，反馈必须立即可见，且至少包含：当前阶段、下一步动作、预估进度。仅显示“处理中”会直接违背本 Epic 的核心体验。
- 状态表达不要只用颜色区分；至少同时显示中文文本标签和进度数值或进度条。
- 校验失败需要告诉用户如何改，而不是只说“输入无效”。建议直接提示“请至少输入 6 个字符，说明希望系统规划的目标”。
- 项目页应优先展示最近规划请求，而不是跳转到新页面再查看。这能最大化降低从目标输入到状态确认的切换成本。
- 进度是“预估进度”，不是“实际技能执行百分比”。在 3.1 中，进度的价值是传达阶段感，而不是制造精确执行幻觉。

### 最新技术信息

- **Next.js 16 的 `revalidatePath()` 仍然是服务端 mutation 后刷新页面数据的官方方式。** 官方文档明确它只能在 Server Functions/Route Handlers 中调用，不能在 Client Components 中调用；同时它基于**路由文件结构**而不是用户可见 URL 匹配缓存，因此在本仓库里最稳妥的做法仍是直接 revalidate 具体项目路径，而不是泛化或猜测路径模式。
- **Better Auth 在 Next.js 16 下继续支持服务端 `auth.api.getSession({ headers: await headers() })` 模式。** 这与当前仓库 `getAuthenticatedSession()` 的实现一致，因此 3.1 不需要重新封装另一套会话获取方式。
- **Prisma ORM 7 已经带来生成器与 driver adapter 方面的 breaking changes。** 仓库当前已经使用 `prisma-client` generator 且依赖固定在 `6.19.2`；Story 3.1 只需要在当前 6.x 轨道上新增 schema/migration，不应该在需求实现中顺手升级到 Prisma 7。
- **Prisma 官方事务文档仍建议用 `$transaction()` 处理需要原子性的多表写入。** 如果 3.1 同时创建 `PlanningRequest` 和 `AuditEvent`，就应该把它们放进单个事务，避免半成功状态。
- **Zod 4 当前是稳定版。** 结合本仓库的 `strict` TypeScript 与现有 action 习惯，继续使用 Zod 做输入边界校验是最一致的选择。

- ✅ 新增项目级 `PlanningRequest` 记录模型

- ✅ 创建规划请求并展示初始阶段、下一步与预估进度

- ✅ 为权限、校验、中文反馈与状态映射补测试
