# Story 1.8: 授权范围内任务可见性与数据隔离

Status: done

## Story

作为平台用户，
我希望只能看到和操作我被授权的工作空间与项目范围内的任务、记录和产物，
以便不同团队和项目之间的数据严格隔离。

## 验收标准

1. **Given** 用户属于工作空间 A 但不属于工作空间 B **When** 用户尝试访问工作空间 B 的任何资源 **Then** 系统返回 403 或重定向到授权范围内的页面 **And** 任何 API 调用都不返回工作空间 B 的数据

2. **Given** 用户在团队工作空间中具有 MEMBER 角色且被分配到项目 X **When** 用户查看任务列表 **Then** 仅显示项目 X 范围内的任务 **And** 不可见同工作空间下其他未授权项目的任务

3. **Given** 数据隔离机制已实现 **When** 任何数据库查询执行 **Then** 查询条件必须包含 workspaceId 过滤 **And** 跨工作空间或跨项目的隐式数据泄露被阻止（NFR7, NFR8）

4. **Given** 用户在授权范围内查看任务 **When** 页面加载 **Then** 任务列表首屏反馈在 2 秒内完成（NFR1） **And** 页面满足键盘可访问（NFR25）和语义化结构（NFR27）要求

  - [x] 4.1 创建 `src/lib/workspace/page-guard.ts` — 页面级权限检查工具函数：

    import { notFound } from "next/navigation";
    import { getAuthenticatedSession, getWorkspaceBySlug, getWorkspaceMembership } from "@/lib/db/helpers";
    import type { WorkspaceRole } from "@/generated/prisma/client";

    export async function guardWorkspacePage(slug: string) {
      const session = await getAuthenticatedSession();
      if (!session) notFound();

      const workspace = await getWorkspaceBySlug(slug);
      if (!workspace) notFound();

      const membership = await getWorkspaceMembership(workspace.id, session.userId);
      if (!membership) notFound();

      const role = membership.role as WorkspaceRole;
      const isTeam = workspace.type === "TEAM";
      const canManage = role === "OWNER" || role === "ADMIN";

      return { session, workspace, membership, role, isTeam, canManage };
    }

## Dev Notes

### 关键架构约束

- **权限判定在服务端**：所有权限校验在 Server Action、Route Handler 或领域服务层执行，不依赖中间件或客户端判定（架构要求）。中间件仅检查 session cookie 存在性。
- **workspaceId 强制过滤**：所有涉及 workspace 资源的数据库查询必须显式包含 `workspaceId` 条件，禁止隐式跨工作空间读取（NFR7, NFR8）。
- **非成员返回 404**：工作空间页面对非成员用户返回 `notFound()` 而非 403，避免泄露工作空间存在性。Server Action 对非成员返回 `FORBIDDEN`。
- **MVP 项目级权限**：当前版本不做项目级细粒度成员分配（AC #2 中"被分配到项目 X"在 MVP 简化为"属于工作空间即可见所有项目"）。`requireProjectAccess` 签名预留 `projectId` 参数，后续 Story 可扩展为项目级 ACL。
- **Repo 归属独立**：`Repo.userId` 绑定到导入用户而非工作空间。项目详情页通过前置 workspace membership 检查间接保护 Repo 数据访问。

### 权限检查统一模式

```
页面路由（Server Component）：
  guardWorkspacePage(slug) → { session, workspace, membership } | notFound()
  
Server Action：
  getAuthenticatedSession() → session | UNAUTHORIZED
  requireWorkspaceAccess(workspaceId, userId, level) → { membership, role, permissions } | FORBIDDEN
  
API Route Handler（后续扩展）：
  extractAuth(request) → userId | 401
  requireWorkspaceAccess(workspaceId, userId, level) → { membership, role, permissions } | 403
```

### 前序 Story 关键教训（来自 Story 1.1–1.7）

- **Prisma 导入路径**：`import { Prisma, WorkspaceRole } from "@/generated/prisma/client"`（不是 `@prisma/client`）
- **cuid2 Zod 校验**：ID 字段使用 `z.string().cuid2()`
- **测试文件顶部 `import "dotenv/config"`**，必须实现 `assertSafeDatabaseUrl()` 安全检查
- **Vitest 配置文件为 `vitest.config.mts`**（`.mts` 扩展名）
- **错误消息全部中文**：`errors.ts` 中所有新增 `ERROR_MESSAGES` value 为中文
- **Server Actions 返回 `ActionResult<T>`**：统一格式，catch 中用 `sanitizeError()`
- **`revalidatePath()` 在 mutation 后调用**：保持控制面数据一致性
- **Server Action 嵌套禁止**：不能在一个 Server Action 中调用另一个 Server Action
- **Next.js params 是 Promise**：App Router 页面 `params` 需要 `await`
- **数据读取用 cache() helper**：不要在页面中直接 `prisma.xxx.findFirst()`，必须封装到 `helpers.ts` 并用 `cache()` 包装
- **非成员访问返回 notFound() 而非 403**：避免泄露工作空间存在性（Story 1.2–1.7 一致行为）
- **Server Action 中硬编码英文消息（"Not authenticated"/"Invalid input"/"Access denied"）是已知技术债**，本 Story 的新增代码使用中文错误码，不负责迁移旧消息
- **`repo-actions.ts` 中有法语残留消息**（`"Trop de requêtes"` 在 `listUserRepos` 和 `importLocalFolder` 中），属已知技术债，本 Story 不负责修复但新增代码禁止引入非中文消息
- **`archive-project.ts` 中两条法语错误消息**需在 Task 3.2 中修正（本 Story 发现的预存问题，修复不改变外部行为）
