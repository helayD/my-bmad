# Story 1.5: 角色与权限模型

Status: done

## Story

作为组织管理员，
我希望能为团队成员分配不同角色，并让系统按角色严格控制读取、执行和治理三类权限，
以便不同职责的成员只能访问和操作其被授权的功能。

## 验收标准

1. **Given** 管理员在成员管理页面 **When** 管理员为某成员指定角色（OWNER/ADMIN/MEMBER/VIEWER/AUDITOR） **Then** 成员的 `WorkspaceMembership.role` 更新为指定值 **And** 权限立即生效，无需重新登录

2. **Given** 角色权限定义已建立 **When** VIEWER 角色用户尝试执行任务派发操作 **Then** 系统拒绝并返回权限不足错误（FR48）

3. **Given** 角色权限定义已建立 **When** AUDITOR 角色用户访问工作空间 **Then** 用户可以查看状态、里程碑和审计记录 **And** 用户无法触发执行或修改配置（FR50）

4. **Given** 角色权限定义已建立 **When** MEMBER 角色用户在授权项目范围内操作 **Then** 用户拥有读取和执行权限，但不具备治理权限（如修改团队策略、管理成员）

5. **Given** 权限判定逻辑已实现 **When** 任何 Server Action 或 Route Handler 被调用 **Then** 权限校验在服务端执行，不依赖中间件或客户端判定（架构要求） **And** 权限不足时返回结构化错误，遵循 `ActionResult<T>` 格式

## Dev Notes

### 关键架构约束

- **权限模型三层设计**：架构文档明确将权限分为读取（READ）、执行（EXECUTE）、治理（GOVERN）三类。`permissions.ts` 是整个平台未来权限判定的统一入口，后续 Epic 2-9 中的 Server Action 都应调用 `hasPermission()` 而不是散点 `includes()` 检查。
- **服务端权限判定**：权限校验**必须**在 Server Action / Route Handler 层执行，中间件只做 session cookie 存在性检查。架构明确禁止把角色判断下沉到中间件（`src/middleware.ts` 只检查登录态）。
- **权限立即生效**：Session 中不缓存角色信息（Better Auth session 只存 userId），每次 Action 调用时通过 `getWorkspaceMembership()` 查询最新角色，自动保证权限变更立即生效，无需重新登录。
- **无 Schema 变更**：本 Story 不需要 Prisma Schema 迁移。`WorkspaceRole` 枚举（OWNER/ADMIN/MEMBER/VIEWER/AUDITOR）和 `WorkspaceMembership.role` 字段均已在 Story 1.1 中建立。
- **错误消息语言**：所有新增 `ERROR_MESSAGES` 值**必须使用中文**（Story 1.9 已完成语言一致性规范）。禁止使用法语或英语。
- **Prisma 导入路径**：`import { ... } from "@/generated/prisma/client"` — **不是** `@prisma/client`。
- **Zod 版本**：项目使用 Zod 4.x，`import { z } from "zod"`，使用 `z.enum([...])` 而非 `z.nativeEnum(WorkspaceRole)`（与现有 types.ts 风格一致）。

### 权限矩阵（完整定义）

| 操作 | OWNER | ADMIN | MEMBER | VIEWER | AUDITOR |
|------|-------|-------|--------|--------|---------|
| 查看工作空间/项目 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 查看审计记录 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 发起任务执行（未来）| ✅ | ✅ | ✅ | ❌ | ❌ |
| 修改配置/策略 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 管理成员/角色 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 赋予 OWNER 角色 | ✅ | ❌ | ❌ | ❌ | ❌ |

### 角色变更权限规则

```
执行者 OWNER → 可赋予 OWNER/ADMIN/MEMBER/VIEWER/AUDITOR 任意角色
执行者 ADMIN → 可赋予 ADMIN/MEMBER/VIEWER/AUDITOR，不能赋予 OWNER
执行者 MEMBER/VIEWER/AUDITOR → 无权变更角色
```

### 自我变更保护

- 执行者不能修改自己的角色（`actorUserId === targetMembership.userId`）
- 原因：防止管理员误降自己权限导致工作空间无人管理的边缘情况
- UI 层同步不对当前登录用户自己的行显示"更改角色"按钮（防止提交到被服务端拒绝的往返）

### 唯一 OWNER 降级保护

复用 Story 1.4 已有 `CannotRemoveSoleOwnerError`：
- 若目标当前 role 为 OWNER，且 workspace 内只有一个 OWNER，则拒绝变更
- 防止工作空间失去所有 OWNER 导致无人可以管理

### Action 调用流程

```
updateMemberRoleAction(input) 调用 →
  1. getAuthenticatedSession() — 认证
  2. updateMemberRoleInputSchema.safeParse(input) — Zod 校验
  3. getWorkspaceMembership(workspaceId, userId) — 获取执行者 membership
  4. 检查执行者 role 为 OWNER 或 ADMIN → 否则 FORBIDDEN
  5. updateMemberRole({ workspaceId, membershipId, newRole, actorUserId, actorRole })
     ↓ 领域函数内部：
     5a. 查找目标 membership（by membershipId + workspaceId）
     5b. 自我变更检查 → CannotChangeOwnRoleError
     5c. canChangeRole(actorRole, newRole) → CannotAssignOwnerRoleError
     5d. 若目标是唯一 OWNER → CannotRemoveSoleOwnerError
     5e. prisma.workspaceMembership.update(...)
  6. revalidatePath(`/workspace/${slug}/members`)
  7. return { success: true, data: { membershipId, role: newRole } }
```

### 前序 Story 关键教训（来自 Story 1.1-1.4）

- **测试必须连接真实 PostgreSQL 测试数据库**：`import "dotenv/config"` 在测试文件顶部，必须实现 `assertSafeDatabaseUrl()` 安全检查
