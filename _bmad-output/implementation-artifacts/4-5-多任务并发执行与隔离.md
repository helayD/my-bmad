# Story 4.5: 多任务并发执行与隔离

Status: ready-for-dev

## Story

作为系统，
我希望支持同一项目内至少 10 个任务独立并发执行，并保持状态、日志和上下文三类隔离，
以便一个任务的异常不影响其他并发任务。

## Acceptance Criteria

1. **Given** 项目内已有 9 个任务并发执行
   **When** 第 10 个任务进入执行
   **Then** 第 10 个任务成功创建独立的 tmux session 并开始执行（FR23）
   **And** 所有 10 个任务的状态、日志和上下文互不干扰

2. **Given** 10 个任务并发执行中
   **When** 第 3 个任务因异常崩溃
   **Then** 仅第 3 个任务的 ExecutionSession 进入异常状态
   **And** 其余 9 个任务不受影响，继续正常执行
   **And** 第 3 个任务的日志和上下文保留用于排查

3. **Given** 项目并发任务数达到配额上限
   **When** 用户尝试启动新任务
   **Then** 系统将任务排入等待队列，并告知用户当前并发数和预估等待
   **And** 不直接拒绝而是优雅降级（NFR24）

## Tasks / Subtasks

> **建议实现顺序：** Task 1（明确并发真值与排队语义）→ Task 2（把 admission / queue 编排接入 supervisor）→ Task 3（落实异常隔离与保留）→ Task 4（打通读模型、详情与审计）→ Task 5（对齐治理设置与边界）→ Task 6（测试与验证）

  - [ ] 1.7 明确区分"planning handoff 队列顺位"（`src/lib/planning/handoff.ts` / `planning-request-detail-sheet.tsx` 中的 `queuePosition`）与"execution 并发等待队列"。两者仅复用字段命名和 UI 形态，execution queue 的真值来源是 `Task.metadata.executionQueue` + `AgentRun.createdAt`，不是 planning handoff 的关系表。

- [ ] Task 2: 把 admission control、排队与自动补位接入 supervisor 生命周期 (AC: #1, #3)

  - [ ] 2.8 若任务已在队列中，再次触发启动请求时应返回幂等结果并更新最新队列快照，而不是重复创建第二条等待记录或重复启动会话。

### 默认实现决策

- **并发上限优先复用 workspace 治理配置。** FR23 要求平台能力达到至少 10 并发，不等于无视配置强跑到 10。
- **活跃槽位以 `ExecutionSession` relation 为准。** 不要把 task 状态、页面徽章或内存变量当并发事实源。
- **排队中的任务仍属于"已派发未启动"。** 不应被 UI 或审计误写成 `running`。
- **admission claim 必须短事务、可重试、可 OCC。** 任何与 `tmux` 的外部 I/O 仍留在事务外。
- **队列顺位必须可持久化重建。** Supervisor 重启后，系统仍应能从数据库重建当前排队顺序。
- **单任务异常只影响单任务。** 禁止出现"某条 session 崩了，整个项目 running task 一起被改状态"的批量副作用。
- **保留异常摘要而不是完整明文日志。** 4.5 要给后续排障链路留证据，但不能破坏最小暴露原则。
- **不要把 4.5 做成分布式调度器。** 这是单机 self-hosted 执行面的 admission / isolation story，不是多机任务编排平台。

### 前序 Story 情报

- **来自 Story 4.2 的守栏：**
  - `Task.status = "dispatched"` 表示已完成路由、等待 execution supervisor 真正启动

  - 4.4 的 launch 编排已采用两阶段模式，4.5 的 admission 编排应与之一脉相承

### 最新技术信息

- tmux 官方手册当前仍明确：`has-session` 可用于探测目标 session 是否存在；`kill-session -t <session>` 用于销毁指定 session；`new-session` 负责创建新 session。4.5 的 admission / cleanup 仍应复用这些原语。4.4 的 `tmux client.ts` 已封装 `hasSession()` 和 `killSession()`，两者都是幂等的，4.5 直接复用。
- Node.js 官方 `child_process` 文档当前仍强调：`exec()` 通过 shell 执行命令，`execFile()` / `spawn()` 更适合直接传参数数组。4.5 的并发场景尤其不应把 tmux 控制命令都做成 shell 字符串拼接——`tmux client.ts` 已正确使用参数数组。
- Prisma 官方 transactions 文档当前仍把 interactive transaction 定位为 read-modify-write 场景的"escape hatch"，同时强调 idempotent API 与 OCC。4.5 的 admission claim 正是"读当前占用槽位 → 判断 → 写回队列 / 保留启动资格"的典型 read-modify-write，应保持事务短小、可重试、可幂等。当前 4.4 的 `launchTask()` 已使用 `Serializable` 隔离级别，4.5 的 admission 编排应保持一致。
- Next.js 官方 `revalidatePath()` 文档当前仍说明：它用于精确失效特定 path/page/layout 的缓存。4.5 若在启动、排队、补位后需要刷新详情页或项目页，应继续精确 revalidate，而不是依赖客户端手工刷新。
- 当前仓库真实版本基线仍以 `package.json` 与 `_bmad-output/project-context.md` 为准：Next.js `16.1.6`、React `19.2.3`、Prisma `6.19.2`、Zod `4.3.6`。本 story 不做依赖升级。

### 范围边界

**本 Story 包含：**

- ✅ 单机 self-hosted 执行面的并发 admission、等待队列与自动补位
- ✅ 单任务失败隔离，不影响其他并发任务
- ✅ 队列顺位、当前并发数、预计等待的最小可见性
- ✅ 最小日志 / 上下文摘要保留，用于后续调查
- ✅ 与 4.4 `ExecutionSession` 真值的深度整合

**本 Story 不包含：**

- ❌ 实时日志流、原始日志查看与日志摘要聚合平台（Epic 5）
- ❌ 心跳可信度、状态漂移检测与自动恢复全链路（Epic 5 / 6）
- ❌ 多机分布式调度、优先级抢占、公平性调度器
- ❌ 商业化并发配额仪表盘、workspace 级容量报表（Epic 9 / FR64）
- ❌ 完整人工接管工作台、异常调查中心与恢复策略面板（Epic 6）
- ❌ 在 `Task.status` 顶层枚举中新增"queued" / "waiting" 状态（通过 `currentStage` / metadata 摘要表达）

### Project Structure Notes

- 4.5 最自然的落点仍是 `src/lib/execution/supervisor/**` 作为 admission / queue 编排核心，`src/lib/tasks/**` 作为读模型与文案出口，`src/components/tasks/**` / `src/components/planning/**` 作为最小可见性承载面。
- 不建议为 4.5 新建独立"执行队列管理页"主导航；当前任务详情和 planning detail 已足够承接 MVP 可见性。
- `src/lib/audit/**` 应只承载责任链和事件真值，不要把 UI 展示文案和审计 payload 混在一起。
