# Story 5.1: 任务状态机与细粒度状态流转

Status: ready-for-dev

## Story

作为平台开发者，
我希望建立统一的任务执行状态机，覆盖从计划到回写的完整生命周期，
以便所有状态变化都经过受控流转且可追溯。

## Acceptance Criteria

1. **Given** `Task` 领域模型已建立
   **When** 状态机定义完成
   **Then** 状态机覆盖以下关键状态（全部使用小写，与 Prisma schema 一致）：`planned`、`dispatched`、`starting`、`running`、`waiting_for_input`、`recovering`、`awaiting_takeover`、`completed`、`failed`、`terminated`、`writeback_pending`、`writeback_done`（FR42）
   **And** 每次状态变化产生 `TaskStateEvent` 记录，包含 `fromStatus`、`toStatus`、`trigger`、`timestamp`、`reason`

2. **Given** 任务处于 `running` 状态
   **When** 系统尝试将其转为 `planned`
   **Then** 状态机拒绝非法转换，记录拒绝事件
   **And** 当前状态保持不变

3. **Given** 状态变化发生
   **When** 查看任务详情
   **Then** 展示状态流转时间线，每个节点包含状态、时间和触发原因（FR31）

## Tasks / Subtasks

> **建议实现顺序：** Task 1（定义状态机核心类型与规则）→ Task 2（实现状态转换服务）→ Task 3（打通状态机与现有状态变更入口）→ Task 4（建立 TaskStateEvent 事件记录）→ Task 5（更新 UI 状态展示）→ Task 6（补充回归测试）

- [ ] 2.1 在 `src/lib/execution/state-machine/validator.ts` 中实现状态转换验证器：

  ```typescript
  import { TaskStatus, VALID_TRANSITIONS, STATUS_LABELS, TASK_STATUS_VALUES } from "./types";

  // 验证状态转换是否合法
  export function isValidTransition(
    fromStatus: string,
    toStatus: string
  ): boolean {
    const transitions = VALID_TRANSITIONS[fromStatus as TaskStatus];
    if (!transitions) return false;
    return transitions.includes(toStatus as TaskStatus);
  }

  // 获取转换失败的原因
  export function getTransitionError(
    fromStatus: string,
    toStatus: string
  ): string {
    if (!TASK_STATUS_VALUES.includes(fromStatus as TaskStatus)) {
      return `未知的状态：${fromStatus}`;
    }
    if (!TASK_STATUS_VALUES.includes(toStatus as TaskStatus)) {
      return `未知的目标状态：${toStatus}`;
    }
    if (!isValidTransition(fromStatus, toStatus)) {
      const fromLabel = STATUS_LABELS[fromStatus as TaskStatus]?.zh ?? fromStatus;
      const toLabel = STATUS_LABELS[toStatus as TaskStatus]?.zh ?? toStatus;
      return `状态机拒绝非法转换：从「${fromLabel}」不能直接转换到「${toLabel}」`;
    }
    return "";
  }

  // 获取允许的转换目标列表
  export function getAllowedTransitions(fromStatus: string): TaskStatus[] {
    return VALID_TRANSITIONS[fromStatus as TaskStatus] ?? [];
  }

  // 检查是否为终态
  export function isTerminalStatus(status: string): boolean {
    const terminalStatuses: TaskStatus[] = ["completed", "failed", "terminated", "writeback_done"];
    return terminalStatuses.includes(status as TaskStatus);
  }

  // 检查是否为恢复相关状态
  export function isRecoveryStatus(status: string): boolean {
    const recoveryStatuses: TaskStatus[] = ["recovering", "awaiting_takeover"];
    return recoveryStatuses.includes(status as TaskStatus);
  }

  // 检查是否为进行中状态
  export function isActiveStatus(status: string): boolean {
    const activeStatuses: TaskStatus[] = [
      "planned", "pending", "dispatched", "starting",
      "running", "waiting_for_input",
    ];
    return activeStatuses.includes(status as TaskStatus);
  }

  // 检查是否需要回写
  export function needsWriteback(status: string): boolean {
    const writebackTriggers: TaskStatus[] = ["completed", "failed", "terminated"];
    return writebackTriggers.includes(status as TaskStatus);
  }
  ```

- [ ] 2.2 在 `src/lib/execution/state-machine/transitioner.ts` 中实现状态转换执行器：

  ```typescript
  import { TransitionInputSchema, type TransitionInput, type TransitionOutcome, type TransitionResult } from "./types";
  import { isValidTransition, getTransitionError, isTerminalStatus, isRecoveryStatus } from "./validator";
  import { deriveStageContext } from "./context";
  import { recordTaskStateEvent } from "./event-recorder";
  import { triggerStateTransitionSideEffects } from "./side-effects";
  import { sanitizeError } from "@/lib/errors";
  import { prisma } from "@/lib/db";

  // 主转换函数
  export async function transitionTask(
    input: TransitionInput
  ): Promise<TransitionResult> {
    try {
      // 1. 解析和验证输入
      const parsed = TransitionInputSchema.safeParse(input);
      if (!parsed.success) {
        return {
          success: false,
          error: "输入参数无效",
          code: "TRANSITION_FAILED",
        };
      }

      const { taskId, toStatus, trigger, reason, actorType, actorId } = parsed.data;

      // 2. 获取当前任务状态
      const task = await prisma.task.findUnique({
        where: { id: taskId },
        select: { status: true, workspaceId: true, projectId: true, currentAgentRunId: true },
      });

      if (!task) {
        return {
          success: false,
          error: "任务不存在",
          code: "TASK_NOT_FOUND",
        };
      }

      const fromStatus = task.status as string;

      // 3. 验证转换合法性
      if (!isValidTransition(fromStatus, toStatus)) {
        const errorMessage = getTransitionError(fromStatus, toStatus);

        // 记录拒绝事件
        await recordTaskStateEvent({
          taskId,
          agentRunId: task.currentAgentRunId,
          fromStatus: fromStatus as any,
          toStatus: toStatus as any,
          trigger: trigger as any,
          reason: errorMessage,
          actorType: actorType as any,
          actorId: actorId ?? null,
          rejected: true,
        });

        return {
          success: false,
          error: errorMessage,
          code: "INVALID_TRANSITION",
        };
      }

      // 4. 计算新的上下文文案
      const stageContext = deriveStageContext(toStatus as any, trigger as any);

      // 5. 在事务中执行状态更新和事件记录
      const [updatedTask, event] = await prisma.$transaction(async (tx) => {
        // 更新任务状态
        const updated = await tx.task.update({
          where: { id: taskId },
          data: {
            status: toStatus,
            currentStage: stageContext.currentStage,
            currentActivity: stageContext.currentActivity,
            nextStep: stageContext.nextStep,
            updatedAt: new Date(),
          },
        });

        // 创建状态事件记录
        const stateEvent = await tx.taskStateEvent.create({
          data: {
            taskId,
            agentRunId: task.currentAgentRunId ?? null,
            fromStatus,
            toStatus,
            trigger,
            reason: reason ?? null,
            actorType,
            actorId: actorId ?? null,
          },
        });

        return [updated, stateEvent];
      });

      // 6. 触发副作用（不阻塞主流程）
      triggerStateTransitionSideEffects({
        taskId,
        agentRunId: task.currentAgentRunId ?? undefined,
        fromStatus: fromStatus as any,
        toStatus: toStatus as any,
        trigger: trigger as any,
      }).catch((err) => {
        console.error("[StateMachine] Side effect error:", err);
      });

      return {
        success: true,
        data: {
          taskId,
          fromStatus: fromStatus as any,
          toStatus: toStatus as any,
          eventId: event.id,
        },
      };
    } catch (error) {
      console.error("[StateMachine] Transition error:", error);
      return {
        success: false,
        error: sanitizeError(error, "DB_ERROR"),
        code: "TRANSITION_FAILED",
      };
    }
  }

  // 简化的检查函数（不执行转换）
  export function canTransition(fromStatus: string, toStatus: string): TransitionOutcome {
    const allowed = isValidTransition(fromStatus, toStatus);
    return {
      allowed,
      fromStatus: fromStatus as any,
      toStatus: toStatus as any,
      errorMessage: allowed ? undefined : getTransitionError(fromStatus, toStatus),
    };
  }
  ```

  ```typescript
  export async function getTaskStateTransitionSummary(taskId: string): Promise<{

    rejectedTransitions: number;
    lastTransition: { fromStatus: string; toStatus: string; createdAt: Date } | null;
    currentStateAge: string;  // 如 "2小时前"
  }> {
    const [events, task] = await Promise.all([

      prisma.task.findUnique({

        select: { status: true, updatedAt: true },

    const rejectedCount = await prisma.taskStateEvent.count({
      where: { taskId, rejected: true },
    });

    const lastEvent = await prisma.taskStateEvent.findFirst({
      where: { taskId },
      orderBy: { createdAt: "desc" },
      select: { fromStatus: true, toStatus: true, createdAt: true },
    });

    // 计算状态持续时间
    const stateAge = task?.updatedAt
      ? formatDistanceToNow(new Date(task.updatedAt), { addSuffix: true, locale: zhCN })
      : "未知";

    return {
      totalTransitions: events,
      rejectedTransitions: rejectedCount,
      lastTransition: lastEvent
        ? { fromStatus: lastEvent.fromStatus, toStatus: lastEvent.toStatus, createdAt: lastEvent.createdAt }
        : null,
      currentStateAge: stateAge,
  ```

1. **状态值必须小写**：所有状态枚举值必须使用小写字符串（`planned`、`running`），与 Prisma schema 一致。**不要**使用大写（`PLANNED`、`RUNNING`）。

2. **状态机是唯一权威**：所有 `Task.status` 变更必须通过 `transitionTask()`，**禁止**直接调用 `prisma.task.update()` 修改状态。

3. **现有状态映射**：`in-progress` → `running`，`review` 和 `blocked` 在新状态机中不再作为主状态存在。

4. **与 ExecutionSession 联动**：Task 状态与 Session 状态有明确映射关系（见 Task 1.10）。

5. **Prisma 索引语法**：`@@index([workspaceId = task.workspaceId, createdAt])` 是无效语法，应使用单独索引或通过 relation 查询。

6. **Zod 输入校验**：所有状态转换入口必须使用 `TransitionInputSchema` 校验输入。

7. **原子事务**：状态更新和事件记录必须在同一个 Prisma transaction 中。

8. **中文 UI 文本**：所有用户可见文本必须中文，`sanitizeError()` 处理错误消息。

### 核心实现目标

- Story 5.1 的核心是建立**统一状态机**，约束和记录任务从"已计划"到"已回写"的完整生命周期状态流转。
- 状态机必须作为**唯一的事实源**来验证和执行所有状态转换。
- 状态转换必须产生**可追溯的事件记录**（TaskStateEvent）。
- 状态机必须与现有代码**平滑集成**，不破坏已实现的 Epic 4 执行链路。

### 默认实现决策

- **状态机是权威验证层。** 所有状态变更必须经过状态机的合法性验证，不允许绕过。
- **状态转换必须原子且可追溯。** 每次转换同时更新 Task 记录和创建 TaskStateEvent 记录。
- **状态语义必须与 UX 规范对齐。** 每个状态必须有中文描述、状态分类和下一步建议。
- **状态转换与副作用分离。** 状态转换器负责更新状态，副作用通过事件机制触发。
- **兼容现有代码基线。** 状态机必须能覆盖 `dispatch.ts`、`redispatch.ts`、`launch.ts`、`lifecycle.ts`、`writeback.ts` 中的现有状态变更逻辑。

### 前序 Story 情报

#### Story 4.1 - 任务定义与执行意图提交

- `Task.status = "planned"` 是任务的初始状态
- `Task.currentAgentRunId` 在派发时设置
- `Task.metadata` 中存储执行意图摘要

#### Story 4.2 - Agent 自动路由与选择

- `AgentRun.status = "dispatched"` 表示已派发但未启动
- 路由决策记录在 `AgentRun.decisionSource` 和 `AgentRun.routingSummary`

#### Story 4.3 - 路由策略调整与重新派发

- 重新派发需要终止当前运行并创建新 AgentRun
- 状态转换必须处理运行中 → 终止 → 重新派发的链路

#### Story 4.4 - tmux 后台执行会话

- `ExecutionSession` 与任务状态紧密关联
- 会话启动/完成/终止会影响任务状态
- 需要定义 `Task.status` 与 `ExecutionSession.status` 的联动规则

#### Story 4.5 - 多任务并发执行与隔离

- `pending` 状态用于排队中的任务
- 容量管理可能触发 `dispatched` ↔ `pending` 转换

#### Story 4.6 - 执行边界与上下文安全控制

- 边界准备失败不应让任务停留在 `dispatched`，应正确处理失败状态
- 边界违规可能触发 `running` → `awaiting_takeover` 转换
