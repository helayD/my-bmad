# Story 1.1: 统一领域模型与 Workspace/Project 数据基础
 
 Status: done
 
 <!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->
 
 ## Story

作为平台开发者，
我希望在现有 brownfield 基座上建立 Workspace、Project、WorkspaceMembership 核心 Prisma 模型与迁移，
以便后续所有工作空间、项目、成员和权限功能拥有统一的数据基础。

## 验收标准

1. **Given** 现有 Prisma schema 包含 User、Session、Account、Verification、Repo 模型 **When** 执行 Prisma 迁移 **Then** 数据库新增 `workspaces`、`projects`、`workspace_memberships` 表，且现有表结构不受影响。

2. **Workspace 模型字段要求：**
   - `id` — `String @id @default(cuid())`
   - `name` — `String`
   - `slug` — `String @unique`
   - `type` — 枚举 `WorkspaceType`（`PERSONAL` / `TEAM`）
   - `ownerId` — `String`，外键关联 `User`
   - `settings` — `Json?`（承载后续策略配置）
   - `createdAt` — `DateTime @default(now())`
   - `updatedAt` — `DateTime @updatedAt`
   - `@@index([ownerId])`
   - 表名映射 `@@map("workspaces")`

3. **Project 模型字段要求：**
   - `id` — `String @id @default(cuid())`
   - `name` — `String`
   - `slug` — `String`
   - `workspaceId` — `String`，外键关联 `Workspace`
   - `repoId` — `String?`，可选关联 `Repo`
   - `status` — `String @default("active")`
   - `settings` — `Json?`
   - `createdAt` — `DateTime @default(now())`
   - `updatedAt` — `DateTime @updatedAt`
   - `@@unique([workspaceId, slug])`（同一 Workspace 内 slug 唯一）
   - `@@index([workspaceId])`
   - `@@index([repoId])`
   - 表名映射 `@@map("projects")`

4. **WorkspaceMembership 模型字段要求：**
   - `id` — `String @id @default(cuid())`
   - `workspaceId` — `String`，外键关联 `Workspace`
   - `userId` — `String`，外键关联 `User`
   - `role` — 枚举 `WorkspaceRole`（`OWNER` / `ADMIN` / `MEMBER` / `VIEWER` / `AUDITOR`）
   - `createdAt` — `DateTime @default(now())`
   - `updatedAt` — `DateTime @updatedAt`
   - `@@unique([workspaceId, userId])`（同一用户在同一 Workspace 只能有一条 membership）
   - `@@index([workspaceId])`
   - `@@index([userId])`
   - 表名映射 `@@map("workspace_memberships")`

5. **外键约束与级联行为：**
   - `Workspace.ownerId` → `User.id`，`onDelete: Restrict`（不允许删除仍拥有 workspace 的用户）
   - `Project.workspaceId` → `Workspace.id`，`onDelete: Cascade`
   - `Project.repoId` → `Repo.id`，`onDelete: SetNull`
   - `WorkspaceMembership.workspaceId` → `Workspace.id`，`onDelete: Cascade`
   - `WorkspaceMembership.userId` → `User.id`，`onDelete: Cascade`

6. **User 模型扩展：** 在 `User` 上增加反向关系字段：`ownedWorkspaces`、`workspaceMemberships`。

7. **Repo 模型扩展：** 在 `Repo` 上增加反向关系字段：`projects`。

8. **迁移可通过 `prisma migrate dev` 在本地数据库上正确执行，生产使用 `prisma migrate deploy`。**

9. **迁移不应破坏现有数据。** 所有新表和新字段均为新增或可空，不修改现有列。

## Dev Notes

 ### 关键架构约束

- **Brownfield 安全第一**：当前 schema 有 User、Session、Account、Verification、Repo 五个模型且已有 3 个已执行迁移。本次变更只做 **增量添加**，绝不修改现有模型的已有字段或删除任何列。
- **Prisma 版本**：项目使用 Prisma `^6.19.2`，client 输出到 `src/generated/prisma`（见 `prisma.config.ts` 与 `schema.prisma` generator 配置）。
- **枚举处理**：Prisma 枚举值使用 `SCREAMING_SNAKE_CASE`；Prisma 6.x 的 `enum` 会自动映射为 PostgreSQL 原生 `CREATE TYPE`，**不需要**为枚举名加 `@@map()`——只需保持 Prisma 层命名即可。
- **表名映射**：所有模型必须使用 `@@map()` 映射到复数 `snake_case` 表名（如 `workspaces`、`projects`、`workspace_memberships`），保持与现有 `users`、`sessions`、`accounts`、`verifications`、`repos` 一致。
- **字段命名**：Prisma 层使用 `camelCase`（如 `workspaceId`、`ownerId`）。
- **外键索引**：**所有外键字段**必须显式创建 `@@index()`（包括 `ownerId`、`workspaceId`、`repoId`、`userId`），与现有模型惯例一致。
- **JSON 字段**：`settings` 使用 `Json?` 类型，后续用 Zod 4 在应用层校验 settings 结构。本 Story 不需要实现 Zod schema，只需确保字段类型正确。
- **slug 字段**：Workspace 的 `slug` 为全局唯一 `@unique`，Project 的 `slug` 在 Workspace 内唯一 `@@unique([workspaceId, slug])`。slug 的生成逻辑（基于用户名/团队名）留给 Story 1.2/1.3 在应用层实现，本 Story 只需确保约束正确。

### 统一领域模型预览

本 Story 建立的三个模型是整个平台统一领域模型的第一层。后续 Epic 将在此基础上扩展以下模型（本 Story 不需要实现，但了解上下文有助于正确设计）：

| 后续模型 | 关联点 | 所属 Epic |
|----------|--------|-----------|
| BmadArtifact | `projectId` → Project | Epic 2 |
| Task | `projectId` → Project, `workspaceId` → Workspace | Epic 4 |
| AgentRun | `taskId` → Task | Epic 4 |
| ExecutionSession | `agentRunId` → AgentRun | Epic 4 |
| AuditEvent | `workspaceId`, `projectId` | Epic 7 |

因此：
- `Workspace.id` 和 `Project.id` 将作为几乎所有后续实体的根归属字段，类型必须是 `cuid()` String。
- `WorkspaceMembership.role` 枚举值集合必须完整（OWNER/ADMIN/MEMBER/VIEWER/AUDITOR），后续权限判定将直接引用这些枚举值。

### 现有模型的反向关系快照（必须保留，只追加）

当前 User 模型已声明的反向关系：
```prisma
  sessions      Session[]
  accounts      Account[]
  repos         Repo[]
```
本 Story 需要**追加**（不是替换）：
```prisma
  ownedWorkspaces       Workspace[]
  workspaceMemberships  WorkspaceMembership[]
```

当前 Repo 模型没有反向关系字段。本 Story 需要追加：
```prisma
  projects  Project[]
```

- **Repo 模型**已有 `userId` 外键。新增的 `Project.repoId` 是可选的，因为项目可以先创建再关联仓库，也可以不关联仓库。
- **Repo 的 `@@unique([userId, owner, name])`** 约束不受影响——Project 通过 `repoId` 间接关联，不改变 Repo 的唯一性约束。
- **User.repos** 反向关系已存在，不要重复声明或修改。

### 范围边界与禁止事项

**本 Story 范围仅限：** Prisma schema 变更（枚举 + 模型 + 反向关系）、迁移生成与执行、模型回归测试。**不涉及任何应用层代码。**

- ❌ 不要修改 User、Session、Account、Verification、Repo 的现有字段
- ❌ 不要在 User 上新增 `workspaceId` 或 `currentWorkspaceId`（后续通过 membership 查询）
- ❌ 不要实现 Server Actions、API 路由、UI 页面或 Zod schema（留给 Story 1.2+/1.6）
- ❌ 不要把枚举定义为 TypeScript 联合类型——必须使用 Prisma 原生 `enum`
- ❌ 不要使用 `prisma db push`——必须使用 `prisma migrate dev` 生成正式迁移文件
