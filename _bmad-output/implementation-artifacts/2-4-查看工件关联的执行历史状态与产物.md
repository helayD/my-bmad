# Story 2.4: 查看工件关联的执行历史、状态与产物

Status: in-progress

## Story

作为用户，
我希望能在任意 BMAD 工件上查看其关联的完整执行历史、当前状态和产出物，
以便从工件视角理解执行进展与成果。

## Acceptance Criteria

1. **Given** 用户查看某个 Story 工件详情 **When** 用户切换到“执行历史”标签 **Then** 展示该 Story 关联的所有任务、Agent Run 和产物列表（FR11） **And** 每条记录包含任务状态、执行时间、agent 类型、关键产物摘要

2. **Given** 某个任务已产出代码变更或文件产物 **When** 用户在执行历史中点击该任务 **Then** 展示产物列表，包含产物类型、文件路径、生成时间 **And** 提供查看产物详情或跳转到任务详情的入口

3. **Given** 用户查看 Epic 级别工件 **When** 用户查看执行历史 **Then** 聚合展示该 Epic 下所有 Story 的执行状态分布（已完成/执行中/待执行/失败） **And** 提供按 Story 展开查看的能力

### 读取与展示优先级

- `agentType` / `agentTypeLabel` 读取优先级建议为：真实 `AgentRun.agentType` 或等价字段 → `Task.metadata.agentTypeLabel` / `Task.metadata.agentType` → `待分配`。
- `executionTime` 读取优先级建议为：真实 `AgentRun.startedAt` / `completedAt` → 轻量 metadata 中明确记录的执行时间字段（若存在） → 显示“尚未开始执行”；**不要**把 `Task.createdAt` 冒充为执行时间。
- `artifactSummary` 读取优先级建议为：真实产物引用聚合（数量、最近生成时间、核心路径） → 轻量 metadata 摘要字段（若已存在） → `暂无关键产物`。
- 任务展开后的 `artifacts` 列表读取优先级建议为：真实执行产物记录 → 轻量 metadata 快照（若已存在且字段可信） → 空数组。
- Story 级聚合排序继续按最近相关任务倒序；Epic 级聚合以 Story 为单位归约，不以原始 Task 数量直接充当状态分布。

### 聚合状态映射规则

- `无任务` → `待执行`
- `存在进行中的任务（例如 Task.status = in-progress）` → `执行中`
- `最近相关任务状态为 review 或 done，且没有更高优先级的进行中任务` → `已完成`
- `最近相关任务状态为 blocked` → `失败`
- `最近相关任务状态为 pending` → `待执行`

> 这套映射应落在共享 helper 中，后续如果 Epic 4 / Epic 5 引入更细粒度的执行状态机，再集中替换，不要在多个组件中散落 hardcode。

### 实现护栏

- **不要创建新的平行历史中心。** Story 2.4 的主入口仍然是 `artifact-detail-sheet.tsx` 中的工件详情上下文，而不是新建 `/history`、`/runs` 或控制塔页面。
- **不要伪造执行事实。** 当前没有 `AgentRun` 或产物记录时，必须明确显示“暂无记录”，而不是把 Task 行直接改名成 Run 或随意生成假的运行 ID、文件路径、生成时间。
- **不要把 PRD / TASK 也顺手做成聚合历史。** 本 Story 的新增能力严格锁定在 Story 直连历史和 Epic 聚合视图；其他工件类型保持明确的只读提示即可。
- **不要在本 Story 中实现 tmux、执行监督器、SSE、心跳、Writeback 或完整证据链。** 这些分别属于 Epic 4、Epic 5 和 Story 2.5 / 5.10。
- **不要修改现有 Task 状态枚举来迁就聚合视图。** Epic 视图需要的“已完成 / 执行中 / 待执行 / 失败”应通过共享映射派生，而不是直接扩充数据库状态枚举并影响现有任务流。
- **不要把大体量产物内容写进 `Task.metadata`。** 如果确实需要临时 fallback，只能保存轻量引用，不得保存完整 diff、日志正文或文件内容。
- **不要破坏 Story 2.2 的“发起执行”主路径。** 执行历史标签是补充上下文，不是替换当前创建任务入口。
- **不要把权限判断下沉到客户端。** 所有执行历史、聚合统计、任务展开明细的读取边界都必须在服务端按 `workspaceId + projectId` 限制。
- **不要让客户端树结构成为数据真值。** `buildArtifactTree()` 适合做展示组织，但 descendant Story 收集、聚合状态和历史查询结果必须由服务端基于数据库关系计算。

### 技术要求

- 所有读取入口继续放在 `src/actions/task-actions.ts` 中或复用其共享领域 helper；返回值统一为 `ActionResult<T>`，错误统一使用 `sanitizeError()`。
- 读取 helper 优先放入 `src/lib/db/helpers.ts` 并继续使用 `cache()` 包装，以匹配当前项目对项目页、任务页和工件页的读取模式。
- 组件状态仍优先使用局部客户端状态；当前侧栏内的标签切换、筛选、展开项无需为此引入全局状态管理。
- 继续复用现有 `TASK_STATUS_LABELS`、`TASK_STATUS_VALUES`、`toLocaleString("zh-CN", { hour12: false })` 等已有中文展示约定；不要复制一套新的状态 label 映射或日期格式器。
- 若使用 Tabs、Accordion、Collapsible 等交互基础组件，优先复用项目已有 shadcn/ui 组件目录，而不是额外引入新 UI 依赖。
- 只有 mutation 才需要 `revalidatePath()` 或 `revalidateTag()`；执行历史读取是只读路径，不要无意义地触发缓存失效。

### 架构一致性要求

- 架构文档明确要求 PostgreSQL 继续作为系统事实源，因此 Story / Epic 执行历史和聚合结果必须来自数据库查询与共享领域 helper，而不是前端缓存拼装或临时 URL 数据。
- 架构文档同时强调“核心实体 + 状态投影 + 事件审计”分层；本 Story 可以先交付 artifact-centric 的**读取投影**，但不要在这里把未来的执行事件流、日志轨迹或审计导出一并塞进 Task 详情。
- 所有 `Task`、未来的 `AgentRun`、产物引用与聚合结果都必须显式受 `workspaceId` 与 `projectId` 保护，禁止隐式跨项目读取。
- UX 文档要求状态“说真话”，因此任何缺失的运行时间、产物时间、执行链路都应该在界面上以缺失状态呈现，而不是被伪装成已知值。
- Story 5.10 将来还要实现 `Evidence Chain Viewer`，所以 Story 2.4 的 payload 设计应对 `Story -> Task -> AgentRun -> Artifact` 关系预留结构，但不要在本 Story 里抢先做完整图谱组件。

### 库与框架要求

- 当前仓库技术栈为：Next.js `16.1.6`、React `19.2.3`、TypeScript strict、Prisma `6.19.2`、Tailwind `4.2.2`、Vitest `4.0.18`、Zod `4.3.6`。
- 官方 Next.js 文档当前仍将 `use server` 文件级 Server Functions 作为客户端导入执行服务端逻辑的标准做法，并强调安全、认证与授权应在服务端完成；这与本项目现有 `task-actions.ts` 模式一致。
- 官方 React 文档当前仍推荐 `useTransition()` 处理客户端的非阻塞状态切换和 pending UI；当前侧栏中的筛选、标签切换、局部展开交互应继续沿用这一思路，而不是引入额外状态库。
- 官方 Prisma 升级文档说明从 6.x 升到 7.x 存在模块格式、adapter、环境变量加载等 breaking changes；因此本 Story **不要**把 Prisma 大版本升级与业务实现耦合。
