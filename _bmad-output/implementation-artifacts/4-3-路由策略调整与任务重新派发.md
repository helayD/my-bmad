# Story 4.3: 路由策略调整与任务重新派发

Status: review

## Story

作为授权用户，
我希望能调整任务的路由策略或重新派发任务到不同的 agent，
以便在初次路由不合适时进行纠正。

## Acceptance Criteria

1. **Given** 任务已派发但尚未开始执行（状态为 `DISPATCHED`）
   **When** 授权用户选择“重新派发”并指定新的 agent 类型
   **Then** 系统创建新的 `AgentRun` 作为替代 run（不要原地改旧 run 的 agent 类型）（FR20）
   **And** 记录路由调整的审计事件

2. **Given** 任务已在执行中（当前存在活跃 run / session，任务处于运行态）
   **When** 授权用户请求重新派发
   **Then** 系统先终止当前执行会话，保留已有上下文
   **And** 创建新的 `AgentRun` 并重新派发
   **And** 展示确认对话框说明重新派发的影响（UX-DR17）

3. **Given** `VIEWER` 角色用户查看任务
   **When** 用户尝试调整路由或重新派发
   **Then** 系统拒绝操作，按角色权限返回权限不足错误

### 默认实现决策

- **4.3 一律创建 replacement `AgentRun`，不原地改旧 run。** 这是对 4.2 append-only 决策的延续，也是避免审计链断裂的唯一稳妥方式。
- **当前 active run 必须有单一权威指针。** 推荐 `Task.currentAgentRunId`（或等价一等字段）；不要靠 `latest createdAt` 猜测谁是当前 run。
- **运行中 reroute 是“两阶段编排”，不是“一个大事务”。** 停止旧会话属于外部 I/O，不应长时间占用 Prisma 事务连接。
- **reroute 成功后的 Task 真值回到 `dispatched`。** 新 run 还没启动前，页面应明确显示“等待新会话启动”，而不是继续展示“执行中”。
- **不要新增顶层 `Task.status = "rerouting"` / `"terminating"`。** 这类短暂过程更适合落在 `currentStage`、`currentActivity` 或 run 级状态上；否则 3.5、4.2、Epic 5 的既有状态语义会再次分裂。
- **AC 里的 `RUNNING` 表达当前执行态，不代表 4.3 要单独引入顶层 `Task.status = running`。** 在 Epic 5 的细粒度状态机落地前，继续以现有 task status + active run/session truth 表达运行态即可。
- **reroute 是显式治理动作。** 用户必须主动指定目标 agent，并给出可审计的理由摘要；4.3 不做静默自动 reroute。
- **同 agent 改派默认视为 no-op。** 4.3 的目标是纠正错误路由，而不是触发“重新试一次当前 agent”；后者更接近 retry / recovery 语义。
- **权限仍以 `execute` 为准。** 当前产品没有“只有 OWNER/ADMIN 才能 reroute”的额外规则，4.3 不要擅自把 MEMBER 排除掉。

- **来自 Story 3.5 的守栏：** planning detail 和 derived task 列表已经开始以数据库 `Task` 真值驱动 UI，同时对 legacy 数据做诚实降级。4.3 新增 reroute 后也要保持这条“DB truth first, JSON summary second”的路径。
- **来自 Story 4.1 的守栏：** `preferredAgentType` 与 `intentDetail` 应成为 reroute UI 的默认上下文，而不是被完全忽略；不过 4.3 仍要求用户显式确认新 target agent。

### 架构一致性要求

- 所有 mutation 继续遵循 `ActionResult<T>`、Zod 顶部校验、`sanitizeError()` 中文错误清洗，以及 `requireProjectAccess(..., "execute")` 的服务端权限边界。

- reroute 的 route summary、agent label、reason code 和 payload 结构应尽量复用 4.2 的共享 helper；不要在 4.3 单独实现第二套 formatter / mapper / label table。
- 审计事件继续沿用 architecture 约定的过去时、点分命名，如 `task.redispatched`、`agentRun.superseded`；**不要**用 `console.info` 代替正式审计。
- `Control Plane ↔ Execution Plane` 的边界必须保持清晰：UI 不应直接调用 `tmux`；reroute 的“停止旧执行”只能通过 execution/supervisor 领域契约完成。
- relation-first 继续优先于 metadata-summary：`AgentRun` / `ExecutionSession` 是事实源，`Task.metadata` 只保留摘要和 UI 便利字段。
- 缓存刷新继续走服务端 `revalidatePath()`；`router.refresh()` 只能作为客户端补充，不能代替服务端失效。

### 最新技术信息

- Next.js 官方 `revalidatePath()` 文档当前仍明确：它可在 Server Functions / Route Handlers 中调用；用于 Server Function 时，受影响路径会立即更新，而 `router.refresh()` 本身不会失效服务端缓存。4.3 的 reroute action 仍应坚持“服务端 mutation + 精确 `revalidatePath()` + 客户端补充刷新”的组合，而不是只靠客户端局部状态。  
- Prisma 官方 transactions 文档当前仍强调：interactive transaction 要尽量短，避免在事务内执行网络请求或慢操作。4.3 的“停止旧 session → 再创建 replacement run”正是典型的两阶段编排场景，不应把 supervisor / `tmux` 停止动作包进一个长事务里。  
- Prisma 官方同一页也继续把 **idempotent APIs** 与 **optimistic concurrency control** 作为 read-modify-write 的推荐方案。4.3 应使用 `expectedAgentRunId` 或等价并发 token，避免重复点击、两个操作者同时 reroute 时追加出两条新 run。  
- React 官方 `useTransition()` 文档当前仍指出两个与 4.3 直接相关的限制：`await` 之后的状态更新需要再次包进 `startTransition()`；自定义 async transition 还可能出现请求完成顺序反转。4.3 若继续沿用仓库现有 `useTransition` 模式，服务端 OCC 和幂等保护仍然是必需的；如果确认 UI 交互进一步复杂，可以评估 `useActionState` / `<form>` actions 这类 React 官方更稳妥的动作封装。  
- 当前仓库真实版本基线仍以 `package.json` 与 `_bmad-output/project-context.md` 为准：Next.js `16.1.6`、React `19.2.3`、Prisma `6.19.2`、Zod `4.3.6`。本 story 不做依赖升级。
