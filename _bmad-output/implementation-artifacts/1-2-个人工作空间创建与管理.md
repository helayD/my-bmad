# Story 1.2: 个人工作空间创建与管理

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为个人用户，
我希望在注册或首次登录后自动拥有一个个人工作空间，并能查看和管理该空间下的项目列表，
以便我有一个明确的起点来承载 BMAD 项目和执行任务。

## 验收标准

1. **Given** 用户已完成认证且尚无个人工作空间 **When** 用户首次进入控制面 **Then** 系统自动创建一个 `type=PERSONAL` 的工作空间，并将当前用户设为 OWNER **And** 个人工作空间的 slug 基于用户标识自动生成 **And** workspace 的 `name` 设为用户名或 email 前缀的可读形式（如 `"David Zhang's Workspace"`） **And** 同时创建一条 `WorkspaceMembership`（`role=OWNER`）记录

2. **Given** 用户已有个人工作空间 **When** 用户进入工作空间首页 **Then** 显示该空间下的项目列表，包含项目名称、状态和最近活动时间 **And** 提供创建新项目的入口 **And** 空项目列表展示引导提示，说明为什么为空、系统现在能做什么、用户下一步如何开始创建项目（UX-DR26）

3. **Given** 用户查看个人工作空间 **When** 页面加载 **Then** 首屏可见反馈在 2 秒内完成（NFR1）

4. **Given** 用户已有个人工作空间 **When** 用户再次访问控制面 **Then** 系统不重复创建个人工作空间，直接加载已有工作空间

5. **Given** 系统调用 `ensurePersonalWorkspace` **When** 存在并发请求（如多 tab 同时打开） **Then** slug 唯一约束和 membership 唯一约束保证不会产生重复数据，且第二个请求直接返回已创建的工作空间

### 关键架构约束

- **Brownfield 集成**：当前 Dashboard 页面 (`src/app/(dashboard)/page.tsx`) 基于 Repo 维度展示项目。Story 1.2 的工作空间首页是**新增路由** `/workspace/[slug]`，不替换现有 Dashboard 页面。`/` 路由保持不变。
- **数据模型已就绪**：Story 1.1 已创建 `Workspace`、`Project`、`WorkspaceMembership` Prisma 模型和迁移。本 Story 在应用层使用这些模型，**不修改 schema**。
- **自动供应模式**：个人工作空间采用"首次访问时自动创建"模式（lazy provisioning），在 Dashboard layout 中触发。不修改 Better Auth 注册流程。
- **并发安全靠数据库约束**：不做"先查后写"；必须依赖 `slug @unique` + `@@unique([workspaceId, userId])` + `$transaction` + `P2002` 捕获来保证幂等。
- **领域函数 vs Server Action 职责分离**：`ensurePersonalWorkspace` 是纯领域函数（在 `src/lib/workspace/`），不标记 `"use server"`。只有客户端需要调用的变更操作才放入 `src/actions/workspace-actions.ts` 作为 Server Action。Layout 作为 Server Component 可直接调用领域函数。
- **数据读取模式**：读取 helper 放 `src/lib/db/helpers.ts`，使用 `cache()` 去重。
- **前端架构**：工作空间首页和项目列表都是 Server Component；只有需要浏览器状态的交互才标记 `"use client"`。
- **缓存失效**：`ensurePersonalWorkspace` 创建新工作空间后，调用方需要 `revalidatePath("/")` 确保 sidebar 刷新。

### slug 生成规则

```typescript
// 输入: userIdentifier (优先用 email，取 @ 前缀；如果 email 无效则用 name)
// 处理步骤:
// 1. 取 email 的 @ 前缀，或 name
// 2. toLowerCase()
// 3. 替换所有非 [a-z0-9] 字符为 "-"
// 4. 合并连续 "-"、去首尾 "-"
// 5. 截断至 39 字符（为冲突后缀 -xxxx 预留空间，总长 ≤ 44）
// 6. 如果结果为空字符串，回退到 "user"
// 7. 如果 slug 冲突（P2002），追加 "-" + 4位随机 hex
//
// 示例:
//   "user@example.com" → slug: "user", name: "user's Workspace"
//   "David Zhang" (name) → slug: "david-zhang", name: "David Zhang's Workspace"
//   "张三" (非ASCII) → slug: "user" (非ASCII字符被替换为空后回退), name: "张三's Workspace"
//   冲突时 "user" → "user-a3f1"
```

### name 生成规则

```typescript
// Workspace.name 是面向用户的显示名称（必填 String）
// 规则: 取 user.name || email 前缀，追加 "'s Workspace"
// 示例: name="David Zhang" → "David Zhang's Workspace"
//       name=null, email="user@example.com" → "user's Workspace"
```

### 并发冲突处理模式（复用 repo-actions.ts 已有模式）

```typescript
import { PrismaClientKnownRequestError } from "@prisma/client/runtime/library";

try {
  const result = await prisma.$transaction(async (tx) => {
    const workspace = await tx.workspace.create({ ... });
    const membership = await tx.workspaceMembership.create({ ... });
    return { workspace, membership };
  });
  return { ...result, created: true };
} catch (error) {
  if (error instanceof PrismaClientKnownRequestError && error.code === "P2002") {
    // 唯一约束冲突 → 已有其他请求创建成功，回退查询
    const existing = await prisma.workspace.findFirst({
      where: { ownerId: userId, type: "PERSONAL" },
      include: { memberships: { where: { userId }, take: 1 } },
    });
    if (existing) return { workspace: existing, membership: existing.memberships[0], created: false };
  }
  throw error; // 非冲突错误继续抛出
}
```
