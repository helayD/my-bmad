# Story 3.4: 规划结果自动衔接执行链路

Status: review

## Story

作为用户，
我希望规划阶段产出的可执行任务能自动衔接到后续编码执行链路，
以便规划完成后无需手动将每个 Story / Task 逐一派发。

## Acceptance Criteria

1. **Given** 规划执行已完成并产出 Epic → Story → Task 工件 **When** 用户确认规划结果 **Then** 系统识别可执行的 Task 级工件
   **And** 为每个已确认的可执行 Task 创建待派发的执行任务记录，关联到对应 `BmadArtifact` 与 `PlanningRequest`（FR16）
   **And** 执行任务进入 `planned` 状态，等待用户手动派发或后续自动派发策略触发

2. **Given** 团队工作空间配置了“规划完成后自动派发”策略 **When** 用户确认规划结果 **Then** 已确认的可执行任务进入派发准备队列，并按优先级排序
   **And** 若工作空间同时开启“执行前需审批”，系统只标记为“待审批后派发”，不提前创建 `AgentRun` 或宣称已开始编码

3. **Given** 用户不确认某些规划产出 **When** 用户将某些 Story / Task 标记为“暂不执行”后再确认 **Then** 对应工件不会生成执行任务记录，也不会进入派发准备队列
   **And** 规划工件保留，并在当前规划请求的衔接摘要中被标记为“待定”

## Dev Notes

### 当前基线与关键依赖

- Story 3.3 已经建立“规划执行 → 工件写入 → artifact sync”真值链。当前 `executePlanningRequest()` 在全部 planning skill 完成后会把 `PlanningRequest.status` 更新为 `awaiting-confirmation`，并写入 `artifactSummary`、`generatedArtifactCount` 与 execution step 结果；3.4 应从这个确认节点继续向前，而不是重写 3.3。
- Story 3.3 还通过 `projectStoriesFromEpicsDocument()` 把 `epics.md` 投影成 `_bmad-output/implementation-artifacts/*.md` 的实现故事 stub，并统一写入 `Status: planned`。这意味着 3.4 的“可执行任务”上游并不是重新解析 `epics.md`，而是复用已经被 scanner 识别出来的 `STORY` / `TASK` artifact。
- 当前 `src/lib/artifacts/scanner.ts` 会从 Story 文件里的 checkbox 任务自动生成 `TASK` artifact，路径格式为 `.../story.md#task-N`。3.4 必须复用这套约定，才能与 2.1 / 2.2 / 2.4 / 2.5 的来源追踪、任务上下文与历史视图保持一致。
- 当前 `Task` 模型仍是 2.2 时代的最小执行任务事实源，只保存 `status = pending / in-progress / review / done / blocked`，且没有 `planningRequestId`。3.4 的一个核心补强点就是把“planning 请求衍生任务”显式接回 Task 域，而不是靠 `metadata` 拼出来。
- 当前从 artifact 手动建任务的链路已经存在：`createTaskFromArtifactAction()` 会复用 `buildTaskCreationContext()`、`buildTaskSourceContextSnapshot()`、`Task.sourceArtifactId` 与 `metadata.sourceContext`。3.4 应沿着这条链路批量生成 planning-derived Task，而不是另做第二套任务字段组装逻辑。
- 当前 `Workspace.settings` 只包含 `agentRoutingPreference`、`maxConcurrentTasks`、`autoRecoveryEnabled`、`requireApprovalBeforeExecution`。验收标准里的“规划完成后自动派发”策略尚未存在，因此 3.4 需要在现有治理设置 JSON 上扩一个明确字段，而不是新建平行配置入口。
- 当前 `PlanningRequestList` 只支持“重新分析”与“执行规划”，尚无“确认并生成执行任务”的 UI；而 direct-execution 请求已经会进入 `execution-ready`。3.4 要补的是 planning 分支的确认衔接，不要破坏 3.2 的 direct-execution handoff。
- 当前 `BmadArtifact.metadata` 在扫描同步时会被受控更新，运行态手工标记如果直接写在 metadata 顶层会被覆盖。因此 3.4 的“待定 / 已衔接 / 派发模式”必须落在 `PlanningRequest` 级摘要或等价显式字段上。

### 默认实现决策

- **确认节点在 `awaiting-confirmation`。** 3.4 的执行衔接必须从 3.3 已建立的确认节点出发，只有在用户确认后才创建 execution Task；不要在 `executePlanningRequestAction()` 完成时自动建任务，避免夺走用户的“暂不执行”选择权。
- **可执行真值来自 `BmadArtifact TASK`。** 以数据库里已同步的 `TASK` artifact 为准，再复用 `buildTaskCreationContext()` 提取标题、目标、摘要和来源上下文；不要在前端基于 Markdown 文本再猜一遍可执行项。
- **planning-confirmed 请求进入 `execution-ready`。** direct-execution 与 planning-confirmed 最终都应汇入同一个“待进入执行”状态，但 direct-execution 继续使用单任务 `executionHandoffDraft`，planning-confirmed 使用多任务 `taskHandoffSummary`。
- **只为已确认工件创建 Task。** 被用户标为“暂不执行”的 Story / Task 只记录在 handoff summary 与审计里，不提前建空壳 Task。这样 3.5 可以如实展示“待定”而不是把用户未确认的工作伪装成已计划。
- **`planned` 是新的执行准备态，`pending` 是 legacy 兼容态。** 3.4 生成的 Task 应统一使用 `planned`；若短期内手动建任务入口仍产出 `pending`，共享读模型至少要把两者清楚区分并保持兼容，直到 4.1 统一任务创建语义。
- **自动派发策略只做到“派发准备”，不真正派发。** 3.4 的 auto mode 只负责排序、记录下一步和暴露“自动派发候选”真值；Agent 选择、`Task.status: planned -> dispatched`、`AgentRun` 与 tmux 会话都属于 Epic 4。
- **审批门控优先于自动派发。** 如果工作空间开启 `requireApprovalBeforeExecution`，3.4 只能把 Task 标记为“待审批后派发”，不能因为 auto mode 就提前宣称已进入执行。

### 架构一致性要求

- 所有 mutation 继续遵循 `ActionResult<T>`、Zod 顶部校验、`sanitizeError()` 中文错误清洗与 `requireProjectAccess(..., "execute")` 授权边界；不要在 planning UI 组件里直接落库。
- 规划确认、Task 批量创建、`PlanningRequest` 衔接摘要更新和审计事件写入必须通过短事务保持一致；不要把后续路由 / supervisor / tmux 逻辑耦合进 3.4 的事务。
- 3.4 只负责“规划结果进入执行准备态”，**不负责** agent 路由、后台 session、日志监听、写回或完整链路中心。这些分别属于 Story 4.2、4.4、Epic 5 与 Story 3.5。
- 共享状态字符串必须集中定义在 `src/lib/planning/types.ts` 与 `src/lib/tasks/types.ts`，不要在 action、组件、测试里散落 `planned` / `execution-ready` / `auto-after-planning` 等新字符串。
- 变更后需要精确刷新项目 planning 区域、项目任务页和受影响的任务详情页；不要只靠客户端本地 state 假装“已确认成功”。
