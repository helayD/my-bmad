# Story 5.4: 执行中补充指令与上下文输入

Status: review

## Story

作为用户，
我希望能在 agent 执行过程中发送补充指令、确认信息或上下文输入，
以便在不终止任务的情况下引导执行方向。

## Acceptance Criteria

1. **Given** 任务处于 RUNNING 或 WAITING_FOR_INPUT 状态
   **When** 用户在任务详情的交互面板中输入补充指令并提交
   **Then** 系统将指令通过执行监督器转发到对应 tmux session 中的 agent（FR27）
   **And** 指令记录到 Interaction 记录，关联任务和 AgentRun

2. **Given** 用户提交确认信息（如批准 agent 的某个提议）
   **When** 确认信息发送到 agent
   **Then** agent 继续执行，任务状态从 WAITING_FOR_INPUT 变回 RUNNING
   **And** 确认动作记录到审计日志

3. **Given** 用户发送的补充指令
   **When** 传递给 agent 时
   **Then** 指令内容不被截断或篡改
   **And** 指令发送后用户看到"已发送"反馈

### Task 1: 在 tmux client 中实现 send-keys 功能

- [x] 1.1 在 `src/lib/execution/tmux/client.ts` 中添加 `sendKeys()` 函数：

  ```typescript
  /**
   * 向 tmux session 中的当前 pane 发送按键或字符串。
   *
   * 职责（Story 5.4 — FR27）：
   * - 将用户输入的指令或确认信息注入到 tmux session
   * - 支持发送普通文本（模拟键盘输入）和特殊按键（如 Enter）
   * - 确保指令内容不被 shell 转义篡改
   *
   * 安全性：
   * - sessionName 必须通过 buildSessionName() 格式验证（格式：bmad-{taskId}-{agentRunId}）
   * - 输入内容长度限制（单次发送不超过 10,000 字符）
   * - 禁止发送二进制或控制字符序列
   */

  export interface TmuxSendKeysConfig {
    sessionName: string;
    /** 要发送的内容（文本或特殊键如 "Enter", "C-c"） */
    content: string;
    /** 是否在末尾自动追加换行符，默认 true */
    addNewline?: boolean;
  }

  /**
   * 向 tmux session 发送按键。
   * @throws TmuxAdapterError on failure.
   */
  export async function sendKeys(config: TmuxSendKeysConfig): Promise<void> {
    const { sessionName, content, addNewline = true } = config;

    // 1. 验证 sessionName 格式
    if (!isValidSessionName(sessionName)) {
      throw new Error(`Invalid session name "${sessionName}": must match /^[a-zA-Z0-9_-]+$/`);
    }

    // 2. 内容长度限制
    const MAX_CONTENT_LENGTH = 10_000;
    if (content.length > MAX_CONTENT_LENGTH) {
      throw new Error(`发送内容超过长度限制（${MAX_CONTENT_LENGTH} 字符）`);
    }

    // 3. 检查 session 是否存在
    const exists = await hasSession(sessionName);
    if (!exists) {
      throw mapTmuxError(TMUX_ERROR_CODES.TMUX_SESSION_NOT_FOUND, { sessionName });
    }

    // 4. 构造发送内容
    const textToSend = addNewline ? `${content}\n` : content;

    // 5. 使用 tmux send-keys 发送
    //    - 需要将文本作为参数传递，避免 shell 转义问题
    //    - tmux send-keys 的参数直接传给 tmux，不会经过 shell
    try {
      await execFileAsync("tmux", [
        "send-keys",
        "-t", sessionName,
        "--",    // 分隔符，后续参数不会被解释为选项
        textToSend,
      ], { timeout: 5_000 });
    } catch (error) {
      throw mapTmuxError(TMUX_ERROR_CODES.TMUX_SEND_KEYS_FAILED, {
        sessionName,
        stderr: error instanceof Error ? error.message : undefined,
      });
    }
  }
  ```

### Task 2: 实现用户补充指令 Server Action

- [x] 2.1 在 `src/actions/execution-actions.ts` 中添加 `submitSupplementaryInput` action：

  ```typescript
  /**
   * 提交补充指令 — 将用户的补充指令、确认信息或上下文输入转发到执行中的 tmux session。
   *
   * 职责（Story 5.4 AC-1, AC-3 — FR27）：
   * - 接收用户输入的指令内容
   * - 验证任务状态（必须是 RUNNING 或 WAITING_FOR_INPUT）
   * - 通过 sendKeys() 将指令注入到 tmux session
   * - 更新 InteractionRequest 记录（复用现有模型，不新建 InteractionRecord）
   * - 如果任务处于 WAITING_FOR_INPUT，触发状态变更为 RUNNING
   * - 记录审计事件
   */

  "use server";

  import { z } from "zod";
  import { sanitizeError } from "@/lib/errors";
  import { prisma } from "@/lib/db/client";
  import { transitionTask } from "@/lib/execution/state-machine";
  import { sendKeys } from "@/lib/execution/tmux";
  import { getScheduler } from "@/lib/execution/heartbeat";

  const SubmitSupplementaryInputSchema = z.object({
    taskId: z.string().min(1, "任务 ID 不能为空"),
    agentRunId: z.string().min(1, "Agent Run ID 不能为空"),
    /** 用户输入的补充指令内容 */
    content: z.string()
      .min(1, "指令内容不能为空")
      .max(10_000, "指令内容不能超过 10,000 字符"),
    /** 指令类型：supplementary（补充指令）、confirmation（确认）、rejection（驳回） */
    inputType: z.enum(["supplementary", "confirmation", "rejection"]).default("supplementary"),
    /** 可选：关联的 InteractionRequest ID */
    interactionRequestId: z.string().optional(),
  });

  export async function submitSupplementaryInput(
    raw: z.infer<typeof SubmitSupplementaryInputSchema>
  ): Promise<ActionResult<{ success: true; delivered: boolean }>> {
    const parsed = SubmitSupplementaryInputSchema.safeParse(raw);
    if (!parsed.success) {
      return {
        success: false,
        error: sanitizeError(new Error(parsed.error.message), "VALIDATION_ERROR"),
        code: "VALIDATION_ERROR",
      };
    }

    const { taskId, agentRunId, content, inputType, interactionRequestId } = parsed.data;

    // 1. 查询任务状态
    const task = await prisma.task.findUnique({
      where: { id: taskId },
      select: {
        id: true,
        status: true,
        currentAgentRunId: true,
        workspaceId: true,
        projectId: true,
      },
    });

    if (!task) {
      return { success: false, error: "任务不存在", code: "TASK_NOT_FOUND" };
    }

    if (task.currentAgentRunId !== agentRunId) {
      return { success: false, error: "Agent Run 与当前任务不匹配", code: "RUN_MISMATCH" };
    }

    // 2. 状态验证：RUNNING 或 WAITING_FOR_INPUT 才可发送
    const ALLOWED_STATUSES = ["running", "waiting_for_input"];
    if (!ALLOWED_STATUSES.includes(task.status)) {
      return {
        success: false,
        error: `当前任务状态「${task.status}」不支持发送指令`,
        code: "INVALID_TASK_STATUS",
      };
    }

    // 3. 查询 ExecutionSession 获取 sessionName
    const session = await prisma.executionSession.findUnique({
      where: { agentRunId },
      select: { id: true, sessionName: true, status: true },
    });

    if (!session) {
      return { success: false, error: "找不到执行会话记录", code: "SESSION_NOT_FOUND" };
    }

    if (session.status !== "running") {
      return {
        success: false,
        error: "执行会话已结束，无法发送指令",
        code: "SESSION_ENDED",
      };
    }

    // 4. 通过 sendKeys 发送到 tmux
    let delivered = false;
    try {
      await sendKeys({
        sessionName: session.sessionName,
        content,
        addNewline: true,
      });
      delivered = true;
    } catch (error) {
      console.error("[submitSupplementaryInput] sendKeys failed:", error);
      return {
        success: false,
        error: sanitizeError(
          error instanceof Error ? error : new Error(String(error)),
          "TMUX_SEND_FAILED"
        ),
        code: "TMUX_SEND_FAILED",
      };
    }

    // 5. 更新 InteractionRequest 记录（复用现有模型）
    //    如果有关联的 InteractionRequest，更新其响应字段和状态
    if (interactionRequestId) {
      await prisma.interactionRequest.updateMany({
        where: { id: interactionRequestId, taskId },
        data: {
          status: "responded",
          response: content,
          respondedAt: new Date(),
        },
      }).catch(() => { /* ignore if not found */ });
    }

    // 6. 如果任务处于 WAITING_FOR_INPUT，触发状态变更为 RUNNING
    if (task.status === "waiting_for_input") {
      const transitionResult = await transitionTask({
        taskId,
        toStatus: "running",
        trigger: "user_response",
        actorType: "user",
        reason: "用户响应了 Agent 请求",
      });

      if (!transitionResult.success) {
        console.warn("[submitSupplementaryInput] 状态变更失败:", transitionResult.error);
      }

      // 刷新 HeartbeatScheduler 记录
      const scheduler = getScheduler(taskId);
      if (scheduler) {
        scheduler.recordWithSnapshot({
          status: "running",
          currentStage: "运行中",
          currentActivity: "等待 Agent 响应补充指令……",
        });
      }
    }

    // 7. 记录审计事件
    await prisma.auditEvent.create({
      data: {
        workspaceId: task.workspaceId,
        projectId: task.projectId,
        taskId,
        eventName: "supplementary_input.submitted",
        occurredAt: new Date(),
        payload: {
          agentRunId,
          inputType,
          contentLength: content.length,
          interactionRequestId: interactionRequestId ?? null,
          taskStatusBefore: task.status,
          taskStatusAfter: task.status === "waiting_for_input" ? "running" : task.status,
        },
      },
    });

    return { success: true, data: { success: true, delivered } };
  }
  ```

  > **注意：** 本 story 复用 `InteractionRequest` 模型的 `response`/`respondedAt`/`status` 字段，无需新建独立的 `InteractionRecord` 模型。

        toast({
          title: "发送失败",
          description: result.error ?? "无法发送指令",

      <div className="flex items-center justify-between">
        <span className="text-xs font-medium text-muted-foreground">
          发送补充指令
        </span>
        {sentFeedback && (
          <span className="text-xs text-green-600">{sentFeedback}</span>

## Dev Notes

### 关键约束（来自 Checklist 自动分析）

> **这些约束必须严格遵守，违反将导致实现失败或破坏现有功能。**

1. **tmux send-keys 必须避免 shell 转义问题**：必须使用 `-t sessionName` 指定目标 pane，使用 `--` 分隔符防止后续参数被解释为选项。正确命令：`tmux send-keys -t <sessionName> -- <text>`。使用 `execFile` 而非 `exec`，参数传递为数组而非字符串拼接。

2. **`sendKeys()` 的输入必须有限制**：长度不超过 10,000 字符，禁止发送二进制字符。sessionName 必须通过 `isValidSessionName()` 格式验证，防止注入。

3. **`transitionTask()` API 签名**：函数接受**单个 `TransitionInput` 对象参数**，`actorType` 枚举值为 `"user"`、`"system"`、`"agent"`。用户响应触发 `toStatus: "running"`, `trigger: "user_response"`, `actorType: "user"`。

4. **OutputMonitor 生命周期**：在 `launch.ts` 中与 HeartbeatScheduler 同时启动，在 `lifecycle.ts` 中同时停止。同时需在 `endSession` 中调用 `unregisterScheduler(input.taskId)` 以保持 scheduler 注册表的对称性。

5. **复用 `InteractionRequest` 而非新建 `InteractionRecord`**：`InteractionRequest` 模型已有 `response`、`respondedBy`、`respondedAt`、`status` 字段。本 story 在 `submitSupplementaryInput` 中直接更新 `InteractionRequest` 的响应字段并设置 `status: "responded"`，无需新增独立的交互记录模型。

6. **交互请求超时不静默丢弃**：Story 5.5 负责超时检测和升级；本 story 只需确保指令能发送和记录。

7. **中文 UI 文本**：所有用户可见文本必须中文。

8. **AuditEvent 记录**：`supplementary_input.submitted` 事件需记录 `inputType`、`contentLength`、`interactionRequestId`。

### 核心实现目标

- Story 5.4 的核心是建立**用户到 agent 的指令回传管道**：用户输入 → Server Action → sendKeys → tmux session → agent。
- 当 agent 发出交互请求（Story 5.3 检测），用户在本 story 的输入面板响应，agent 继续执行。
- 状态从 `WAITING_FOR_INPUT` 回到 `RUNNING`。
- SSE 推送保证 UI 能实时感知状态变化。

### 最新技术信息

- Next.js 16.1.6、React 19.2.3、Prisma 6.19.2、Zod 4.3.6 版本基线不变。
- tmux send-keys 使用 `--` 分隔符避免选项解析：`tmux send-keys -t <session> -- <text>`。
- 单次 sendKeys 建议不超过 10,000 字符以避免 tmux 缓冲区溢出。
- `useTransition` 用于非阻塞提交，避免阻塞 UI 响应。

### Project Structure Notes

- tmux client 是无状态适配器，每个操作打开独立的 tmux 进程。
- sendKeys 的错误码必须与现有 TMUX_ERROR_CODES 一致风格。
- Server Action 的权限校验遵循现有模式：`requireAuth()` 在 action 层保证。
- Prisma 模型变更需运行迁移并提交迁移文件。
- UI 组件使用 `"use client"`，提交逻辑放在 Server Action 中。
