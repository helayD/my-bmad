# Story 3.5: 规划到执行派发的完整链路可见性

Status: review

## Story

作为用户，
我希望能查看从规划技能执行到编码任务派发的完整链路与状态，
以便理解每个规划请求经历了哪些步骤、产出了哪些工件、衍生了哪些执行任务。

## Acceptance Criteria

1. **Given** 用户查看某个规划请求的详情 **When** 详情区域加载完成 **Then** 按顺序展示完整链路：原始目标 → 意图识别结果 → skill 执行序列 → 产出工件列表 → 衍生执行任务列表
   **And** 每个环节显示状态、时间戳和关键输出（FR17）

2. **Given** 用户在项目级别查看规划历史 **When** 用户访问规划请求列表 **Then** 展示该项目的规划请求，包含目标摘要、状态、产出工件数量和衍生任务数量
   **And** 支持按状态筛选

3. **Given** 规划链路中某个环节出现问题 **When** 用户查看链路详情 **Then** 问题环节被高亮标记，并展示错误原因与建议动作
   **And** 用户可以从该环节继续分析、重试失败步骤、确认规划结果，或基于当前信息重新调整目标

- `src/app/(dashboard)/workspace/[slug]/project/[projectSlug]/page.tsx` 当前只通过 `getRecentPlanningRequestsByProjectId(project.id)` 取最近请求，并把结果传给 `PlanningRequestComposer`；这与 3.5 的“项目级规划历史 + 状态筛选”验收标准不匹配。

- `src/components/planning/planning-request-composer.tsx` 已经集中管理 `useTransition`、toast、执行规划、handoff 预览与确认对话框。3.5 的详情打开、筛选切换与问题恢复动作最好继续在这里汇合，避免 action 分散到多个小组件。
- `src/lib/planning/queries.ts` 当前 list item 已包含 `artifactSummary`、`executionSteps`、`confirmedAt`、`taskHandoffSummary`、`derivedTaskCount`、`deferredArtifactCount`，说明 3.5 不缺“事实源字段”，缺的是“完整链路读取模型”和“项目页筛选入口”。
- `src/lib/db/helpers.ts` 已存在 `getTasksByPlanningRequestId(projectId, planningRequestId, status?)`，这是 3.5 展示衍生执行任务列表的首选数据库真值，不需要再去 `Task.metadata` 或 UI state 中猜测来源关系。
- 当前 `artifactSummary` 只保存 `path / title / kind / summary / sourceSkillKey / status / storyId / epicId`，没有 artifactId；3.5 需要在查询阶段把路径解析回真实 `BmadArtifact`，而不是新增一个仅为 UI 服务的平行存储。
- 现有深链已经足够支撑链路跳转：任务详情用 `buildTaskDetailHref(...)`，来源工件用 `buildSourceArtifactHref(...)`，任务详情页和工件侧栏也已经能继续查看 writeback / history / source hierarchy。

### 前序 Story 情报

- **来自 Story 3.4 的关键真值：** `PlanningRequest.confirmedAt`、`taskHandoffSummary`、`derivedTaskCount`、`deferredArtifactCount`、`Task.planningRequestId`、`Task.status = planned` 已经建立；3.5 要做的是“看清这条链”，不是再实现一次 handoff。
- **来自 Story 3.3 的关键真值：** `PlanningExecutionStep`、`artifactSummary`、`awaiting-confirmation`、story stub 投影已经成立；3.5 应直接消费这些结构，不要再从 `epics.md` 反推步骤状态。
- **来自 Story 2.4 / 2.5 的现有能力：** Artifact / Task 详情页已经具备执行历史和回写视图，因此 3.5 更适合扮演“规划链路索引与入口”，而不是抢先实现 Epic 5 的日志、心跳或证据链中心。

### 前序经验与回归护栏

- 3.4 的 handoff 预览曾暴露“旧请求返回过晚覆盖当前状态”的风险；3.5 新增的详情面板与筛选切换必须用 `planningRequestId` 绑定异步结果，并在关闭或切换后丢弃过期响应。
- 3.4 还暴露过用户可见英文文案渗出的风险；3.5 中所有分组标题、空状态、按钮与问题提示都必须保持中文，只有技术标识符和 role/slug 等例外。
- 3.4 已明确 `planned` 是新真值、`pending` 是 legacy 兼容态；3.5 详情与列表不能把 `pending` 简单当成错误状态，也不能把它伪装成与 `planned` 完全相同。
- 3.4 的零候选任务场景要求“说真话”；3.5 在 `taskHandoffSummary` 缺失、候选任务为 0 或工件路径无法回查时，也必须提供诚实的降级说明和下一步动作。

### 性能与查询护栏

- 项目页是高频入口，3.5 的链路详情 enrichment 不应在组件层按请求逐条 `await` 任务与工件查询；优先在 query/helper 层做批量读取与结构化整形。
- 架构文档对控制面关键页面的首屏反馈有 2-3 秒要求，因此 3.5 需要优先复用 `cache()`、批量查询和服务端筛选，避免把读取压力转移到浏览器端。

### Git Intelligence

- 最近相关提交集中在 planning/history 方向：
  - `feat(planning): add execution workflow and history views`
  - `fix(planning): address execution review findings`
  - `feat(planning): add project planning request flow`
- 这说明实现时应优先延续现有 `src/lib/planning/*` + `src/components/planning/*` + `src/actions/planning-actions.test.ts` 的分层与测试模式，而不是引入新的跨域目录。

### 默认实现决策

- **详情选择与筛选优先走 URL。** 推荐用 `planningStatus` 保存列表筛选，用 `planningRequestId` 保存当前打开的详情请求；这样刷新、分享链接、浏览器前进后退都能保持上下文。
- **详情承载优先使用 Sheet / 同页侧栏。** UX 文档已经明确“复杂详情不适合大模态”；3.5 的链路详情包含步骤、工件、任务与恢复动作，适合侧边详情，不适合对话框。
- **衍生任务读取以数据库关系为主，JSON 摘要为辅。** `Task` 表能给出真实状态、下一步和详情入口，`taskHandoffSummary` 更适合补充 `readyState`、`queuePosition`、待定项等 handoff 语义。
- **工件链接通过路径解析回真实 artifact。** 不建议为了 3.5 单独修改 `PlanningArtifactSummaryItem` 的持久化结构；如果读时能通过 `projectId + filePath` 回查，就优先做查询层 enrichment。
- **问题环节判定必须集中。** 建议新增如 `resolvePlanningRequestProblemStage(...)` 的 helper，供列表和详情共用；不要让“失败原因 + 下一步动作”在两个组件里各写一套条件分支。

### 技术要求

- 若 3.5 新增只读 action（例如 `getPlanningRequestDetailAction`），必须在 action 顶部使用 Zod 校验，并使用 `getAuthenticatedSession()` + `requireProjectAccess(..., "read")` 守住读取边界；不要沿用执行类 action 的 `"execute"` 权限去读取历史详情。
- 所有新增用户可见文案继续保持中文，并在错误路径中使用 `sanitizeError()`；不要把 Prisma / Next.js 原始报错直接暴露到详情面板。
- 列表筛选、详情打开等高频交互继续沿用 `useTransition()` 与局部状态，不要为了 3.5 引入全局客户端状态库。
- 3.5 主要是读取与呈现，不应新增数据库写入事实字段，除非实现中发现现有字段无法表达验收标准；优先通过 query enrichment 解决，而不是修改 schema 扩列。
- 如果需要在项目页切换 `searchParams`，优先利用现有 App Router 页面和 `router.replace()` / `Link` 方式同步 URL，不要只在客户端保留隐藏状态。
- 如果需要批量读取衍生任务或工件链接，优先在 `src/lib/db/helpers.ts` 新增 `cache()` 包装的批量 helper（例如按 `planningRequestId[]`、`filePath[]` 查询），不要在组件或 action 里循环调用单条 helper。

### 架构一致性要求

- 架构文档要求“数据真值来自服务端，不在客户端构造独立业务真相”，因此 3.5 的链路详情必须来源于 `PlanningRequest` / `PlanningExecutionStep` / `Task` / `BmadArtifact` 等真实读模型，而不是在组件层手工合并异步片段后当作新事实源。
- 架构文档同时要求“所有列表筛选状态优先体现在 URL 参数中”，所以项目级规划状态筛选不应只靠 `useState` 做一次性本地过滤。
- 3.5 仍属于 Epic 3 的“规划到执行准备态链路”，**不负责** Agent 选择、`Task: planned -> dispatched`、`AgentRun`、tmux session、日志流、心跳或恢复动作；这些分别属于 Story 4.2、4.4 与 Epic 5/6。
- 详情中的状态必须继续“说真话”：`execution-ready` 只表示进入执行准备态；`planned` 只表示任务已建立但未派发；`approval-required` / `auto-ready` 只是准备状态，不是执行状态。

### 库与框架要求

- 当前仓库真实版本基线：Next.js `16.1.6`、React `19.2.3`、Prisma `6.19.2`、Zod `4.3.6`、Tailwind CSS `4.2.2`、Vitest `4.0.18`。
- 3.5 不做框架升级，也不引入新的数据获取框架；继续沿用 App Router + Server Components + 局部客户端组件的当前模式。
- 如果详情面板采用惰性加载，继续沿用现有项目对客户端 read action 的使用方式；不要为了这个故事把整个 planning 区域重写成全客户端页面。
