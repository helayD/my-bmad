# Product Requirements Document - my-bmad

**Author:** David
**Date:** 2026-04-04T16:27:00+08:00

## Executive Summary

my-bmad 将从现有的 BMAD 项目浏览与管理能力，扩展为一个面向个人与团队的 AI 驱动研发流程编排与进度管理平台。产品目标不是仅帮助用户记录 PRD、Epic、Story 或调用单个 AI 编码工具，而是将研发工作转化为可分解、可路由、可执行、可跟踪、可治理的执行流水线。

该平台支持基于任务类型将工作路由给不同 agent，并在与项目同机的 self-hosted 环境中通过 `tmux` 后台运行 `codex` 或 `claude code` 持续推进任务。系统还应支持用户仅通过一句自然语言目标发起项目，由 PM 型 agent 自动调用合适的 BMAD skills 完成项目规划，生成或更新 PRD、Epic、Story、Task 等工件，再将可执行的编码工作自动派发给编码 agent。对于编码执行阶段，平台不能只负责“启动”任务，而必须支持长时间运行、持续监听执行过程，并在 `codex` 或 `claude code` 执行编码过程中出现提问、确认、补充上下文、请求授权或需要策略调整时，与 agent 进行动态交互。系统需要覆盖从 PRD、Epic、Story 到 Task、Subtask、Agent Run、tmux Session、Prompt Dispatch、日志心跳、Artifact、代码变更与发布的最细颗粒度控制，使用户不仅能定义工作，还能管理工作如何被规划、如何被执行、如何被观察、如何被接管以及如何被验收。

对个人用户，这意味着可以把一句研发目标或一段研发目标直接委托给系统，让系统先完成规划，再自主拆解上下文、选择执行 agent、创建后台会话、持续推进并回传结果。即使编码任务长时间运行，用户也应能够看到 agent 当前在做什么、何时提出交互请求，以及系统或人工是如何回应这些请求的。对团队用户，这意味着可以在统一平台中实现从项目规划到任务执行的跨角色协作、执行状态透明化、审批与治理闭环，以及对 AI 驱动研发过程的标准化管理。产品成功的标志，不是增加一个 AI 面板，而是让用户能够放心把研发任务交给系统规划和执行，并在任何时刻清楚知道任务处于什么状态、为什么这样推进、何时需要人工介入。

### What Makes This Special

my-bmad 的独特性在于，它将 BMAD 工件体系、任务编排、agent 路由、`tmux` 后台执行、执行可观测性和商业化治理整合为一个统一控制面。现有替代方案通常只能覆盖其中一个局部：要么是项目管理，要么是 AI 对话，要么是脚本自动化，要么是单点编码代理。my-bmad 试图解决的是它们之间的断层：任务定义和任务执行脱节，AI 能力和工程治理脱节，自动化推进和团队协作脱节。

产品的核心洞察是，研发流程的真正瓶颈往往不在“能不能生成代码”，而在“能不能把任务可靠地交给系统执行，并在执行过程中保持上下文完整、状态透明、权限受控、异常可接管、结果可审查”。因此，my-bmad 提供的不是一个更聪明的聊天入口，而是一套让 AI 参与研发工作的执行基础设施。用户获得的关键价值不是更少打字，而是更少调度成本、更少跟进损耗、更强执行连续性，以及对 AI 产出过程的真实掌控。

从体验上，产品追求接近 `yolo` 的使用感：用户给出明确目标后，系统默认积极推进而不是反复等待确认。从治理上，产品必须满足企业级要求：每个任务、每次派发、每个后台会话、每条关键日志、每份产物和每次人工介入都应具备清晰的权限边界、审计记录和状态追踪能力。这种“前台极简、后台严控”的组合，构成了产品的核心差异化。

## Success Criteria

### User Success

个人用户能够把一句研发目标、Story、Task 或明确的研发目标直接交给系统，由系统自动选择合适的 PM 型 agent 与 BMAD skills 完成规划，再选择合适的执行 agent、创建 `tmux` 后台会话、注入上下文并持续推进，用户无需手工在规划工具、终端窗口和编码 agent 之间来回切换。对于长时间运行的编码任务，用户能够看到系统持续监听执行过程，并在 `codex` 或 `claude code` 需要补充说明、确认策略、请求上下文或等待人工决策时收到清晰提示。用户的成功体验不是“AI 给了建议”，而是“系统已经先完成规划并开始推进，我随时能看见状态、理解交互，并在必要时接管”。

团队用户能够在统一控制面中看到从 PRD、Epic、Story 到 Task、Subtask、Agent Run、tmux Session、日志、产物、代码变更的完整执行链路。团队的成功体验不是“流程被记录”，而是“流程被执行且可治理”：成员知道谁发起、谁审批、哪个 agent 在执行、当前卡在哪里、何时需要人工介入、最终结果是否可审查和可验收。

用户在关键时刻应感受到明确价值：任务从创建到执行被快速启动；执行过程状态可信；失败时系统可恢复或明确告警；人工接管路径清晰；结果能稳定回写到 BMAD 工件流与研发进度视图中。

### Business Success

短期成功（3 个月）表现为：产品能够支撑真实个人用户与小团队在 self-hosted 环境中稳定运行 AI 驱动研发任务，证明“从任务定义到后台执行再到结果回收”的闭环成立；早期用户愿意把真实开发任务而不是试玩任务交给系统；平台在权限、审计、日志和接管能力上达到团队可试用标准。

中期成功（12 个月）表现为：产品从单人/小团队工具升级为团队级研发执行平台，形成持续活跃的任务量、稳定的团队留存和扩张能力；企业用户愿意基于其治理、可观测性和执行效率为产品付费；平台成为 BMAD 驱动研发团队的默认控制面，而不是附属工具。

业务成功的核心不只是活跃用户数量，而是高质量执行量：有多少真实任务被系统接收、启动、持续推进、被团队审查并最终交付。

### Technical Success

系统必须稳定支持先通过 PM 型 agent 调用对应 BMAD skills 进行项目规划，再按任务类型将工作路由给 `codex` 或 `claude code`，并在与项目同机的 self-hosted 环境中通过 `tmux` 执行后台任务。系统必须保证规划链路、会话生命周期、日志采集、心跳上报、状态同步、失败检测、人工接管、结果回写与审计链路都可信。对于长时间运行的编码任务，系统还必须具备持续监听执行过程、识别 agent 交互请求、支持动态注入补充指令或上下文、并保留完整交互历史的能力。

技术成功的关键不是单一吞吐量，而是执行闭环可靠性：任务创建后能否稳定启动；运行中状态是否持续可见；异常时能否自动恢复或快速升级；人工能否低成本接管；最终产物是否能回写到对应 BMAD 流程节点并支撑后续审查、测试与发布。

企业级治理能力属于技术成功的一部分，而非附加项。这包括权限边界、审计完整性、任务隔离、执行历史可追溯、关键操作可归因，以及对误路由、误执行、状态漂移和日志缺失的控制能力。

### Measurable Outcomes

- 任务创建到首次 agent 接单或启动成功的中位时间小于 30 秒
- 一句自然语言目标到首版 BMAD 规划工件生成完成的中位时间小于 5 分钟
- 任务创建到首个可见执行心跳的中位时间小于 60 秒
- 任务创建到首个可审查产出的中位时间小于 10 分钟
- 自动任务启动成功率达到 98% 以上
- 至少 60% 的一句目标发起请求能够在无需人工重新编排的情况下生成可执行的 BMAD 规划结果
- 后台执行中的状态可见率达到 99% 以上
- 长时间运行任务在执行期间的持续监听覆盖率达到 99% 以上
- `codex` 或 `claude code` 发出的交互请求可在 10 秒内被系统识别并暴露到控制面
- 至少 80% 的编码执行交互请求能够由系统或授权用户在不丢失上下文的情况下完成有效响应
- `tmux` session 异常终止的检测时间小于 15 秒
- 人工接管入口可在 60 秒内完成定位并接入目标任务
- 至少 80% 的标准研发任务可在无需人工重新编排的情况下完成自动路由与执行
- 至少 70% 的任务执行结果能够自动回写到对应 BMAD 工件或进度节点
- 审计关键事件覆盖率达到 100%，包括创建、派发、启动、失败、重试、接管、完成、回写
- 团队用户在 30 天内的周活跃留存达到 60% 以上
- 付费团队在导入项目后的 14 天内至少完成一次真实任务闭环的比例达到 50% 以上

## Project Scope

### MVP - Minimum Viable Product

MVP 必须完成从自然语言目标到 BMAD 规划再到后台执行的最小可用闭环。核心包括：基于一句目标或基于 PRD、Epic、Story、Task 发起任务；自动选择合适的 PM 型 agent 和对应的 BMAD skills 完成项目规划，生成或补全 PRD、Epic、Story、Task 等工件，并把其中已经具备执行条件的编码任务自动路由到更适合当前任务的 `codex` 或 `claude code`；通过 `tmux` 创建与管理后台 session；支持长时间运行任务的持续监听；可见的任务状态、心跳与执行日志；在编码 agent 执行过程中识别交互请求并支持动态响应；失败检测与基础重试；人工接管入口；执行结果与关键状态回写到 BMAD 工件流；基础 RBAC、审计日志与项目级隔离。MVP 必须让个人用户和小团队真实完成一段从规划到执行的研发工作，而不只是演示自动化。

### Growth Features (Post-MVP)

Growth 阶段强化团队协作与商业化能力。包括更细粒度权限模型、审批流、多角色视图、通知与告警中心、任务模板与自动派发策略、批量任务编排、跨项目看板、执行质量评分、成本与配额管理、更多失败恢复策略、任务 SLA 与健康度指标、团队级报表与运营分析，以及更强的多技能规划链路与跨阶段自动派发能力。此阶段目标是让平台从“可用”升级到“可扩展、可治理、可运营”。

### Vision (Future)

Vision 阶段实现完整的 AI-native 研发控制平台。平台不仅负责任务路由与执行，还能主动识别阻塞、建议下一步、自动串联多个 BMAD 规划与执行链路、在不同阶段选择不同 agent、根据历史数据优化执行策略，并最终形成从一句目标到 PRD 再到发布的半自治研发操作系统。对于编码执行阶段，系统应能够像一个持续在线的执行协调者一样，与 `codex` 或 `claude code` 在长时间运行过程中保持动态互动，而不是只负责启动后放任不管。用户体验上接近 `yolo`：只给目标，系统持续推进；治理上保持企业级可控：所有规划与执行都有边界、证据、历史和接管机制。

## User Journeys

### Journey 1: 主用户成功路径 - 独立开发者把一个 Story 交给系统持续推进

林涛是一名独立开发者，已经在 my-bmad 中维护自己的 BMAD 项目。他不缺想法，也不缺编码工具，真正拖慢他的，是频繁在 PRD、Story、终端、AI 对话和待办之间切换。每次任务开始时，他都要重新整理上下文、选择工具、组织 prompt、盯着执行过程，最后还得自己把结果回写到项目里。

某天他不再先手工写完整 PRD，而是直接输入一句话目标，希望系统替他推进整个项目。系统首先识别这是一个需要先规划后执行的请求，调用合适的 PM 型 agent 和对应的 BMAD skills 完成项目规划，生成或补全 PRD、Epic、Story、Task 等工件，并把其中已经具备执行条件的编码任务自动路由到更适合当前任务的 `codex` 或 `claude code`。随后系统在本机环境中创建对应的 `tmux` 后台 session。林涛没有再打开多个终端，也没有继续盯着 AI 聊天窗口，而是在控制面中看到规划已生成、任务已拆解、执行已启动、首个执行心跳已返回。

随着任务推进，他看到状态从“规划中”变为“已派发”再到“执行中”，可以查看摘要日志、关键操作、产生的产物以及当前阻塞点。系统不是只告诉他“还在运行”，而是持续告诉他规划技能在做什么、哪些任务已经进入执行、做到哪一步、是否偏离预期。当 `codex` 在实现过程中需要确认一个接口选择，或 `claude code` 需要补充一段上下文说明时，系统会立即把交互请求暴露到控制面，并允许林涛或授权规则直接回应。回应后的上下文会继续留在同一执行链路中，任务不会因为交互而丢失状态。当产出达到第一版可审查状态时，系统将结果回写到对应工件链路，并提示他进入 review。林涛第一次感受到的价值，不是 AI 写了多少代码，而是他真的把从规划到执行的工作交出去了，而且过程可信、结果可接。

这条旅程揭示出核心能力需求：从自然语言目标触发 BMAD 规划、自动选择并执行对应 BMAD skills、将规划结果映射为可执行任务、自动 agent 路由、`tmux` 后台执行、长时间运行监听、执行中动态交互、状态流转可视化、执行日志摘要、产物回写、可 review 交付节点。

### Journey 2: 主用户异常路径 - 任务中途卡死但系统能发现、恢复或升级

同样是林涛，这次他发起的是一个更复杂的实现任务。任务初期推进顺利，但运行十几分钟后，`tmux` session 中的 agent 因环境问题卡住，不再产生日志，也没有新的心跳。如果这是传统脚本或普通 AI 工具，他往往要很久之后才意识到任务已经停了，前面的时间全部被浪费。

在 my-bmad 中，系统发现该任务心跳超时，状态从“执行中”转为“异常待恢复”，同时保留最近一次上下文、最后有效输出和失败前的关键操作轨迹。系统先按预设策略尝试一次自动恢复：重新附着 session、重新注入必要上下文、恢复任务执行。如果恢复成功，任务继续推进，并在审计中留下恢复记录；如果恢复失败，系统升级为需要人工介入，并明确告诉林涛问题发生在哪一层：是 agent 无响应、session 中断、日志链路异常，还是回写失败。

林涛点开任务后，不需要从零排查。他看到异常时间点、最后一条有效日志、之前已经完成的步骤，以及推荐的接管方式。他可以选择手动进入 `tmux` session、改派到另一个 agent、终止本次执行后重新派发，或者标记为人工继续。对他来说，真正的重要时刻不是任务一直成功，而是任务失败时系统没有“失声”。

这条旅程揭示出关键异常能力：心跳监控、超时检测、失败分类、自动恢复策略、升级告警、人工接管入口、失败上下文保留、审计记录。

### Journey 3: 管理路径 - 团队负责人编排任务、设置边界并验收结果

周岚是一支 AI-native 工程小团队的负责人。她不直接盯每一段代码，但她必须确保团队任务推进有序、AI 执行有边界、结果可以验收。过去她的问题是：任务虽然很多，但真正发生了什么没人说得清；即便有人在用 AI，也无法保证上下文一致、过程留痕、风险可控。

在 my-bmad 中，她从项目视图进入团队控制面，可以看到 PRD、Epic、Story、Task 的状态，以及每个任务对应的 agent run、后台 session、最近心跳、风险标记和产物状态。她为不同任务类型设置默认路由策略，例如实现类优先走 `codex`，重构或探索类根据规则切换到 `claude code`。她还配置团队边界：哪些任务可自动执行，哪些必须审批，哪些项目启用更严格的日志保留和审计要求。

当一个关键 Story 被提交执行后，周岚并不需要时时盯盘。她只关注几个高风险信号：卡死、重试过多、长期无产出、接近配额上限、跨阶段回写失败。当系统提示某条任务已达到可 review 状态，她可以查看摘要、关键产物和执行链路，决定批准继续、要求重跑或转人工处理。对她而言，产品成功不是“看板更漂亮”，而是 AI 执行终于进入团队治理体系。

这条旅程揭示出团队治理能力：团队视图、任务和执行链路关联、路由策略配置、审批机制、风险队列、验收节点、权限边界、配额与审计控制。

### Journey 4: 支持/排障路径 - 运维或支持人员调查异常任务并快速接管

陈工负责平台支持与排障。他不是任务发起人，但当团队反馈“任务没推进”“结果没回写”“日志断了”时，必须由他快速定位根因。过去最痛苦的地方，是每个问题都散落在不同工具里：一部分在终端，一部分在日志系统，一部分在项目管理工具，最终没有谁能给出完整因果链。

在 my-bmad 中，陈工接到告警后进入异常任务详情页，看到任务是从哪个 Story 发起、由谁批准、路由到了哪个 agent、绑定了哪个 `tmux` session、最近心跳何时中断、最后一次成功操作是什么。系统同时展示失败分类与上下游影响，例如“agent run 已停止，但 session 仍在”“执行完成但回写失败”“日志采集链路异常导致状态不可见”。

他可以进一步查看原始日志、摘要日志、状态变更历史和关键审计事件，并决定最合适的处置方式：恢复采集、重新附着 session、手动回写产物、切换执行路径、通知负责人审批后重试。处理完后，系统记录完整处置链路，便于事后复盘和优化恢复策略。陈工最需要的不是更多日志，而是可操作的上下文完整性。

这条旅程揭示出支持能力：异常中心、任务到执行实例的追踪、故障分类、上下游影响视图、原始日志与摘要日志、处置动作、复盘记录。

### Journey 5: 集成路径 - 高级用户通过 API 或自动规则批量触发执行

何远是团队里的平台工程师，希望把 my-bmad 接入现有研发流程。他不满足于人工逐个点击发起任务，而是希望当 BMAD 工件达到某种状态时，系统自动批量派发执行任务，或由外部系统通过 API 触发任务创建和状态查询。

在理想状态下，他为项目配置了一条规则：当 Story 进入“ready-for-implementation”且依赖满足时，自动创建执行任务并走默认 agent 路由。外部系统也可以通过 API 提交任务、查询执行状态、拉取审计事件或订阅关键状态变化。何远关心的不是页面体验，而是系统是否具备稳定、可预测、可治理的自动化接口。

这条旅程揭示出集成能力：API 触发、状态查询、事件订阅、自动派发规则、幂等控制、外部系统身份与权限管理。

### Journey Requirements Summary

以上旅程共同揭示出以下能力域：

- 从自然语言目标触发 BMAD 规划、自动选择并执行对应 BMAD skills、将规划结果映射为可执行任务、自动 agent 路由、`tmux` 后台执行、长时间运行监听、执行中动态交互、状态流转可视化、执行日志摘要、产物回写、可 review 交付节点
- 从 BMAD 工件直接发起任务，并保持工件到执行实例的可追踪映射
- 按任务类型执行 agent 路由，并支持策略配置与后续改派
- 通过 `tmux` 创建、管理、恢复和接管后台执行会话
- 提供任务状态、心跳、日志、产物、回写结果的统一可视化
- 支持异常检测、失败分类、自动恢复、人工接管和升级告警
- 为团队负责人提供审批、验收、风险视图、权限边界、审计能力
- 为支持/排障角色提供从任务到执行链路的完整调查视图
- 为高级用户与外部系统提供 API、规则触发、状态查询和事件能力

这些旅程说明，本产品的核心需求不只是“让 AI 跑起来”，而是让任务从定义、派发、执行、观察、恢复、接管到交付形成可信闭环。

## Domain-Specific Requirements

### Compliance & Regulatory

本产品不属于医疗、金融或政务等强监管垂直行业，因此不存在必须遵循的单一行业监管框架作为产品前提。但作为面向团队与商业化场景的 AI 驱动研发控制平台，系统必须满足企业级软件治理要求，尤其是在审计、权限控制、数据访问边界和执行可追溯性方面达到可采购、可评估、可上线的标准。

产品应支持完整的审计记录能力，覆盖任务创建、派发、agent 路由、`tmux` session 启动、失败恢复、人工接管、产物回写、审批与关键配置变更。审计记录应具备时间、主体、对象、动作和结果五类核心信息，支持后续导出、检索与归因。

产品应具备满足企业安全评估的基础能力，包括最小权限模型、敏感配置隔离、凭据不暴露、关键操作可审计、项目间执行隔离、团队边界控制，以及必要的数据保留与清理策略。虽然初期不必以正式认证为上线前提，但产品架构应兼容后续向 SOC 2、ISO 27001 类治理要求靠拢。

### Technical Constraints

产品必须运行在与项目同机的 self-hosted 环境，并通过 `tmux` 承载后台 agent 执行。这意味着系统架构必须正视本地执行的不确定性：会话可能异常终止，日志可能中断，环境可能漂移，文件系统状态可能变化，外部 CLI 工具版本可能不一致。因此平台必须将“执行可恢复性”视为一等能力，而不是附加功能。

所有任务执行必须与其所属项目、工件、agent run 和 session 建立稳定映射，避免出现“任务仍显示运行，但底层 session 已失效”或“session 仍存在，但平台状态已漂移”的情况。系统必须提供可靠的心跳机制、状态同步机制和异常检测机制，以保证控制面显示的信息可信。

由于平台允许自动把任务交给 `codex` 或 `claude code`，系统必须具备清晰的执行边界控制能力，包括任务级上下文注入、项目范围限制、敏感文件访问策略、执行命令可观测性，以及人工接管前后的状态一致性。平台不能把 agent 当黑箱，而必须把 agent 运行过程转化为可见、可控、可恢复的执行单元。

对于长时间运行的编码任务，系统还必须支持持续监听执行过程中的标准输出、结构化事件、交互提示和状态变化，并在 agent 请求更多信息、要求确认、建议变更方向或等待授权时，能够以同一执行上下文继续与 `codex` 或 `claude code` 动态交互。若系统无法监听或响应这些交互，自动派发就会在真实编码场景中退化为一次性启动器，而不是可持续推进的执行控制面。

### Integration Requirements

产品必须与 BMAD 工件体系深度集成，至少能够识别并关联 PRD、Epic、Story、Task 与执行任务之间的映射关系，并在执行完成或中断后将关键结果、状态和产物回写到相应链路。没有这层映射，平台将退化为一个独立的 agent 调度器，而不是研发流程控制面。

产品还需要与本地项目目录、代码仓库、日志视图和后续外部触发能力形成一致的数据边界。对于商业化和团队场景，平台应预留 API、Webhook、事件订阅和自动派发规则等扩展接口，以支持未来接入外部 CI、审批系统、通知系统或内部研发平台。

由于执行发生在本机 self-hosted 环境，系统还需要明确区分控制面与执行面的责任边界：控制面负责路由、状态、审计、治理与可视化；执行面负责 agent 调用、`tmux` session 生命周期、环境访问与本地产物生成。两者之间必须通过稳定的事件、状态和日志接口对齐。

### Risk Mitigations

本产品最大的风险不是“AI 不够聪明”，而是“自动化执行不可信”。因此首要风险控制目标包括：错误路由导致任务走错 agent、session 卡死但平台未发现、日志断流导致状态不可见、自动恢复造成重复执行、回写失败导致流程断层，以及权限边界不清导致跨项目或敏感内容泄露。

系统必须针对这些风险建立明确缓解机制。对于执行风险，应提供心跳检测、超时判定、失败分类、自动恢复与人工升级路径；对于治理风险，应提供审批、RBAC、审计导出、关键配置保护与项目隔离；对于数据风险，应提供敏感配置保护、执行上下文最小化注入、日志脱敏与结果回写校验；对于平台可信性风险，应确保控制面展示的状态、日志和产物始终可以被追溯和验证。

另一个常被忽略的风险是“用户误以为系统比实际更可靠”。因此产品在设计上必须区分：已计划、已派发、已启动、执行中、待恢复、待接管、已完成、已回写等关键状态，避免用模糊状态掩盖真实执行情况。平台的可信度来自透明，而不是来自表面自动化程度。

## Innovation & Novel Patterns

### Detected Innovation Areas

本产品的创新不在于单独引入 AI、任务管理、终端自动化或 BMAD 工件，而在于把这些原本割裂的层组合成一个统一的研发执行系统。my-bmad 试图建立一种新的产品范式：用户不再主要“操作工具”，而是“委托系统持续推进工作”。这使产品从传统项目管理平台、AI coding wrapper 或脚本自动化工具中区分出来。

第一类创新体现在 BMAD 工件流与执行控制面的深度耦合。现有系统通常只能管理文档、待办、代码或 AI 会话中的某一部分，而 my-bmad 将 PRD、Epic、Story、Task 与 agent run、`tmux` session、日志、产物、回写结果连接为一条可观测执行链路。它不是简单地从任务跳到 AI，而是把任务本身变成系统可执行、可治理、可追踪的对象。

第二类创新体现在多 agent 路由与本地执行基础设施的产品化。`codex` 和 `claude code` 不再只是两个独立工具，而是被纳入统一路由层，依据任务类型、上下文和策略自动选择。`tmux` 也不再只是终端技巧，而是被提升为平台执行面的基础设施。平台将这些技术性原语包装为任务生命周期的一部分，这是“把终端工作流产品化”的关键创新。

第三类创新体现在前台 `yolo`、后台严控的双层体验设计。多数自动化工具在体验和治理之间二选一：要么很灵活但不可控，要么很合规但很笨重。my-bmad 的创新尝试是让用户以接近 `yolo` 的方式交付目标，同时在后台保留审批、审计、权限、接管、回滚、状态分层与风险治理。这种双层设计如果成立，会使产品从“聪明工具”升级为“可信执行系统”。

### Market Context & Competitive Landscape

当前市场上的相关产品大致分为四类：项目管理平台、AI coding 助手、终端自动化工作流、以及面向工程团队的 DevOps 和 CI 工具。项目管理平台擅长记录任务，但不直接执行；AI coding 助手擅长生成内容，但缺少稳定治理；终端自动化擅长本地执行，但缺少协作与控制面；DevOps 工具擅长流水线，但主要围绕构建与部署，而非围绕研发任务本身。

my-bmad 的差异不在于某个单点功能更强，而在于它横跨这些类别：既理解 BMAD 工件与研发语义，又具备 agent 调度与后台执行能力，还能提供团队级治理与审计。竞争上，它面对的不是单一对手，而是多个工具组合的替代关系。用户当前可能使用“项目管理工具 + AI 聊天工具 + 本地终端 + 手工跟进”的拼装方案，my-bmad 试图把这套拼装流程收敛为一个统一控制面。

因此，市场验证的重点不应是“是否有功能重叠”，而应是“是否能显著降低拼装成本、跟进成本和执行不透明性”。如果产品能把原本分散在多个系统中的执行信息、状态判断和接管能力统一起来，它就具备形成新类别切口的机会。

### Validation Approach

创新是否成立，不能靠“看起来很酷”来判断，而必须靠执行闭环的真实性来验证。第一层验证是任务委托闭环验证：用户是否真的愿意把真实 Story 或 Task 交给系统，而不是只把它当演示工具。若用户持续把真实任务交给平台执行，说明“委托系统推进工作”的范式成立。

第二层验证是控制面可信性验证：用户是否相信平台显示的状态、日志、心跳、产物与实际执行一致。若用户仍然必须频繁打开底层终端自行核实，说明创新没有成立，平台仍只是一个外层包装。创新必须通过减少人工核对、减少工具切换和减少执行不透明来证明价值。

第三层验证是治理可接受性验证：团队负责人是否愿意在真实团队流程中开启自动执行、审批和接管机制。若团队只能把平台当个人工具，而不愿纳入团队治理，说明产品还没有真正跨过“工具”到“平台”的门槛。

第四层验证是替代拼装工具链的能力验证：产品是否能替代原本由多个工具共同完成的工作流，而不是再叠加一个新界面。若它只能提供更好的展示，却不能减少上下文整理、派发、监控、恢复和回写成本，则创新点不足以成立。

### Risk Mitigation

最主要的创新风险，是把多个已知概念组合在一起但没有形成真正的新价值，即“创新拼装感”强于“创新闭环”。为降低这一风险，产品应优先验证最核心的创新链路：从 BMAD 工件发起任务，到 agent 自动路由，到 `tmux` 后台执行，到状态可见，再到结果回写。只有这条链路被证明真实可用，其他扩展能力才有意义。

第二个风险，是过度强调 `yolo` 体验而削弱治理能力，导致产品在团队场景下失去信任。对此，平台必须坚持状态透明、审批边界、人工接管和审计记录为不可牺牲能力，确保“大胆自动化”始终建立在“可信控制”之上。

第三个风险，是把 `tmux`、agent 路由和本地执行等技术细节误当作用户价值本身。用户真正买单的不是底层技术组合，而是“可放心委托、可持续推进、可随时接管”的结果体验。因此创新表达必须始终围绕用户价值，而不是围绕底层实现炫技。

第四个风险，是产品试图过早覆盖所有角色、所有流程和所有自动化入口，导致创新范围过散。缓解方式是以“个人用户与小团队的真实任务闭环”为首个创新验证点，在此基础上再扩展到 API、批量编排、更复杂治理和更自治的执行策略。

## SaaS B2B Specific Requirements

### Project-Type Overview

my-bmad 作为一个 `saas_b2b` 产品，不只是提供单用户自动化体验，而是面向个人用户、小团队和未来企业团队，提供可治理、可协作、可商业化的研发执行控制面。它的核心不是传统 SaaS 的内容协作，而是围绕 AI 驱动执行任务建立稳定的团队运行机制：谁能发起、谁能审批、谁能接管、谁能查看、谁能配置路由、谁能审计历史，都必须具备明确边界。

该产品同时具备 developer tool 与 execution control plane 属性，因此其 SaaS 结构必须服务于执行治理，而不是只服务于账户管理。换句话说，多租户、权限、配额、审计和集成，不是运营附属功能，而是产品可信执行能力的一部分。

### Technical Architecture Considerations

平台应采用清晰的层级结构来支持商业化和团队化场景，至少包括：用户、组织或团队、项目、BMAD 工件、执行任务、agent run、`tmux` session、产物与审计事件。每一层对象都必须具备清晰归属关系，以保证权限判定、计费边界、审计归因与执行隔离可落地。

控制面应统一处理身份、授权、任务编排、状态聚合、审计记录、配额策略和 UI 展示；执行面应处理本地环境访问、agent 调用、会话管理、日志采集、心跳上报和产物生成。SaaS 架构设计的关键不是“全部云端”，而是“云端控制面 + 本地 self-hosted 执行面”的协同关系，这决定了平台必须具备强状态同步、边界清晰和异常容错机制。

平台还应支持从“个人模式”平滑升级到“团队模式”。也就是说，单用户项目不能成为独立分支产品，而应是同一租户模型在轻量配置下的特例。这样商业化和团队扩展才不会要求重做数据模型。

### Tenant Model

产品应支持以组织或团队为核心的租户模型。个人用户可以以“个人工作空间”形式使用平台，小团队和企业团队则以组织或团队租户方式管理多个项目、成员、策略和执行资源。租户边界必须覆盖数据访问、任务可见性、执行归属、审计记录和配额统计。

每个租户内应支持多个项目，每个项目可绑定独立的 BMAD 工件流、执行策略和 agent 配置。任务、session、日志和产物都必须明确归属于某个租户和项目，避免出现跨团队可见或跨项目执行污染。

对于 self-hosted 场景，租户模型还必须考虑“同一控制面下多个项目执行环境”的隔离需求。即使多个项目在同一台机器上运行，平台仍应在逻辑上保证项目级隔离、凭据边界和执行上下文分离。

### RBAC Matrix

平台必须提供面向团队协作和治理的角色分层。最低应覆盖以下角色能力：

- Workspace Owner / Organization Admin：管理成员、团队级策略、权限边界、计费与全局配置
- Product Manager / Planner：发起一句目标规划、选择或审核 BMAD skills 规划链路、确认 PRD、Epic、Story、Task 拆解结果，并决定何时进入编码执行
- Project Admin / Tech Lead：管理项目配置、执行策略、agent 路由、审批规则、验收节点
- Contributor / Operator：发起任务、查看执行状态、参与 review、按授权进行人工接管
- Support / Operations：查看异常任务、调查日志、执行恢复动作，但不默认拥有敏感配置修改权
- Auditor / Read-only Stakeholder：查看状态、里程碑和记录而不触发执行或修改配置

权限体系应支持至少三类动作控制：读取、执行、治理。读取决定可见性；执行决定是否能发起、重试、接管、终止任务；治理决定是否能改策略、改权限、改路由、导出审计或调整配额。平台必须避免“看得见就能动”“能运维就能改配置”这类粗粒度授权。

### Subscription Tiers

产品应从一开始为套餐与能力分层预留模型，即使早期不立即完全商业化。套餐边界不应只围绕 seat 数，而应围绕执行能力与治理能力分层。典型分层可包括：

- 个人层：单用户或轻量项目使用，提供基础任务执行与状态可见
- 团队层：多成员协作、项目共享、审批、基础治理
- 专业或高级团队层：更高任务量、更细权限、更多审计与告警能力
- 企业层：更强治理、策略控制、审计导出、配额管理、定制集成与支持

产品计费与配额设计应至少支持以下维度：成员数、项目数、任务执行量、并发执行量、日志保留、审计保留、集成能力等级。关键原则是：收费边界必须与用户感知价值一致，即围绕“可委托执行能力”和“团队治理能力”收费，而不是只围绕存储或页面访问收费。

### Integration List

平台必须支持与以下能力域的深度集成：

- BMAD 工件体系：PRD、Epic、Story、Task 的读取、映射与结果回写
- 本地执行环境：`tmux` session 生命周期、CLI agent 调用、日志与心跳采集
- 项目目录与代码上下文：项目根目录、产物路径、执行上下文范围限制
- API / Webhook / Event：外部任务触发、状态查询、关键事件订阅、回调通知
- 通知渠道：异常告警、审批提醒、任务完成通知
- 后续企业系统：审批系统、内部研发平台、CI/CD、报表与运营系统

集成设计必须遵循“控制面统一、执行面可替换、事件接口稳定”的原则，避免未来因单一 agent、单一执行方式或单一通知系统被架构锁死。

### Compliance Requirements

虽然该产品不以行业监管合规为首要门槛，但面向 B2B 商业化时，企业客户会默认要求基础治理能力。因此产品必须将以下能力视为 `saas_b2b` 基础要求而非后补特性：

- 全量关键操作审计
- 最小权限与角色分离
- 项目与租户边界隔离
- 敏感配置与凭据保护
- 关键状态与执行历史可追溯
- 数据保留与删除策略
- 恢复、接管与异常升级流程留痕

随着产品进入更高等级商业化阶段，架构应兼容更正式的安全治理要求，如更强审计导出、日志保留策略、组织级策略控制、企业级身份集成等。

### Implementation Considerations

`saas_b2b` 类型对本产品最关键的实现要求，不是“多几个管理页面”，而是从底层数据模型开始把团队治理、租户边界、任务执行与商业化约束放进同一个系统。实现时应优先保证以下几个设计原则：

- 个人模式与团队模式使用同一核心对象模型
- 每个执行单元都能回溯到租户、项目、工件与发起主体
- 权限判定与状态机设计必须覆盖执行生命周期，而不仅是页面访问
- 执行配额、审计记录与告警能力应从早期架构中预留
- 本地 self-hosted 执行面与云端控制面应通过明确契约交互，而不是隐式耦合

这类产品一旦在权限、边界和执行映射上设计失误，后续增长期会非常难修复。因此项目类型要求本身就意味着：核心架构必须以团队可治理和商业化可扩展为前提，而不能先做单机玩具，再试图往上补。

## Functional Requirements

### Workspace & Organization Management

- FR1: 个人用户可以创建和管理个人工作空间以承载其 BMAD 项目与执行任务。
- FR2: 组织管理员可以创建和管理团队工作空间，并在其中维护多个项目。
- FR3: 组织管理员可以邀请、移除和管理团队成员。
- FR4: 组织管理员可以为成员分配不同角色与权限范围。
- FR5: 组织管理员可以配置团队级执行策略、治理策略和默认控制规则。
- FR6: 用户可以在被授权的工作空间与项目范围内查看与执行任务。

### Project & BMAD Context Management

- FR7: 用户可以导入、关联和管理基于 BMAD 的项目上下文。
- FR8: 系统可以识别并维护 PRD、Epic、Story、Task 之间的层级关系。
- FR9: 用户可以从 PRD、Epic、Story 或 Task 上下文直接发起执行任务。
- FR10: 系统可以将执行任务与其来源工件保持可追踪映射关系。
- FR11: 用户可以查看某个 BMAD 工件关联的执行历史、状态和产物。
- FR12: 系统可以在执行完成、中断或失败后将关键结果回写到对应 BMAD 工件链路。

### Planning & PM Skill Orchestration

- FR13: 用户可以通过一句自然语言目标发起新的项目规划请求。
- FR14: 系统可以识别需要先规划后执行的请求，并选择合适的 PM 型 agent 与 BMAD skills 组合。
- FR15: 系统可以基于规划请求生成或更新 PRD、Epic、Story、Task 等 BMAD 工件。
- FR16: 系统可以将规划阶段产出的可执行任务自动衔接到后续编码执行链路。
- FR17: 用户可以查看从规划技能执行到编码任务派发的完整链路与状态。

### Task Orchestration & Agent Routing

- FR18: 用户可以为任务指定目标、优先级和执行意图。
- FR19: 系统可以根据任务类型、上下文和策略自动选择合适的 agent。
- FR20: 授权用户可以调整任务的路由策略或重新派发任务。
- FR21: 系统可以为任务创建、管理和结束后台执行会话。
- FR22: 系统可以维护任务、agent run 与后台执行会话之间的一致关联关系。
- FR23: 系统可以支持同一项目内多个任务的独立执行与状态隔离。
- FR24: 系统可以限制任务执行在其授权的项目与上下文边界内进行。

### Long-Running Execution & Agent Interaction

- FR25: 系统可以支持长时间运行的编码任务在同一执行链路中持续推进，而不因运行时间长而丢失上下文或状态可见性。
- FR26: 系统可以持续监听 `codex` 或 `claude code` 在执行过程中的输出、状态变化和交互请求。
- FR27: 系统可以在 `codex` 或 `claude code` 执行过程中接收并转发新的补充指令、确认信息或上下文输入。
- FR28: 授权用户可以对执行中的 agent 交互请求进行人工回应、批准或策略调整，而无需终止当前任务。
- FR29: 系统可以保留编码 agent 在执行期间的完整交互历史，并将其关联到对应任务、agent run 和审计事件。

### Execution Visibility & Progress Tracking

- FR30: 用户可以查看任务的当前状态、阶段和最近一次有效活动。
- FR31: 系统可以为每个执行任务记录并展示细粒度状态流转。
- FR32: 用户可以查看执行日志摘要以及原始执行记录。
- FR33: 系统可以展示任务的关键产物、输出结果和执行轨迹。
- FR34: 用户可以查看从 PRD、Epic、Story 到 Task、Agent Run、Session 的完整链路状态。
- FR35: 系统可以为执行任务提供心跳、活动更新和状态可信度信息。
- FR36: 用户可以在统一控制面中查看项目级和团队级执行进度。

### Recovery, Intervention & Operational Control

- FR37: 系统可以识别任务长时间无活动、状态异常或执行中断的情况。
- FR38: 系统可以对符合策略的异常任务执行自动恢复尝试。
- FR39: 用户可以对异常任务执行人工接管、重试、终止或改派操作。
- FR40: 支持或运维角色可以调查异常任务的上下文、日志和状态历史。
- FR41: 系统可以保留失败前后的关键上下文以支持恢复和复盘。
- FR42: 系统可以区分已计划、已派发、已启动、执行中、待恢复、待接管、已完成、已回写等关键状态。
- FR43: 系统可以向相关角色暴露需要处理的风险任务与异常任务队列。

### Review, Governance & Audit

- FR44: 团队负责人可以查看任务的执行摘要、关键产物和验收状态。
- FR45: 团队负责人可以对指定任务设置审批、放行或人工确认节点。
- FR46: 系统可以记录任务创建、派发、执行、恢复、接管、完成与回写等关键操作的审计事件。
- FR47: 授权用户可以检索、筛选和导出审计记录。
- FR48: 系统可以对不同角色提供差异化的读取、执行与治理权限。
- FR49: 系统可以在项目、团队和租户边界上隔离任务数据、执行记录和产物可见性。
- FR50: 审计或只读角色可以查看状态、里程碑和记录而不触发执行或修改配置。

### Team Collaboration & Notification

- FR51: 用户可以看到与自己相关的任务、审批、异常和结果通知。
- FR52: 系统可以向相关角色发送任务完成、异常、待审批和待接管通知。
- FR53: 团队成员可以围绕任务结果进行 review、确认或反馈。
- FR54: 团队负责人可以查看高风险任务、长时间阻塞任务和需要人工决策的任务。
- FR55: 系统可以支持任务在个人模式与团队模式中的一致协作流转。

### Integrations, APIs & Platform Extension

- FR56: 系统可以支持外部系统触发执行任务创建。
- FR57: 系统可以支持外部系统查询任务状态、执行结果和关键事件。
- FR58: 系统可以提供关键状态变化的事件通知或回调能力。
- FR59: 授权用户可以配置任务自动派发规则或触发条件。
- FR60: 系统可以与本地项目目录和代码上下文建立受控关联关系。
- FR61: 系统可以支持后续接入审批系统、通知系统、CI/CD 或内部研发平台。

### Commercialization, Quotas & Plan Controls

- FR62: 系统可以区分个人、团队和企业等不同服务层级。
- FR63: 系统可以对不同层级应用不同的执行能力、治理能力和集成能力边界。
- FR64: 组织管理员可以查看其工作空间的任务量、并发使用量或配额消耗情况。
- FR65: 系统可以对项目、团队或租户应用执行配额与使用限制策略。
- FR66: 系统可以为未来的计费、套餐和能力分层保留可管理的控制入口。

## Non-Functional Requirements

### Performance

- 关键用户操作必须具备可接受的交互响应时间：工作空间切换、项目进入、任务详情打开、状态筛选与执行链路视图加载在正常负载下应于 2 秒内完成首屏可见反馈。
- 新任务从创建到进入“已派发”或“已启动”状态的控制面可见反馈，在正常执行环境下中位时间应小于 30 秒。
- 执行任务的心跳或活动状态变化在控制面中的可见延迟应小于 15 秒。
- 最近一次执行日志摘要加载应在 3 秒内返回，原始日志按分页或分段方式加载时应在 5 秒内返回首批内容。
- 团队级任务列表、风险任务视图和审计查询在约定规模的数据量下应保持可操作，不得因单个大项目导致控制面不可用。

### Security

- 所有敏感数据必须在传输过程中加密，并对持久化的敏感配置与凭据进行安全存储。
- 系统必须对工作空间、团队、项目、任务、执行记录和产物实施严格的访问控制，未授权用户不得读取、触发、修改或导出相关数据。
- 执行任务必须被限制在其授权项目和上下文边界内，平台不得默认允许跨项目上下文泄露或敏感文件越权访问。
- 关键治理操作必须具备身份可追溯性，包括任务派发、路由调整、恢复、接管、审批、配置变更和审计导出。
- 系统必须支持最小权限原则，默认权限配置不得授予超出角色职责的执行或治理能力。
- 平台必须具备敏感信息暴露防护措施，避免在日志、状态摘要、通知或导出结果中意外泄露凭据、密钥或受限内容。

### Reliability & Recoverability

- 系统必须能够持续检测执行任务、agent run 与 `tmux` session 的状态一致性，并在状态漂移出现时标记异常而不是继续展示误导性状态。
- 对于心跳超时、session 中断、日志断流或回写失败等关键异常，系统必须在 15 秒内完成检测并进入明确的异常状态。
- 系统必须支持预定义范围内的自动恢复机制，并在恢复失败后提供清晰的人工接管路径。
- 任务执行过程中的关键上下文、最近有效状态和关键产物引用必须在异常后可恢复，用于支持重试、接管和复盘。
- 单个任务失败、单个 session 异常或单个项目执行问题不得导致整个工作空间或控制面不可用。
- 平台应具备明确的数据与状态一致性策略，避免出现“已显示完成但未回写”“已回写但状态未更新”“任务仍执行但控制面显示终止”等冲突状态长期存在。
- 平台必须支持长时间运行任务在数小时级别执行期间保持状态连续、监听连续和上下文连续，不得因轮询中断或会话切换导致任务失声。
- 对于 `codex` 或 `claude code` 发出的交互请求，系统不得静默丢弃；未被处理的交互请求必须进入明确状态并触发提醒或升级。

### Scalability

- 系统架构必须支持从个人模式平滑扩展到团队模式，而无需重建核心对象模型或权限模型。
- 系统必须支持多项目并行执行，并在团队级使用场景下维持任务列表、状态视图、审计检索和异常队列的可用性。
- 系统必须支持执行量、成员数、项目数和审计记录规模增长至少一个数量级，而不需要改变核心产品流程。
- 平台必须支持后续引入更高等级套餐、更多执行策略和更多外部集成，而不破坏既有租户边界与对象映射。
- 在超过设计容量时，系统必须优雅降级，例如限制新任务派发、延后非关键统计计算或标记队列拥塞，而不是直接进入不可预测状态。

### Accessibility

- 作为面向团队的 Web 控制平台，系统必须支持键盘可访问的核心操作流程，包括任务浏览、状态筛选、详情查看、审批和接管入口访问。
- 核心状态信息不得仅依赖颜色表达；任务风险、执行状态、异常类别和审批结果必须提供文本或等价语义标识。
- 控制面必须支持语义化结构与屏幕阅读器可理解的基础信息层级，至少覆盖导航、列表、详情、状态和关键操作区。
- 关键治理与异常处理路径必须满足明确可读性要求，避免因布局、对比度或交互复杂度导致重要状态被忽略。
- 产品应以 WCAG 2.1 AA 作为默认可访问性目标基线，至少覆盖核心工作流页面。

### Integration & Interoperability

- 平台对外提供的 API、Webhook 或事件接口必须具有稳定的契约边界，支持版本管理或兼容性演进。
- 系统必须能处理外部集成失败、超时、重复回调和幂等性问题，而不导致任务状态错乱。
- 外部系统触发的任务、事件订阅和状态查询必须遵循与内部任务一致的授权与审计规则。
- 平台必须支持本地执行面与控制面之间的稳定状态交换，即使网络、进程或执行环境出现短时波动，也不应造成长期不可恢复的不一致。

### Observability & Auditability

- 系统必须为每个任务、agent run 和执行 session 记录可追踪的生命周期事件。
- 关键审计事件必须包含时间、主体、对象、动作和结果，并支持后续筛选、导出和归因。
- 控制面必须区分正常执行、待恢复、待接管、已完成、已回写等状态，不得使用模糊状态隐藏不确定性。
- 平台必须支持对异常任务、恢复动作、人工接管和回写结果进行统一可观测追踪。
- 系统必须保留足够的日志、状态变更和审计证据，以支持问题调查、团队复盘和商业化客户审计需求。
- 当观测链路本身失效时，系统必须显式暴露“状态不可信”或等价信号，而不是默认显示成功或继续沉默。
- 系统必须对编码 agent 的交互提示、人工回应、自动回应和由此引起的状态变化进行完整追踪与审计。

## Initial Epic Candidates

### Epic 1: 一句话目标输入与 PM 规划编排

支持用户通过一句自然语言目标发起项目，系统自动识别需要先规划后执行的请求，并调用合适的 PM 型 agent 与 BMAD skills 生成初始规划结果。

### Epic 2: BMAD 工件生成、更新与追踪映射

支持系统生成或更新 PRD、Epic、Story、Task 等 BMAD 工件，并保持规划工件与后续执行任务之间的稳定映射关系。

### Epic 3: 任务路由与编码 Agent 自动派发

支持系统将规划阶段产出的可执行任务自动衔接到编码执行链路，并根据任务类型、上下文和策略自动路由到 `codex` 或 `claude code`。

### Epic 4: Self-Hosted 执行面与 `tmux` 会话控制

支持在与项目同机的 self-hosted 环境中创建、管理、追踪和结束 `tmux` 后台执行会话，确保任务执行具备稳定的生命周期控制，并支持长时间运行场景下的持续在线能力。

### Epic 5: 执行可视化、日志、心跳与状态机

支持从规划到执行的统一状态流转可视化，覆盖心跳、日志摘要、原始日志、关键产物、执行轨迹、交互提示和状态可信度展示。

### Epic 6: 编码 Agent 动态交互与长时间运行协调

支持在 `codex` 或 `claude code` 编码执行过程中持续监听 agent 输出、识别交互请求、动态注入补充信息，并在不终止任务的前提下完成系统或人工回应。

### Epic 7: 异常恢复、人工接管与执行治理

支持异常检测、失败分类、自动恢复、人工接管、审批节点、风险队列和审计留痕，确保自动化执行在失败场景下仍可控、可恢复。

### Epic 8: 团队协作、角色权限与商业化控制

支持个人与团队模式协作流转，覆盖 PM / Planner、Tech Lead、Contributor、Support、Auditor 等角色，以及租户边界、RBAC、配额和套餐能力控制。

### Epic 9: 集成接口、自动规则与外部事件联动

支持 API、Webhook、事件订阅、自动派发规则，以及与审批系统、通知系统、CI/CD 和内部研发平台的后续集成扩展。
