# my-bmad - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for my-bmad, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: 个人用户可以创建和管理个人工作空间以承载其 BMAD 项目与执行任务。
FR2: 组织管理员可以创建和管理团队工作空间，并在其中维护最多 50 个活跃项目；当项目数量超过上限时，系统必须阻止继续创建并提示归档、升级套餐或申请扩容。
FR3: 组织管理员可以邀请、移除和管理团队成员。
FR4: 组织管理员可以为成员分配不同角色与权限范围。
FR5: 组织管理员可以配置团队级执行策略、治理策略和默认控制规则。
FR6: 用户可以在被授权的工作空间与项目范围内查看与执行任务。
FR7: 用户可以导入、关联和管理基于 BMAD 的项目上下文。
FR8: 系统可以识别并维护 PRD、Epic、Story、Task 之间的层级关系。
FR9: 用户可以从 PRD、Epic、Story 或 Task 上下文直接发起执行任务。
FR10: 系统可以将执行任务与其来源工件保持可追踪映射关系。
FR11: 用户可以查看某个 BMAD 工件关联的执行历史、状态和产物。
FR12: 系统可以在执行完成、中断或失败后将关键结果回写到对应 BMAD 工件链路。
FR13: 用户可以通过一句自然语言目标发起新的项目规划请求。
FR14: 系统可以识别需要先规划后执行的请求，并选择合适的 PM 型 agent 与 BMAD skills 组合。
FR15: 系统可以基于规划请求生成或更新 PRD、Epic、Story、Task 等 BMAD 工件。
FR16: 系统可以将规划阶段产出的可执行任务自动衔接到后续编码执行链路。
FR17: 用户可以查看从规划技能执行到编码任务派发的完整链路与状态。
FR18: 用户可以为任务指定目标、优先级和执行意图。
FR19: 系统可以根据任务类型、上下文和策略自动选择合适的 agent。
FR20: 授权用户可以调整任务的路由策略或重新派发任务。
FR21: 系统可以为任务创建、管理和结束后台执行会话。
FR22: 系统可以维护任务、agent run 与后台执行会话之间的一致关联关系。
FR23: 系统可以支持同一项目内至少 10 个任务独立并发执行，并保持状态、日志和上下文三类隔离，避免一个任务的异常影响其他并发任务。
FR24: 系统可以限制任务执行在其授权的项目与上下文边界内进行。
FR25: 系统可以支持长时间运行的编码任务在同一执行链路中持续推进，而不因运行时间长而丢失上下文或状态可见性。
FR26: 系统可以持续监听 codex 或 claude code 在执行过程中的输出、状态变化和交互请求。
FR27: 系统可以在 codex 或 claude code 执行过程中接收并转发新的补充指令、确认信息或上下文输入。
FR28: 授权用户可以对执行中的 agent 交互请求进行人工回应、批准或策略调整，而无需终止当前任务。
FR29: 系统可以保留编码 agent 在执行期间的完整交互历史，并将其关联到对应任务、agent run 和审计事件。
FR30: 用户可以查看任务的当前状态、阶段和最近一次有效活动。
FR31: 系统可以为每个执行任务记录并展示细粒度状态流转。
FR32: 用户可以查看执行日志摘要以及原始执行记录。
FR33: 系统可以展示任务的关键产物、输出结果和执行轨迹。
FR34: 用户可以查看从 PRD、Epic、Story 到 Task、Agent Run、Session 的完整链路状态。
FR35: 系统可以为执行任务提供心跳、活动更新和状态可信度信息。
FR36: 用户可以在统一控制面中查看项目级和团队级执行进度。
FR37: 系统可以识别任务长时间无活动、状态异常或执行中断的情况。
FR38: 系统可以对符合策略的异常任务执行自动恢复尝试。
FR39: 用户可以对异常任务执行人工接管、重试、终止或改派操作。
FR40: 支持或运维角色可以调查异常任务的上下文、日志和状态历史。
FR41: 系统可以保留失败前后的关键上下文以支持恢复和复盘。
FR42: 系统可以区分已计划、已派发、已启动、执行中、待恢复、待接管、已完成、已回写等关键状态。
FR43: 系统可以向相关角色暴露需要处理的风险任务与异常任务队列。
FR44: 团队负责人可以查看任务的执行摘要、关键产物和验收状态。
FR45: 团队负责人可以对指定任务设置审批、放行或人工确认节点。
FR46: 系统可以记录任务创建、派发、执行、恢复、接管、完成与回写等关键操作的审计事件。
FR47: 授权用户可以检索、筛选和导出审计记录。
FR48: 系统可以对不同角色提供差异化的读取、执行与治理权限。
FR49: 系统可以在项目、团队和租户边界上隔离任务数据、执行记录和产物可见性。
FR50: 审计或只读角色可以查看状态、里程碑和记录而不触发执行或修改配置。
FR51: 用户可以看到与自己相关的任务、审批、异常和结果通知。
FR52: 系统可以向相关角色发送任务完成、异常、待审批和待接管通知。
FR53: 团队成员可以围绕任务结果进行 review、确认或反馈。
FR54: 团队负责人可以查看高风险任务、长时间阻塞任务和需要人工决策的任务。
FR55: 系统可以支持任务在个人模式与团队模式中的一致协作流转。
FR56: 系统可以支持外部系统触发执行任务创建。
FR57: 系统可以支持外部系统查询任务状态、执行结果和关键事件。
FR58: 系统可以提供关键状态变化的事件通知或回调能力。
FR59: 授权用户可以配置任务自动派发规则或触发条件。
FR60: 系统可以与本地项目目录和代码上下文建立受控关联关系。
FR61: 系统可以支持后续接入审批系统、通知系统、CI/CD 或内部研发平台。
FR62: 系统可以区分个人、团队和企业等不同服务层级。
FR63: 系统可以对不同层级应用不同的执行能力、治理能力和集成能力边界。
FR64: 组织管理员可以查看其工作空间的任务量、并发使用量或配额消耗情况。
FR65: 系统可以对项目、团队或租户应用执行配额与使用限制策略。
FR66: 系统可以为未来的计费、套餐和能力分层保留可管理的控制入口。

### NonFunctional Requirements

NFR1: 工作空间切换、项目进入、任务详情打开、状态筛选与执行链路视图加载在正常负载下应于 2 秒内完成首屏可见反馈。
NFR2: 新任务从创建到进入“已派发”或“已启动”状态的控制面可见反馈中位时间应小于 30 秒。
NFR3: 执行任务的心跳或活动状态变化在控制面中的可见延迟应小于 15 秒。
NFR4: 最近一次执行日志摘要加载应在 3 秒内返回，原始日志按分页或分段加载时应在 5 秒内返回首批内容。
NFR5: 团队级任务列表、风险任务视图和审计查询在单工作空间最多 100 个项目、10,000 个任务和 12 个月审计记录规模下，必须在 3 秒内返回首屏可见反馈。
NFR6: 所有敏感数据必须在传输过程中加密，并对持久化的敏感配置与凭据进行安全存储。
NFR7: 系统必须对工作空间、团队、项目、任务、执行记录和产物实施严格的访问控制。
NFR8: 执行任务必须被限制在其授权的项目与上下文边界内，平台不得默认允许跨项目上下文泄露或敏感文件越权访问。
NFR9: 关键治理操作必须具备身份可追溯性，包括任务派发、路由调整、恢复、接管、审批、配置变更和审计导出。
NFR10: 系统必须支持最小权限原则，默认权限配置不得授予超出角色职责的执行或治理能力。
NFR11: 平台必须具备敏感信息暴露防护措施，避免在日志、状态摘要、通知或导出结果中意外泄露凭据、密钥或受限内容。
NFR12: 系统必须能够持续检测执行任务、agent run 与 tmux session 的状态一致性，并在状态漂移出现时标记异常而不是继续展示误导性状态。
NFR13: 对于心跳超时、session 中断、日志断流或回写失败等关键异常，系统必须在 15 秒内完成检测并进入明确的异常状态。
NFR14: 对于心跳超时、session 中断和日志断流三类异常，系统必须在检测后 30 秒内发起首次自动恢复尝试，且单任务连续自动恢复尝试次数不得超过 2 次；恢复失败后必须进入明确的人工接管状态。
NFR15: 任务执行过程中的关键上下文、最近有效状态和关键产物引用必须在异常发生后 60 秒内可恢复。
NFR16: 单个任务失败、单个 session 异常或单个项目执行问题不得导致整个工作空间或控制面不可用。
NFR17: 对于“已显示完成但未回写”“已回写但状态未更新”“任务仍执行但控制面显示终止”等冲突状态，系统必须在 60 秒内检测并标记异常，并在异常解除前禁止继续显示最终成功状态。
NFR18: 平台必须支持长时间运行任务在数小时级别执行期间保持状态连续、监听连续和上下文连续。
NFR19: 对于 codex 或 claude code 发出的交互请求，系统不得静默丢弃；未被处理的交互请求必须进入明确状态并触发提醒或升级。
NFR20: 系统架构必须支持从个人模式平滑扩展到团队模式，而无需重建核心对象模型或权限模型。
NFR21: 系统必须支持多项目并行执行，并在团队级使用场景下维持任务列表、状态视图、审计检索和异常队列的可用性。
NFR22: 系统必须支持执行量、成员数、项目数和审计记录规模增长至少一个数量级，而不需要改变核心产品流程。
NFR23: 平台必须支持后续新增至少 1 个更高等级套餐、2 类新的执行策略和 5 个外部集成，而不需要变更既有租户主键、核心对象映射链路或权限模型。
NFR24: 在超过设计容量时，系统必须优雅降级，例如限制新任务派发、延后非关键统计计算或标记队列拥塞，而不是直接进入不可预测状态。
NFR25: 系统必须支持键盘可访问的核心操作流程，包括任务浏览、状态筛选、详情查看、审批和接管入口访问。
NFR26: 核心状态信息不得仅依赖颜色表达；任务风险、执行状态、异常类别和审批结果必须提供文本或等价语义标识。
NFR27: 控制面必须支持语义化结构与屏幕阅读器可理解的基础信息层级。
NFR28: 关键治理与异常处理路径必须满足明确可读性要求，避免重要状态被忽略。
NFR29: 产品应以 WCAG 2.1 AA 作为默认可访问性目标基线，至少覆盖核心工作流页面。
NFR30: 平台对外提供的 API、Webhook 或事件接口必须具有稳定的契约边界，支持版本管理或兼容性演进。
NFR31: 系统必须能处理外部集成失败、超时、重复回调和幂等性问题，而不导致任务状态错乱。
NFR32: 外部系统触发的任务、事件订阅和状态查询必须遵循与内部任务一致的授权与审计规则。
NFR33: 平台必须支持本地执行面与控制面之间的稳定状态交换，即使网络、进程或执行环境出现短时波动，也不应造成长期不可恢复的不一致。
NFR34: 系统必须为每个任务、agent run 和执行 session 记录可追踪的生命周期事件。
NFR35: 关键审计事件必须包含时间、主体、对象、动作和结果，并支持后续筛选、导出和归因。
NFR36: 控制面必须区分正常执行、待恢复、待接管、已完成、已回写等状态，不得使用模糊状态隐藏不确定性。
NFR37: 平台必须支持对异常任务、恢复动作、人工接管和回写结果进行统一可观测追踪。
NFR38: 系统必须保留不少于 90 天的日志与状态变更记录，以及不少于 180 天的关键审计证据。
NFR39: 当观测链路本身失效时，系统必须显式暴露“状态不可信”或等价信号，而不是默认显示成功或继续沉默。
NFR40: 系统必须对编码 agent 的交互提示、人工回应、自动回应和由此引起的状态变化进行完整追踪与审计。

### Additional Requirements

- Architecture 明确要求继续沿用现有 Next.js 16 App Router brownfield 基座；Epic 1 / Story 1 应先建立并验证扩展基础，而不是重新初始化新仓库。
- Architecture 虽给出了 `pnpm create next-app@latest my-bmad --ts --tailwind --eslint --app --src-dir --import-alias "@/*"` 作为 greenfield 参考，但当前项目不应执行该初始化命令。
- 系统必须采用“模块化单体控制面 + 同机执行监督器”结构：Web 控制面负责身份、授权、任务编排、状态聚合、审计与 UI；执行监督器负责 tmux session 生命周期、agent 调用、日志采集、心跳上报与恢复动作。
- MVP 执行面限定为与项目同机的 single-host self-hosted 模式，通过 tmux 承载后台执行；多主机分布式调度延后到后续阶段。
- PostgreSQL 必须作为系统事实源，并继续沿用 Prisma 迁移策略进行前向演进。
- 统一领域模型必须覆盖 Workspace、Project、BMAD Artifact、Task、Agent Run、Session、Prompt Dispatch、Interaction、Heartbeat、Artifact、Writeback、Audit Event。
- 核心执行模型必须采用“状态机 + 事件日志”双模型：状态机承载当前真值，事件链路承载可追溯历史。
- 授权模型必须继续采用 Better Auth + 服务端 RBAC / Policy Layer，并把权限明确区分为读取、执行、治理三类。
- 内部控制面变更优先使用 Server Actions，并统一返回 `ActionResult<T>`；外部触发、查询和回调使用版本化 REST Route Handlers。
- 实时监控通道必须优先采用 SSE，并提供轮询兜底策略。
- 所有输入边界、执行监督器上报、配置写入与回写请求必须统一使用 Zod 4 校验。
- 原始长日志与较大产物不应进入高频事务热表；数据库应保存引用、摘要、偏移、校验信息与访问控制元数据。
- 实现必须兼容当前同时支持 `github` 与 `local` 两类仓库来源，不能把执行与上下文能力绑定为单一仓库源。
- 本地文件访问必须继承既有安全边界：限制根目录、拒绝路径穿越、拒绝符号链接，并限制深度、文件数与文件大小。
- 关键 mutation 后必须遵循现有缓存失效模式，使用明确的 `revalidatePath()` 或 `revalidateTag()` 维持控制面数据一致性。
- 部署与运行角色必须延续 Docker + Traefik 方向，并在运行时拆分至少 `web` 与 `executor-supervisor` 两类角色。
- 实现阶段必须为关键状态机、恢复逻辑、权限边界和路径安全建立回归测试。
- 首个实现优先级应先定义统一领域模型、状态机以及控制面 / 执行监督器契约，而不是先做局部 UI 或零散 API。

### UX Design Requirements

UX-DR1: 产品必须提供桌面优先的 Web 控制面，并将整体信息架构组织为“总览信号 → 对象详情 → 证据时间线”三层结构。
UX-DR2: 用户应能从一句自然语言目标、Story、Task 或既有 BMAD 工件以低摩擦方式发起任务，并在提交后立即看到当前阶段、系统正在做什么、下一步预计是什么。
UX-DR3: 控制塔首页必须优先展示任务健康度、待响应事项、风险队列、关键运行状态和推荐动作，而不是先暴露所有底层对象模型。
UX-DR4: 任务详情与运行视图必须持续显示当前阶段、最近有效活动、最后心跳、风险级别、阻塞点和下一步建议，不能仅用“运行中”表达状态。
UX-DR5: 产品必须实现 `Execution Status Rail` 组件，用于在列表与详情页表达阶段节点、当前高亮阶段、最近活动时间、风险标记和下一步提示，并支持紧凑列表型、详情型和横向时间线型三种变体。
UX-DR6: `Execution Status Rail` 的每个阶段必须提供文本语义，当前阶段应具备 `aria-current` 或等价可访问性语义。
UX-DR7: 产品必须实现 `Interaction Request Card` 组件，用于展示 agent 在运行中发起的提问、授权请求或上下文补充请求，并支持直接批准、补充说明、驳回、改派或进入人工接管。
UX-DR8: `Interaction Request Card` 必须包含请求标题、请求来源、上下文摘要、建议动作、响应输入区，以及超时或优先级提示，并支持键盘快速聚焦与提交。
UX-DR9: 产品必须实现 `Recovery Timeline Panel` 组件，用于展示异常检测、自动恢复尝试、升级与人工处置的完整时间链，并支持从事件节点跳转到相关日志、session 或审计详情。
UX-DR10: 产品必须实现 `Evidence Chain Viewer` 组件，把 Story、Task、Run、Session、Artifact、Writeback 之间的关系以可追踪方式展示，并支持识别部分缺失、断链和回写失败状态。
UX-DR11: 产品必须实现 `Risk Queue Item` 组件，将异常、阻塞、审批缺失、长期无产出等高风险事项收敛为可排序、可筛选、可处置的统一列表单元。
UX-DR12: 产品必须为团队负责人提供 Review / 决策视图，以高信噪比方式展示摘要日志、关键产物、执行链路和风险提示，并支持批准继续、要求重跑、转人工或审批拒绝等动作。

UX-DR16: 布局与间距系统必须采用 8px 主节奏和 4px 微调节奏，并支持桌面多栏控制面布局，使列表、详情和上下文可并置显示以减少来回跳转。

UX-DR19: 表单模式必须优先支持高价值、小步输入，并在填写时持续显示其影响的任务、run 或 Story 上下文；高风险配置提交前必须展示影响范围摘要。

UX-DR22: 动效与加载体验必须避免无意义动画，并对长时间运行流程优先采用阶段化加载与结构骨架，而不是统一 loading spinner。
UX-DR23: 搜索与筛选必须支持任务 ID、Story 名称、run ID、session 名、状态、负责人和风险等级等高频字段，并提供待响应、异常、review-ready、最近活跃等快捷切换。
UX-DR24: 导航模式必须保持来源上下文连续性，支持面包屑、返回来源队列和跨对象跳转后的链路感知，避免用户在风险页、任务详情和 run 时间线之间迷失。
UX-DR25: Modal 与 Overlay 只应用于高确认密度或短流程任务；需要大量上下文的动作必须优先采用抽屉或内联展开，而不是切断上下文的大模态。
UX-DR26: Empty / Loading State 必须说明为什么为空、系统现在能做什么、用户下一步可以如何开始，并在长时间运行场景中优先展示阶段提示而不是单纯旋转等待。

## Epic 3: 一句话目标到规划结果

**And** 用户立即看到请求已接收的反馈，包含当前阶段和系统即将执行的步骤（UX-DR2）

**Given** 用户提交的目标文本为空或过短
**When** 用户点击提交
**Then** 系统提示需要更明确的目标描述，不创建请求

**Given** 规划请求已创建
**When** 用户查看请求状态

**Given** 系统已完成意图识别
**When** 选择结果确定
**Then** 系统记录选择的 agent 类型、skill 组合和选择理由
**And** 将规划请求状态更新为"规划中"

**Given** 请求被判定为可直接执行（如"修复登录页面的按钮颜色"）
**When** 系统完成意图分析
**Then** 系统跳过规划链路，将请求路由到执行链路（衔接 Epic 4）
**And** 通知用户该请求将直接进入执行

### Story 3.3: 规划执行与 BMAD 工件生成

作为用户，
我希望系统能基于我的规划请求，通过 PM 型 agent 和 BMAD skills 自动生成或更新 PRD、Epic、Story、Task 等工件，

**And** 用户可以查看、编辑或确认规划产出

**Given** 规划执行中某个 skill 失败

**Then** 系统标记失败步骤和原因，不影响已成功完成的工件

## Epic 4: 任务派发与 Self-Hosted 执行启动

**目标：** 让用户能够定义任务目标与执行意图，由系统在受控边界内选择合适 agent、创建后台执行会话，并稳定启动 self-hosted 执行链路。

### Story 4.1: 任务定义与执行意图提交

**When** 用户填写任务表单

**And** 表单持续显示关联的 Story/Epic 上下文（UX-DR19）

**Given** 用户提交任务
**When** 表单验证通过（Zod 4 校验）
**Then** 系统创建 Task 记录，状态为 PLANNED，包含 projectId、workspaceId、目标、优先级、意图、来源工件引用
**And** 用户立即看到任务已创建的反馈和当前阶段（FR18, UX-DR2）

### Story 4.2: Agent 自动路由与选择

作为系统，

**验收标准：**

**Given** 任务已创建且状态为 PLANNED

**When** 路由逻辑执行
**Then** 优先使用工作空间级策略，任务级用户偏好可覆盖默认策略

**Given** agent 路由完成
**When** Task 状态更新
**Then** Task 状态从 PLANNED 变为 DISPATCHED
**And** 创建 AgentRun 记录，关联到 Task，包含选定的 agent 类型、运行 ID 和创建时间

**Then** 系统更新 AgentRun 的 agent 类型或创建新的 AgentRun（FR20）
**And** 记录路由调整的审计事件

**Given** 任务已在执行中（状态为 RUNNING）
**When** 授权用户请求重新派发
**Then** 系统先终止当前执行会话，保留已有上下文
**And** 创建新的 AgentRun 并重新派发
**And** 展示确认对话框说明重新派发的影响（UX-DR17）

**Given** VIEWER 角色用户查看任务
**When** 用户尝试调整路由或重新派发
**Then** 系统拒绝操作，按角色权限返回权限不足错误

### Story 4.4: tmux 后台执行会话创建与管理

作为系统，
我希望能为任务创建、管理和结束 tmux 后台执行会话，
以便编码 agent 在 self-hosted 环境中稳定运行。

**验收标准：**

**Given** AgentRun 已创建且准备执行
**When** 执行监督器启动任务
**Then** 创建独立的 tmux session，session 名称包含任务 ID 以便识别
**And** 创建 ExecutionSession 记录，关联到 AgentRun，包含 session 名称、PID、启动时间（FR21）
**And** 在 tmux session 中启动选定的 agent（codex 或 claude code）

**Given** tmux session 已创建
**When** 查看 ExecutionSession 记录
**Then** 记录包含 sessionName、agentRunId、taskId、projectId、workspaceId、状态和时间戳
**And** Task、AgentRun 与 ExecutionSession 之间的关联关系完整且一致（FR22）

**Given** 任务执行完成或被终止
**When** 执行监督器检测到结束条件
**Then** 执行 tmux session 清理（关闭 session）
**And** ExecutionSession 状态更新为 COMPLETED 或 TERMINATED
**And** 清理动作记录到审计日志

### Story 4.5: 多任务并发执行与隔离

作为系统，
我希望支持同一项目内至少 10 个任务独立并发执行，并保持状态、日志和上下文三类隔离，
以便一个任务的异常不影响其他并发任务。

**验收标准：**

**Given** 项目内已有 9 个任务并发执行
**When** 第 10 个任务进入执行
**Then** 第 10 个任务成功创建独立的 tmux session 并开始执行（FR23）
**And** 所有 10 个任务的状态、日志和上下文互不干扰

**Given** 10 个任务并发执行中
**When** 第 3 个任务因异常崩溃
**Then** 仅第 3 个任务的 ExecutionSession 进入异常状态
**And** 其余 9 个任务不受影响，继续正常执行
**And** 第 3 个任务的日志和上下文保留用于排查

**Given** 项目并发任务数达到配额上限
**When** 用户尝试启动新任务
**Then** 系统将任务排入等待队列，并告知用户当前并发数和预估等待
**And** 不直接拒绝而是优雅降级（NFR24）

### Story 4.6: 执行边界与上下文安全控制

作为系统，
我希望限制任务执行在其授权的项目与上下文边界内进行，
以便防止跨项目上下文泄露或未授权的文件访问。

**验收标准：**

**Given** 任务属于项目 X
**When** 执行监督器配置 tmux session 环境
**Then** 工作目录限定为项目 X 的根目录
**And** 上下文注入仅包含项目 X 授权范围内的文件和配置（FR24）

**Given** 执行中的 agent 尝试访问项目根目录外的路径
**When** 执行监督器检测到路径穿越尝试
**Then** 访问被拒绝，事件记录到审计日志
**And** 任务不因此中断，但异常行为被标记

**Given** 项目关联了本地目录
**When** 执行监督器配置执行环境
**Then** 符号链接访问被拒绝
**And** 文件访问深度、文件数和文件大小受限
**And** 敏感路径（如 .env、密钥文件）被保护，不注入到 agent 上下文

## Epic 5: 执行控制塔：状态、日志、心跳与动态交互

**目标：** 让用户能够在统一控制面中观察任务执行状态、日志、心跳、证据链与进度，并在长时间运行过程中直接处理 agent 的交互请求。

**FRs covered:** FR25, FR26, FR27, FR28, FR29, FR30, FR31, FR32, FR33, FR34, FR35, FR36

### Story 5.1: 任务状态机与细粒度状态流转

作为平台开发者，
我希望建立统一的任务执行状态机，覆盖从计划到回写的完整生命周期，
以便所有状态变化都经过受控流转且可追溯。

**验收标准：**

**Given** Task 领域模型已建立
**When** 状态机定义完成
**Then** 状态机覆盖以下关键状态：PLANNED、DISPATCHED、STARTING、RUNNING、WAITING_FOR_INPUT、RECOVERING、AWAITING_TAKEOVER、COMPLETED、FAILED、TERMINATED、WRITEBACK_PENDING、WRITEBACK_DONE（FR42）
**And** 每次状态变化产生 TaskStateEvent 记录，包含 fromState、toState、triggeredBy、timestamp、reason

**Given** 任务处于 RUNNING 状态
**When** 系统尝试将其转为 PLANNED
**Then** 状态机拒绝非法转换，记录拒绝事件
**And** 当前状态保持不变

**Given** 状态变化发生
**When** 查看任务详情
**Then** 展示状态流转时间线，每个节点包含状态、时间和触发原因（FR31）

### Story 5.2: 长时间运行任务的状态连续与上下文保持

作为用户，
我希望长时间运行的编码任务在数小时级别执行期间保持状态连续、监听连续和上下文连续，
以便我不因运行时间长而丢失可见性。

**验收标准：**

**Given** 任务已运行超过 1 小时
**When** 用户查看任务详情
**Then** 状态、最近活动和心跳信息仍然持续更新（FR25, NFR18）
**And** 不因运行时间长而降级为"状态未知"

**Given** 执行监督器与控制面之间出现短时网络波动
**When** 连接恢复
**Then** 状态同步恢复，波动期间的事件不丢失
**And** 控制面展示恢复后的最新状态而非波动前的缓存状态

**Given** 任务运行数小时
**When** 用户查看执行轨迹
**Then** 所有阶段、活动和产物的时间线完整可见
**And** 不因数据量增长而导致页面加载超时

### Story 5.3: Agent 输出监听与实时状态推送

作为系统，
我希望能持续监听 codex 或 claude code 在执行过程中的输出、状态变化和交互请求，并将关键更新推送到控制面，
以便用户获得准实时的执行可见性。

**验收标准：**

**Given** 任务在 tmux session 中执行
**When** agent 产出新的输出内容
**Then** 执行监督器捕获输出并提取关键状态变化和进度信息（FR26）
**And** 通过 SSE 将状态更新推送到控制面，延迟小于 15 秒（NFR3）

**Given** agent 输出中包含交互请求（如提问、确认请求）
**When** 执行监督器识别到交互请求模式
**Then** 系统创建 InteractionRequest 记录并推送到控制面
**And** 任务状态变为 WAITING_FOR_INPUT

**Given** SSE 连接断开
**When** 用户刷新页面或重新连接
**Then** 系统通过轮询兜底获取最新状态（架构要求）
**And** 不丢失断开期间的状态变化

### Story 5.4: 执行中补充指令与上下文输入

作为用户，
我希望能在 agent 执行过程中发送补充指令、确认信息或上下文输入，
以便在不终止任务的情况下引导执行方向。

**验收标准：**

**Given** 任务处于 RUNNING 或 WAITING_FOR_INPUT 状态
**When** 用户在任务详情的交互面板中输入补充指令并提交
**Then** 系统将指令通过执行监督器转发到对应 tmux session 中的 agent（FR27）
**And** 指令记录到 Interaction 记录，关联任务和 AgentRun

**Given** 用户提交确认信息（如批准 agent 的某个提议）
**When** 确认信息发送到 agent
**Then** agent 继续执行，任务状态从 WAITING_FOR_INPUT 变回 RUNNING
**And** 确认动作记录到审计日志

**Given** 用户发送的补充指令
**When** 传递给 agent 时
**Then** 指令内容不被截断或篡改
**And** 指令发送后用户看到"已发送"反馈

### Story 5.5: 人工回应 Agent 交互请求

作为授权用户，
我希望能对执行中 agent 发出的交互请求进行人工回应、批准或策略调整，而无需终止当前任务，
以便保持执行连续性的同时实现人机协同。

**验收标准：**

**Given** agent 发出交互请求（如"需要确认是否删除旧表"）
**When** 用户查看交互请求卡片
**Then** 卡片展示请求标题、来源、上下文摘要、建议动作和响应输入区（UX-DR7, UX-DR8）
**And** 支持直接批准、补充说明、驳回、改派或进入人工接管

**Given** 用户选择"批准"某个交互请求
**When** 用户提交批准
**Then** 批准决策转发到 agent，任务继续执行（FR28）
**And** 批准记录关联到 InteractionRequest、AgentRun 和审计事件

**Given** 交互请求超时未被处理
**When** 超时时间到达
**Then** 系统不静默丢弃，而是进入明确状态并触发提醒或升级（NFR19）
**And** 交互请求卡片展示超时提示和建议动作

**Given** 交互请求需要键盘快速处理
**When** 用户使用键盘导航
**Then** 交互请求卡片支持键盘快速聚焦与提交（UX-DR8）

### Story 5.6: 完整交互历史记录与审计关联

作为用户，
我希望系统保留编码 agent 在执行期间的完整交互历史，并将其关联到对应任务、agent run 和审计事件，

**Given** 某次交互导致了 agent 行为变化
**When** 查看交互详情
**Then** 展示交互前后的 agent 状态变化

**Given** 审计角色查看交互历史
**When** 检索特定时间范围或类型的交互
**Then** 支持按交互类型（补充指令/确认/批准/驳回）和时间范围筛选（NFR40）

### Story 5.7: 任务当前状态、阶段与最近活动视图

作为用户，
我希望能查看任务的当前状态、阶段和最近一次有效活动，

### Story 5.8: 执行日志摘要与原始日志查看

作为用户，

## Epic 6: 异常恢复、人工接管与风险处置

**目标：** 让系统能够发现异常、自动尝试恢复，并在需要时把任务清晰升级为可人工接管、可调查、可处置的风险项。

**FRs covered:** FR37, FR38, FR39, FR40, FR41, FR42, FR43

### Story 6.1: 异常检测与状态漂移识别

作为系统，
我希望能识别任务长时间无活动、状态异常或执行中断的情况，
以便在问题扩大前及时发现并标记。

**验收标准：**

**Given** 任务在 RUNNING 状态但心跳超时超过配置阈值
**When** 系统执行异常检测逻辑
**Then** 15 秒内完成检测并将任务标记为异常状态（FR37, NFR13）
**And** 异常类型被分类（心跳超时/session 中断/日志断流/回写失败）

**Given** 执行监督器检测到 tmux session 已不存在
**When** 系统确认 session 中断
**Then** 任务状态标记为异常，不继续展示"运行中"（NFR12）
**And** 创建异常事件记录

**Given** 任务状态与 session 实际状态不一致（如"已显示完成但未回写"）
**When** 系统检测到冲突状态
**Then** 60 秒内检测并标记异常（NFR17）
**And** 在异常解除前禁止继续显示最终成功状态

### Story 6.2: 自动恢复尝试

作为系统，
我希望能对符合策略的异常任务执行自动恢复尝试，
以便在无需人工介入的情况下恢复执行。

**验收标准：**

**Given** 任务被标记为心跳超时或 session 中断异常
**When** 自动恢复策略已启用
**Then** 系统在检测后 30 秒内发起首次自动恢复尝试（NFR14）
**And** 任务状态变为 RECOVERING

**Given** 首次自动恢复尝试失败
**When** 系统评估恢复结果
**Then** 在合理间隔后发起第二次恢复尝试
**And** 单任务连续自动恢复尝试次数不超过 2 次（NFR14）

**Given** 两次自动恢复尝试均失败
**When** 恢复次数达到上限
**Then** 任务状态变为 AWAITING_TAKEOVER（NFR14）
**And** 系统通知相关角色需要人工接管

**Given** 自动恢复成功
**When** session 恢复且心跳正常
**Then** 任务状态从 RECOVERING 变回 RUNNING
**And** 恢复事件记录到审计日志，包含恢复方式和耗时

### Story 6.3: 人工接管、重试、终止与改派

作为用户，
我希望能对异常任务执行人工接管、重试、终止或改派操作，

**验收标准：**

**Given** 任务处于 AWAITING_TAKEOVER 状态
**When** 授权用户查看任务详情
**Then** 展示可用操作：人工接管、重试、终止、改派（FR39）
**And** 每个操作附带说明和预期影响

**Given** 用户选择"重试"
**When** 用户确认重试操作
**Then** 系统创建新的 AgentRun 和 ExecutionSession，从失败点或从头开始重试
**And** 原 AgentRun 保留完整历史

**Given** 用户选择"终止"
**When** 用户确认终止操作（UX-DR17 危险动作确认）
**Then** 系统清理 tmux session，任务状态变为 TERMINATED

**Given** 用户选择"改派"到不同 agent
**When** 用户指定新的 agent 类型并确认
**Then** 系统终止当前执行，创建新的 AgentRun 并派发到新 agent
**And** 改派动作记录到审计日志

**Given** 用户选择"人工接管"
**When** 用户确认接管
**Then** 任务标记为人工处理中，系统暂停自动化干预
**And** 用户可以直接操作 tmux session 或补充指令

### Story 6.4: 异常调查与上下文保留

作为支持或运维角色，

**验收标准：**

**Given** 支持角色需要查看完整状态历史

## Epic 8: 平台集成、自动规则与外部联动

**目标：** 让外部系统能够触发任务、查询状态、订阅事件，并让授权用户配置自动派发规则，将 my-bmad 纳入更大研发流程。

**FRs covered:** FR56, FR57, FR58, FR59, FR60, FR61

### Story 8.1: 外部系统触发执行任务创建

作为外部系统集成方，
我希望能通过 API 触发 my-bmad 中的执行任务创建，
以便将 my-bmad 嵌入到现有 CI/CD 或研发流程中。

**验收标准：**

**Given** 外部系统持有有效的 API token 且具有执行权限
**When** 外部系统调用 POST /api/v1/tasks 并提供项目 ID、任务目标和优先级
**Then** 系统创建 Task 记录并返回任务 ID 和初始状态（FR56）
**And** 请求经过 Zod 4 校验、速率限制和授权检查

**Given** 外部系统提供无效的项目 ID 或缺失必填字段
**When** API 调用执行
**Then** 返回结构化错误响应 `{ error: { code, message, details } }`
**And** 不创建任何记录

**Given** 外部系统重复发送相同的幂等键
**When** API 收到重复请求
**Then** 返回已创建的任务信息，不重复创建（NFR31）

**Given** 外部系统触发的任务
**When** 任务执行
**Then** 遵循与内部任务一致的授权与审计规则（NFR32）

### Story 8.2: 外部系统查询任务状态与执行结果

作为外部系统集成方，
我希望能查询 my-bmad 中任务的状态、执行结果和关键事件，
以便在外部系统中展示或决策。

**验收标准：**

**Given** 外部系统持有有效的 API token
**When** 外部系统调用 GET /api/v1/tasks/{taskId}
**Then** 返回任务当前状态、阶段、最近活动、关键产物引用和执行链路摘要（FR57）
**And** 返回数据遵循版本化响应格式

**Given** 外部系统查询任务列表
**When** 调用 GET /api/v1/tasks?projectId={id}&status={status}
**Then** 返回分页任务列表，支持按状态、优先级和时间范围筛选

**Given** 外部系统无权访问某项目
**When** 查询该项目的任务
**Then** 返回 403 错误，不泄露任何任务数据

### Story 8.3: 关键状态变化的事件通知与回调

作为外部系统集成方，
我希望能订阅 my-bmad 中关键状态变化的事件通知或回调，
以便实时响应任务进展。

**验收标准：**

**Given** 外部系统配置了 Webhook 回调 URL
**When** 任务状态发生关键变化（完成、失败、待审批、待接管等）
**Then** 系统向配置的 URL 发送事件通知（FR58）
**And** 事件载荷包含 eventType、taskId、projectId、状态变化和时间戳

**Given** Webhook 回调失败（目标不可达或超时）
**When** 系统检测到回调失败
**Then** 系统按指数退避策略重试，最多重试 3 次（NFR31）
**And** 重试仍失败则记录失败事件，不影响任务本身执行

**Given** 外部系统发送重复的回调确认
**When** 系统收到重复确认
**Then** 幂等处理，不产生重复副作用

### Story 8.4: 任务自动派发规则配置

作为授权用户，
我希望能配置任务自动派发规则或触发条件，
以便特定场景下任务无需手动派发即可自动执行。

**验收标准：**

**Given** 管理员进入自动派发规则配置页面
**When** 管理员创建新规则（如"来自 CI 触发的高优先级任务自动派发"）
**Then** 规则保存到项目或工作空间配置（FR59）
**And** 规则包含触发条件、匹配范围和目标动作

**Given** 新任务创建且匹配某条自动派发规则
**When** 系统评估规则
**Then** 任务自动进入派发流程，无需人工操作
**And** 自动派发动作记录到审计日志

**Given** 管理员修改或禁用自动派发规则
**When** 配置变更提交
**Then** 变更立即生效，后续任务按新规则处理
**And** 变更记录审计事件

### Story 8.5: 本地项目目录与代码上下文受控关联

作为系统，

**Then** 系统在项目根目录边界内建立受控关联（FR60）

**When** 执行 agent 访问代码上下文

### Story 8.6: 外部系统集成扩展预留

作为系统，

**Given** 未来需要接入新的外部系统
**When** 集成开发启动
**Then** 核心对象模型、权限模型和审计模型不需要变更
**And** 新集成通过标准 API + Webhook + 事件订阅模式接入

我希望能区分个人、团队和企业等不同服务层级，
以便按层级提供差异化能力。

**验收标准：**

**Given** 平台需要支持多种服务层级
**When** 服务层级模型建立
**Then** 系统定义至少三个层级：个人（Free/Pro）、团队（Team）、企业（Enterprise）（FR62）
**And** 每个层级包含名称、能力边界定义和默认配额

**Given** 工作空间关联到某个服务层级
**When** 查看工作空间信息
**Then** 展示当前服务层级和对应的能力范围

**Given** 系统需要后续新增更高等级套餐
**When** 新增套餐定义
**Then** 不需要变更既有租户主键、核心对象映射链路或权限模型（NFR23）

### Story 9.2: 按层级应用能力边界

作为系统，
我希望能对不同层级应用不同的执行能力、治理能力和集成能力边界，
以便不同层级用户获得与其套餐匹配的功能。

**验收标准：**

**Given** 个人层级工作空间
**When** 用户使用平台功能
**Then** 执行能力受限（如并发任务数上限较低）、治理能力受限（如无审批节点配置）、集成能力受限（如无 API 触发）（FR63）

**Given** 团队层级工作空间
**When** 管理员使用平台功能
**Then** 解锁更高并发上限、审批和 Review 功能、基础 API 集成能力

**Given** 企业层级工作空间
**When** 管理员使用平台功能
**Then** 解锁最高并发上限、完整治理功能、完整 API 和 Webhook 集成、高级审计导出

**Given** 用户尝试使用超出当前层级的功能
**When** 功能入口被访问
**Then** 系统展示升级提示而非报错，说明该功能需要哪个层级（UX-DR26 引导体验）

### Story 9.3: 配额消耗查看与使用量仪表盘

作为组织管理员，
我希望能查看工作空间的任务量、并发使用量或配额消耗情况，
以便了解当前使用状态和剩余容量。

**验收标准：**

**Given** 管理员进入工作空间使用量页面
**When** 页面加载
**Then** 展示关键指标：当前活跃项目数/上限、本月任务执行数、当前并发任务数/上限、审计存储用量（FR64）
**And** 每项指标包含当前值、上限值和使用百分比

**Given** 配额接近上限（如使用超过 80%）
**When** 使用量页面加载
**Then** 对应指标高亮警告，提示管理员关注

**Given** 管理员查看历史使用趋势
**When** 选择时间范围
**Then** 展示按日/周/月的使用趋势图表

### Story 9.4: 执行配额与使用限制策略

作为系统，
我希望能对项目、团队或租户应用执行配额与使用限制策略，
以便防止资源滥用并确保公平使用。

**验收标准：**

**Given** 工作空间的服务层级已确定
**When** 任务执行请求到达
**Then** 系统检查当前并发数是否在配额内，超出则排队或拒绝（FR65）
**And** 检查本月任务执行总数是否在限额内

**Given** 配额已用尽
**When** 用户尝试创建新任务
**Then** 系统阻止创建并展示明确提示：当前配额已用尽，建议等待配额重置或升级套餐
**And** 不直接进入不可预测状态（NFR24）

**Given** 超过设计容量
**When** 系统负载过高
**Then** 系统优雅降级：限制新任务派发、延后非关键统计计算或标记队列拥塞（NFR24）

### Story 9.5: 计费与能力分层控制入口预留

作为系统，
我希望为未来的计费、套餐和能力分层保留可管理的控制入口，
以便后续商业化功能无需重构核心架构。

**验收标准：**

**Given** 平台需要支持未来计费功能
**When** 评估商业化扩展性
**Then** 服务层级、配额和使用量数据模型已建立，可直接关联计费系统（FR66）
**And** 配额边界和层级切换通过配置而非硬编码实现

**Given** 未来需要新增 2 类执行策略和 5 个外部集成
**When** 扩展开发启动
**Then** 不需要变更既有租户主键、核心对象映射链路或权限模型（NFR23）

**Given** 管理员需要调整套餐
**When** 套餐升级或降级操作
**Then** 系统平滑切换能力边界，不中断正在执行的任务
**And** 套餐变更记录到审计日志

---

**Epic 9 小结：** 共 5 个 Stories，覆盖 FR62, FR63, FR64, FR65, FR66。
